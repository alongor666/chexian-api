## 📊 按分析维度查找

### 业绩分析 (Performance Analysis)
**目标**: 衡量业务人员、机构的业绩表现

**核心字段**:
- `保费` (premium) - 业绩金额
- `业务员` (salesman_name) - 业绩归属
- `三级机构` (org_name) - 组织业绩
- `签单日期` (sign_date) - 业绩时间归属

**参考文档**:
- 分析价值矩阵 § 维度1: 业绩分析
- SQL示例: 业务员业绩排名、机构业绩对比、业绩趋势

**关键指标**:
- 实收保费 = SUM(保费) (正负抵消)
- 毛保费 = SUM(ABS(保费)) (不考虑退费)
- 人均产能 = 实收保费 / 业务员数量

---

### 续保分析 (Renewal Analysis)
**目标**: 分析客户留存和续保表现

**核心字段**:
- `是否续保` (is_renewal) - 续保标识
- `续保单号` (renew_policy_no) - 关联上年保单
- `保险起期` (start_date) - 计算续保周期

**参考文档**:
- 分析价值矩阵 § 维度2: 续保分析
- 业务规则: 续保单号 ↔ 是否续保 (100%一致)

**关键指标**:
- 续保率 = COUNT(是否续保=True) / COUNT(*)
- 续保周期 = 保险起期(当年) - 保险起期(上年)
- 脱保率 = 1 - 续保率

---

### 产品结构分析 (Product Mix Analysis)
**目标**: 分析险种组合和产品销售结构

**核心字段**:
- `险类` (insurance_type) - 交强险/商业险
- `险别组合` (coverage_combo) - 单交/交三/主全
- `是否交商统保` (bundle_type) - 统保类型
- `商车自主定价系数` (pricing_factor) - 定价系数

**参考文档**:
- 分析价值矩阵 § 维度3: 产品结构分析
- 业务规则: 险别组合决定统保类型

**关键指标**:
- 商业险渗透率 = COUNT(险类=商业保险) / COUNT(*)
- 统保渗透率 = COUNT(是否交商统保=套单) / COUNT(*)
- 单均保费 = AVG(保费) (按险别组合分组)

---

### 渠道分析 (Channel Analysis)
**目标**: 分析不同销售渠道的效能和贡献

**核心字段**:
- `终端来源` (channel) - 销售渠道
- `是否电销` (is_telesales) - 电销标识

**参考文档**:
- 分析价值矩阵 § 维度4: 渠道分析
- 关联规则: 终端来源=0110融合销售 ↔ 是否电销=True

**关键指标**:
- 渠道占比 = COUNT(终端来源) / COUNT(*)
- 渠道人均产能 = SUM(保费) / COUNT(DISTINCT 业务员)

---

### 客户细分分析 (Customer Segmentation)
**目标**: 按客户类型和车辆属性细分分析

**核心字段**:
- `客户类别` (customer_category) - 客户类型
- `厂牌车型` (car_model) - 车辆品牌
- `吨位分段` (tonnage_segment) - 货车吨位
- `是否新车` (is_new_car) - 新车标识
- `是否新能源` (is_new_energy) - 新能源车

**参考文档**:
- 分析价值矩阵 § 维度5: 客户细分分析
- 特殊场景: 客户类别=营业货车 → 必须结合吨位分段

**关键指标**:
- 客户占比 = COUNT(客户类别) / COUNT(*)
- 新车渗透率 = COUNT(是否新车=True) / COUNT(*)
- 新能源渗透率 = COUNT(是否新能源=True) / COUNT(*)

---

### 营业货车专项分析 (Commercial Truck Analysis)
**目标**: 针对营业货车细分市场深度分析

**核心字段**:
- `客户类别` (customer_category) - 筛选营业货车
- `吨位分段` (tonnage_segment) - 吨位维度
- `保费` (premium) - 保费分析
- `厂牌车型` (car_model) - 车型分析

**参考文档**:
- 分析价值矩阵 § 维度6: 营业货车专项分析
- 业务规则: 营业货车必须有吨位分段

**关键指标**:
- 营业货车占比 = COUNT(客户类别=营业货车) / COUNT(*)
- 吨位分布 = COUNT(吨位分段) (WHERE 客户类别=营业货车)
- 货车单均保费 = AVG(保费) (按吨位分段分组)

### 批改业务分析 (Endorsement Analysis)
**目标**: 分析批改(退保、过户、变更)对业务的影响

**核心字段**:
- `批单号` (endorsement_no) - 批改标识
- `批改类型` (endorsement_type) - 批改原因
- `保费` (premium) - 退费金额

**参考文档**:
- 分析价值矩阵 § 维度7: 批改业务分析
- 业务规则: 负保费必须伴随批改记录

**关键指标**:
- 批改率 = COUNT(批单号非空) / COUNT(*)
- 退保率 = COUNT(批改类型=16退保) / COUNT(*)
- 退费金额 = SUM(ABS(保费)) (WHERE 保费 < 0)

### 统保优惠分析 (Bundle Analysis)
**目标**: 分析统保(交强+商险同保)的渗透率和价值

**核心字段**:
- `是否交商统保` (bundle_type) - 统保类型
- `险别组合` (coverage_combo) - 险种组合

**参考文档**:
- 分析价值矩阵 § 维度8: 统保优惠分析

**关键指标**:
- 统保渗透率 = COUNT(是否交商统保=套单) / COUNT(*)
- 单交转套单机会 = COUNT(是否交商统保=单交) / COUNT(*)

## ⚠️ 常见陷阱快速参考

### 陷阱1: 空值误判
❌ `WHERE 续保单号 IS NULL`
✅ `WHERE 是否续保 = False`

**原因**: 空值有业务含义,应使用布尔字段判断

### 陷阱2: 负保费忽略
❌ `AVG(保费)` - 会受负保费影响
✅ `AVG(保费) FILTER (WHERE 保费 > 0)` - 仅统计承保保费

**原因**: 负保费是批改退费,需分开统计

### 陷阱3: 险类混淆
❌ `WHERE 商车自主定价系数 > 1.0` - 会漏掉交强险(该字段为空)
✅ `WHERE 险类 = '商业保险' AND 商车自主定价系数 > 1.0`

**原因**: 商车自主定价系数仅对商业险有意义

### 陷阱4: 日期类型错误
❌ `保险起期 - 签单日期` - 字符串相减会报错
✅ `DATEDIFF('day', CAST(签单日期 AS DATE), CAST(保险起期 AS DATE))`

**原因**: 日期字段存储为字符串,必须先转换

### 陷阱5: 去重逻辑错误
❌ `COUNT(DISTINCT 保单号)` - 60万记录,但实际保单数更少
✅ 理解同一保单可能有多条记录(交强险+商业险分开)

**原因**: 需按业务逻辑决定是否去重

## 🔍 业务规则边界 (测试发现的隐性规则)

⚠️ **来源**: 基于早期测试用例分析发现的业务规则边界条件。原 `tests/*.test.ts` 测试文件已随 API-only 拆分重组（现行测试收集范围以 `vite.config.ts` / `vitest.integration.config.ts` 为准），**业务规则本身仍有效**，口径疑问以 `数据管理/knowledge/rules/车险数据业务规则字典.md` 为准。

### 续保单号的条件非空

**口径事实源**: 业务规则字典 § A.2 / C.2

**业务规则**:
- 空字符串和NULL都视为无效续保
- 续保单号仅在 `是否续保 = True` 时有值
- 条件非空率: 36.4%

**正确写法**:
```sql
-- ❌ 错误: 仅检查NULL
WHERE 续保单号 IS NOT NULL

-- ✅ 正确: 同时检查NULL和空字符串
WHERE 续保单号 IS NOT NULL AND 续保单号 <> ''

-- ✅ 最佳实践: 使用布尔字段
WHERE 是否续保 = True
```

**业务含义**:
- 续保单号是判断保单是否续保的关键字段
- 空值有业务含义,表示非续保保单(不是数据缺失)

---

### 负保费的三种含义

**口径事实源**: 业务规则字典 § C.1；净额/承保口径见 `开发文档/KPI口径说明.md`

**业务规则**:
- **正保费**: 正常签单收入 (`保费 > 0`)
- **零保费**: 未生效或起保前退保 (`保费 = 0`)
- **负保费**: 以前年度保单在当年退保 (`保费 < 0`)

**口径差异**:
```sql
-- 净额口径: 统计包含正/零/负保费
SELECT SUM(保费) as 净额保费 FROM PolicyFact

-- 承保口径: 仅统计正保费
SELECT SUM(保费) FILTER (WHERE 保费 > 0) as 承保保费 FROM PolicyFact

-- 毛保费: 不考虑退费
SELECT SUM(ABS(保费)) as 毛保费 FROM PolicyFact
```

**业务场景**:
- 负保费必须伴随批改记录
- 不同口径反映不同的业务视角
- 负保费影响续保率、实收保费等KPI

### 商车自主定价系数的NULL值

**口径事实源**: 业务规则字典 § C.9；类型契约 `server/src/normalize/validator.ts`（codegen 自 `server/src/config/field-registry/fields.json`）

**业务规则**:
- NULL值表示未使用自主定价系数
- 仅当 `险类 = '商业保险'` 时有值
- 条件非空率: 23.8%

**正确写法**:
```sql
-- ❌ 错误: 直接过滤NULL
WHERE 商车自主定价系数 > 1.0  -- 会漏掉交强险

-- ✅ 正确: 先筛选险类
WHERE 险类 = '商业保险' AND 商车自主定价系数 > 1.0

-- ✅ 或使用COALESCE提供默认值
WHERE COALESCE(商车自主定价系数, 1.0) > 1.0
```

**业务含义**:
- 交强险的定价系数为NULL(不是0)
- 计算平均保费时需排除NULL值
- 影响风险分层和定价合理性分析

### 批改类型的枚举值

**口径事实源**: 业务规则字典 § G.1/G.2

**业务规则**:
- 批改类型是保单生命周期的重要标记
- 不同批改类型影响保费计算和统计
- 条件非空率: 1.8% (仅批改保单有值)

**枚举值**:
```python
批改类型 = {
    1: '新增',     # 新增险种
    2: '修改',     # 修改保单信息
    16: '退保',    # 退保批单
    # ... 其他枚举值
}
```

**关联规则**:
- `批单号非空` ↔ `批改类型非空` (100%一致)
- 负保费必须伴随批改记录
- 批改率 = COUNT(批单号非空) / COUNT(*)

**分析场景**:
- 退保率分析: `COUNT(批改类型=16) / COUNT(*)`
- 批改统计: 按批改类型分组统计
- 业务质量: 批改率高的机构可能存在问题

---

### 多别名支持的边界条件

**口径事实源**: `server/src/normalize/mapping.ts`（codegen 自 `server/src/config/field-registry/fields.json`，勿手改）

**业务规则**:
- 精确匹配优先: 第一个匹配项被选中
- 中英文混合支持: 支持英文、中文、混合列名
- 自定义别名配置: 可通过配置扩展别名映射

**边界场景**:
```typescript
// 场景1: 精确匹配优先
aliases: ['policy_no', '保单号', 'PolicyNo']
输入: '保单号'  → 匹配第2个
输入: 'policy_no'  → 匹配第1个

// 场景2: 大小写不敏感
输入: 'Policy_No'  → 匹配 'policy_no'

// 场景3: 模糊匹配警告
输入: 'policyno'  → 产生警告但不阻止处理
```

**业务价值**:
- 适应不同数据源的列名差异
- 提高数据兼容性和容错性
- 支持中英文环境

### 日期转换的容错机制

**口径事实源**: `server/src/normalize/validator.ts`（codegen 产物）

**业务规则**:
- 日期字段支持 DATE 和 VARCHAR(可解析格式)
- VARCHAR类型会在查询时自动转换
- 支持多种日期格式: 'YYYY-MM-DD', 'YYYYMMDD', 'DD/MM/YYYY'

**容错示例**:
```sql
-- ✅ DuckDB自动转换以下格式
CAST('2024-01-01' AS DATE)  -- 标准
CAST('20240101' AS DATE)    -- 紧凑
CAST('01/01/2024' AS DATE)  -- 欧式
```

**注意事项**:
- 无效日期会返回NULL
- 建议在数据加载时统一格式
- 查询时仍需CAST AS DATE确保类型正确
