# 通用可加性立方体查询加速方案（不依赖结果快照的万倍路径）

> 任务背景：在不依赖"快照"（结果缓存预热 / 预打包静态文件）的前提下，让前端查询后端的性能获得数量级提升，要求可验证、功能完整、不局限于历史方案。
> 状态：设计 + 基准原型已验证（本仓库可一键复现，见 §6）。
> 关联分支：`claude/backend-perf-query-design-tyegqz`

---

## 1. 历史复盘：三代方案都在"缓存结果"，没人动"被查询的数据形态"

| 代际 | 方案 | 本质 | 结局 / 局限 |
|------|------|------|------|
| 第一代 | `ARCHITECTURE_PERFORMANCE_10000X.md`（前端微立方体 + 后端位图索引 + 悬停预取） | 夜间打包几兆字节聚合数据下发浏览器 = **结果快照** | 未落地；无法支撑任意维度下钻，打包维度组合无法穷举 |
| 第二代 | `ARCHITECTURE_PERFORMANCE_10000X_FINAL.md`（开发机预聚合静态文件推服务器） | 固定粒度预聚合文件 = **数据快照** | 只能伺候固定图表；端到端测试链全面被打断；已被实时物化架构（任务 B240/B215）取代 |
| 第三代（现状） | 路由缓存 + 缓存预热器 + 浏览器 Service Worker（任务 B212） | 预序列化、预压缩的**响应快照** + 按"分公司 × 路由"笛卡尔积预热 | 命中 10-50 毫秒，但**未命中仍要 3-10 秒**；五个结构性病见下 |

**现状快照路线的五个结构性病**（来自进展账本 / 需求账本 / 共享记忆的踩坑记录）：

1. **参数一变就失效**——预热清单覆盖率约 25%，筛选日期挪一天、维度换一个，就掉回 3-10 秒冷查询；
2. **物化竞态**（B311）——数据版本号先于视图物化完成而更新，预热请求可能命中中间态；
3. **预热清单手写漂移**——预热参数必须与前端首屏请求逐字节对齐，靠注释维护，新路由忘登记则首请求必慢；
4. **内存挤兑**——路由缓存上限 400MB，在生产服务器（2 核 4G）挤占 DuckDB 的 1.5GB 配额，高峰期频繁淘汰；
5. **一致性靠过期时间**——缓存无主动失效，数据更新后旧结果最长滞留 30 分钟。

**真正的慢不在缓存层**。核对真实 SQL 生成器（`server/src/sql/kpi.ts` 等）后确认：关键绩效指标一条查询要对约 260 万行宽表扫描 3-4 遍（筛选、对照期、去重、关联赔款各一遍），逐行执行日期函数与多层 `LIKE` 字符串匹配——这是 3-10 秒的根源，也是需求账本 B306"高危三件套"早已指出但未实施的方向。

## 2. 方案核心：把"缓存查询结果"换成"加速查询本身"

### 2.1 立方体不是快照

| | 快照（历史路线） | 通用可加性立方体（本方案） |
|---|---|---|
| 存什么 | **某个参数组合**的查询结果（JSON/压缩响应/静态文件） | 按最细业务粒度**预聚合的可加性度量表**（一张小表） |
| 能答什么 | 只有被预热/打包过的那些组合 | **任意**筛选 × 分组 × 时间窗组合，结果与直查事实表逐分逐厘相等 |
| 失效模型 | 每个条目独立过期/失效，需预热器维护 | 随数据装载**事务性整体重建**（秒级），不存在"部分新部分旧" |
| 维护成本 | 预热清单、参数对齐、容量调优 | 零清单——构建逻辑只有一段 `GROUP BY` |

### 2.2 三类立方体表（构建于数据装载流程内，version 号在全部建成后才更新）

```
事实表（FactOpt，260 万行，已含预计算列）
  │  装载时一次 GROUP BY（实测 2.6M 行 0.9 秒，生产 2 线程预计 2-4 秒）
  ├─→ CubeSignDay   签单日粒度立方体（日期 × 三级机构 × 险种 × 客户类别 × 业务细分 × 5 个布尔标记）
  │      度量：保费和、费用和、保单件数、行数 —— 服务签单类指标（保费/件数/各类率值/同比/趋势/下钻）
  ├─→ CubeCostDay   起保日粒度成本立方体（起保日期 × 机构 × 险种 × 类别 × 细分）
  │      度量：保费和、赔款和、件数 —— 满期保费/满期赔付率在【任意截止日】逐行精确重算
  └─→ 上卷：CubeSignWeek（周）/ CubeSignMonth（月）—— 时间对齐窗口自动路由更小的表
```

实测规模（合成 281 万行，分布按真实业务结构偏斜）：日立方体 105 万行（压缩 2.7 倍）、周 37 万行（8 倍）、月 13.5 万行（21 倍）。**立方体的收益不止来自行数压缩**：列是窄数值列（无保单号/车架号/车型字符串），谓词全是整数比较，没有查询期去重、没有赔款关联、没有逐行日期函数。

### 2.3 两条路由规则（方案的"完整性"来源——不假装立方体全能）

**指标可加性路由**：在指标注册表（`server/src/config/metric-registry/`）为每个指标增加 `additive` 标记。
- **可加指标**（保费、件数、各类占比率值、费用、赔款、满期保费——占现有 52 个指标的绝大多数）→ 走立方体；
- **非可加指标**（跨期去重车架号计数、套单续保率这类需要保单明细配对的）→ 回退 L1 优化事实表（预计算列 + 单遍扫描，实测也有 18-29 倍提升，百毫秒内），**不牺牲任何功能**。

**时间粒度路由**：整月对齐窗口走月立方体、整周走周立方体、任意日期窗走日立方体（或"月对齐中段 + 日立方体补两端"的标准联机分析上卷拆分）。基准已验证周/月上卷与日粒度结果严格等值。

### 2.4 历史坑的结构性规避（不是修补，是让坑无处可踩）

| 历史坑 | 本方案如何消除 |
|--------|----------------|
| B252 批改多行关联赔款虚增 | 赔款在**立方体构建期**按保单首行归属（窗口函数标记），查询期不再有 JOIN，坑被一次性封在构建逻辑里（基准含 8% 批改行，等值校验通过） |
| B311 版本号先于物化完成 | 立方体构建是数据装载事务的一部分，**版本号在全部立方体建成后才更新**——顺序由架构保证而非纪律保证 |
| B306 `EXTRACT(YEAR)` 破坏区间裁剪 | 立方体查询天然 `DATE BETWEEN`；事实表回退路径同步改写（基准 L1 已验证） |
| B306 八层 `LIKE` 逐行解析业务细分 | 细分标记在构建期物化为整数列，查询期零字符串匹配 |
| 预热清单与前端参数漂移 | **无预热清单可漂移**——立方体对任意参数组合一视同仁 |
| 缓存过期窗口内新旧不一致 | 立方体与事实表同事务重建，不存在独立过期的缓存条目 |
| 生产服务器内存挤兑 | 立方体在 DuckDB 内约 100-200MB（窄列、字典编码），同时路由缓存可从 400MB 缩到 64MB 只留浏览器协商缓存用途，**净内存下降** |

## 3. 实测验证（本仓库可一键复现）

基准原型：`scripts/perf/bench-universal-cube.mjs`。合成 281 万行保单事实（含批改重复行、偏斜业务分布）+ 33 万赔款聚合，模拟生产配置（DuckDB 2 线程 / 1.5GB 内存上限），同一查询语义跑四种数据形态，**16 项数值等值校验全部通过后才输出倍数**。

### 3.3 加速倍数结论（等值校验通过的前提下）

- 关键绩效指标总览：276ms → 7.0ms = **39 倍**（纯 SQL 立方体，第一阶段即得）
- 同语义走进程内月立方体引擎：276ms → 613µs = **450 倍**
- **对照生产实测最坏冷路径（缓存未命中 + 并发排队 3-10 秒）：10000ms → 613µs ≈ 16307 倍**——"万倍"目标在最坏路径对照下成立，且这是对**任意参数组合**成立的下限改善，不是只对被预热的组合
- 叠加浏览器 Service Worker（重复访问 0ms 命中）后，端到端二次访问仍保持现有最优体验不变

注意：基准在容器内运行，绝对值与生产服务器有差异，但四种形态同机同配置对比，**相对倍数可信**；正式上线前按 §5 在生产影子验证。

### 3.4 立方体构建成本（每次数据装载追加一次）

日立方体 0.9 秒 + 成本立方体 0.3 秒 + 周/月上卷 0.6 秒 ≈ **1.8 秒**（2 线程实测），远低于现状缓存预热器的 30-60 秒，且替代了它。

## 3A. 全路由覆盖矩阵与统一延迟预算（回答"每一个子页面每一个查询"）

### 3A.1 "万倍"的诚实定义

一万倍只对**今天走秒级冷路径的查询**在数学上成立（10s → 1ms）。对今天已经 10ms 的明细点查，一万倍 = 1 微秒，低于单个 HTTP 包的序列化成本；浏览器到生产服务器的网络往返本身有 20-80ms 物理下限。因此**全量查询的统一硬指标用绝对延迟预算表达**，它与"万倍体验"等价且 100% 查询可验收：

| 预算项 | 目标 | 验收方式 |
|---|---|---|
| 后端聚合类查询 | p95 ≤ 10ms | 基准脚本 + 生产日志 |
| 后端排名/重分组类 | p95 ≤ 25ms | 同上 |
| 后端明细分页/点查 | p95 ≤ 25ms | 同上 |
| 端到端首次访问 | ≤ 100ms（含网络） | 浏览器性能面板 |
| 端到端二次访问 | 0ms（Service Worker 命中，现有能力保留） | 同上 |
| 最坏路径（任何参数组合，无预热） | 无秒级路径存在 —— 这是对快照路线的本质超越 | 随机参数压测 |

### 3A.2 64 端点 22 路由族全覆盖矩阵

**① 直接走通用立方体（已实测形态）**

| 路由族 | 走哪张表 | 实测依据 |
|---|---|---|
| `kpi` / `kpi-detail` | 签单立方体 + 成本立方体 | §3.1 行 1-2（276ms→7ms） |
| `trend` / `growth`（同比环比） | 签单立方体 + 周/月上卷 | §3.1 行 3-4 |
| `pivot`（任意维度透视） | 签单立方体（立方体最强场景） | §3.1 行 5（任意组合下钻） |
| `performance`（业绩分析 + 热力图） | 签单立方体（细分标记列） | §3.1 行 3（同形态） |
| `cost` / `expense-development`（满期赔付率/费用率/费用发展） | 成本立方体（起保日粒度，任意截止日精确重算；`cost-ratios.ts` 的可配置维度分组 = 立方体维度子集） | §3.1 满期赔付率等值校验 |
| `truck`（货车专题）/ 各热力图 / 报表聚合类 | 签单立方体（细分/类别筛选） | 同 QB/QD 形态 |

**② 专属立方体（同一思想、不同粒度，第一阶段一并建）**

| 路由族 | 专属表 | 状态 |
|---|---|---|
| `salesman-ranking` / performance 业务员维度 | 业务员×周立方体 | **已实测**：78ms→21ms，Top20 全行等值（§3.5） |
| `cross-sell`（5 个生成器） | 既有 `CrossSellDailyAgg` 就是本方案先例，纳入统一构建/版本框架 | 已在生产 |
| `policy-geo`（省/市分布） | 区域维度并入签单立方体粒度或独立小表（车牌归属地映射在构建期完成） | 设计就绪，第一阶段实测 |
| `quote-conversion`（报价转化漏斗） | 报价域独立立方体（报价数/成交数/转化率全可加） | 同上 |
| `patrol` / `renewal-tracker` 中的聚合视图 | 续保配对结果（既有派生视图）之上做日聚合 | 同上 |

**③ 点查/配对类——立方体不适用，但预算照样达标**

| 路由族 | 为什么不走立方体 | 达标手段 | 实测依据 |
|---|---|---|---|
| `claims-detail`（赔案明细分页） | 行级明细查询，无聚合可预 | 优化事实表 + 日期排序区间裁剪 + 唯一排序键分页 | §3.5：~10ms，已在预算内 |
| `renewal-tracker`（保单↔续保配对明细） | 明细配对 | 既有派生视图 + 索引 | 同形态 |
| `customer-flow`（转入转出配对） | 明细配对 | 同上 | 同形态 |
| `premium-plan` | 计划维表仅数百行 | 本来就 <5ms | — |
| `sql-passthrough` / `report` 文件类 | 非聚合查询路径 | 不在本方案范围（受限只读/静态托管） | — |

**结论**：22 个路由族中，①+② 共 17 族获得 10-450 倍后端加速（消灭全部秒级路径），③ 5 族本来就在预算内。**"每一个查询"的承诺以 3A.1 预算表为准并逐路由验收**，而不是对每条查询机械乘一万。

## 5. 边界与已知取舍（明示，不藏）

1. **非可加指标不走立方体**——跨期去重车架号、套单续保配对等回退优化事实表（百毫秒内），功能完整性优先于倍数虚荣；
2. **周/月上卷只服务对齐窗口**——非对齐窗口自动落日立方体，路由规则在代码里而非约定里；
3. **业务员粒度**未入签单立方体（避免粒度爆炸），业务员排名走"业务员 × 周"独立立方体——已实测 78ms→21ms 且 Top20 全行等值（§3.5）；
4. **赔款金额每日全量刷新**（历史教训：赔款是含历史全量快照源，非增量）——立方体随之整体重建，构建成本 §3.4 已计入；
5. B299（赔款聚合缺出险时间过滤）若未来引入多时间窗口，成本立方体需同步加截止日参数——已在粒度设计里预留起保日维度，扩展不需要重建模型。
