# 架构优化计划（2026-06-20 全栈审计）

> 基于 2026-06-20 对前端、后端、数据层的全量架构审计，识别出 **6 大优化方向、28 个具体改进点**。
> 已在 BACKLOG 中登记的条目标注了编号，避免重复；新发现标注 `[NEW]`。

## 一、优化方向总览

| # | 方向 | 核心问题 | 预期收益 | 优先级 |
|---|------|---------|---------|--------|
| 1 | 查询性能 | DuckDB 反模式 + 物化缺口 | 查询延迟降低 50-70% | P1 |

## 二、方向 1：查询性能优化（P1）

### 1.1 DuckDB 性能高危三件套（已登记 B306）

| 编号 | 反模式 | 位置 | 影响 | 修复方案 |
|------|--------|------|------|---------|
| F-01 | `EXTRACT(YEAR FROM col) = N` 破坏 zonemap 过滤 | kpi.ts、growth/*.ts | 全表扫描，慢 30-60% | 改为 `col BETWEEN '2026-01-01' AND '2026-12-31'` |
| F-02 | 增长查询双 CTE 两次扫描 PolicyFact | growth/yoy.ts | ~50% 冗余开销 | 合并为单次扫描 + 窗口函数 |
| F-03 | performance-analysis/shared.ts 8 层嵌套 LIKE+CAST | shared.ts:123-155 | 每行 8 次字符串匹配 | 物化 segment_tag 为整型列（ETL 阶段预处理） |

**工作量**：5-8 天 + 基准测试验证

### 1.2 保单去重 CTE 未物化（关联 B252）

- `buildPolicyDedupCTE()` 在每个 cost/trend/growth 请求中对 ~260 万行 PolicyFact 做 GROUP BY
- Trend/Cost 立方体已部分物化，但**立方体未启用时仍每次重算**
- **方案**：ETL 阶段预物化去重表（`PolicyFactDedup`），查询直接引用

### 1.3 ClaimsAgg JOIN 缺索引

- `LEFT JOIN ClaimsAgg c ON p.policy_no = c.policy_no` 无显式索引
- DuckDB 内存表靠自适应哈希，但大表 JOIN 仍可从排序提示中获益
- **方案**：ClaimsAgg 物化时按 `policy_no` 排序；或纳入立方体预聚合

### 1.4 立方体覆盖缺口（关联 2026-06-11-claude-90a92c）

| 域 | 立方体状态 | 当前延迟 | 目标 |
|----|-----------|---------|------|
| Trend | Phase 4 完成（影子对账中） | 冷启 5s → 7ms | 全量启用 |
| Cost | Phase 3 完成（影子对账中） | 冷启 3s → 10ms | 全量启用 |
| Salesman | Phase 5 完成 | 已启用 | 维持 |
| KPI（cost routes） | 待 B246 完成 | 2-4s | <50ms |
| **报价转化** | **无计划** `[NEW]` | 1-3s | 需新建立方体 |
| **地理分析** | **无计划** `[NEW]` | 2-5s | 需新建立方体 |

### 1.5 同比/YTD 虚假行（已登记 2026-06-11-claude-7a2849）

- `FULL OUTER JOIN` 跨年边界产生 ghost rows（-100% 增长）
- 周粒度 `DATE_TRUNC('week') + 1 year` 跨年错位
- **方案**：PR #640 待合并

## 三、方向 2：前端渲染优化（P1）

### 2.1 巨型组件拆分（关联 B331）

| 组件 | 行数 | 拆分方案 |
|------|------|---------|
| `PerformanceAnalysisPanel.tsx` | 1,401 | → 编排器 (200L) + Summary/Heatmap/Trend/Drilldown 四子组件（各 lazy） |
| `GeoRiskPanel.tsx` | 951 | → 地图 + 表格 + 洞察 三子组件 |
| `CrossSellAnalysisPanel.tsx` | 891 | → 概览 + 趋势 + 热力图 三子组件 |
| `NewEarnedPremiumTable.tsx` | 891 | → 表格框架 + 行虚拟化（react-window） |
| `FilterLayoutV2.tsx` | 690 | → 基础筛选 + 高级筛选面板 |

### 2.2 图表 memo 缺失 `[NEW]`

- 20+ 图表组件（LineChart 720L、BarChart 等）**未包裹 `React.memo()`**
- 父组件任何状态变化触发图表重绘（ECharts 实例重建开销大）
- **方案**：所有图表组件添加 `memo()` + 自定义 `areEqual` 比较（data/title/loading）
- **工作量**：2-3 天

### 2.3 FilterContext 全局重渲染 `[NEW]`

- `useGlobalFilters()` 有 48+ 消费者
- 筛选器对象引用变化 → 全部 12 个 feature 模块重渲染
- `StableContext` 已做缓解（3 个模块采用），但 **9 个模块仍用原始 context**
- **方案**：
  1. 剩余 9 个模块迁移到 `useMemoizedFilters()`
  2. 或引入 selector 模式（`useContextSelector` / zustand slice）

### 2.4 大表缺虚拟化 `[NEW]`

- 已赚保费表、下钻表可能渲染 1000+ 行，无虚拟化
- `VirtualTable.tsx` 存在但行渲染器未 memo
- **方案**：统一用 `react-window` + 行级 memo

## 五、方向 4：缓存与数据流优化（P2）

### 4.1 ETL 版本竞态（已登记 B311）

- `setDataVersion()` 在立方体物化完成**之前**触发
- 请求可能命中正在构建的立方体 → 数据不一致
- **方案**：版本号分两阶段（`dataLoaded` → `cubesReady`），路由层根据 cubes 状态选择查询路径

### 4.2 缓存可观测性缺失 `[NEW]`

- 查询缓存有 `_hits/_misses` 计数器，但**未暴露到 /metrics 端点**
- 连接池仅暴露 `saturatedRecently` 布尔值，无百分位直方图
- **方案**：
  1. 新增 `/api/metrics` 端点，暴露：查询缓存命中率 / 路由缓存命中率 / 连接池利用率 / 立方体新鲜度
  2. 前端可选接入监控面板（P3）

### 4.3 路由缓存字节估算粗糙

- `estimateBytes = JSON.stringify(data).length * 2` 是粗估
- 大结果集（2MB+）容易引发缓存颠簸
- **方案**：按路由类型设差异化 TTL + 大结果集不入缓存（仅 ETag 复用）

### 4.4 Service Worker 缓存命中率不可观测 `[NEW]`

- SW 设置了 `X-SW-Source` header 但无分析管道
- 无法验证 SW 是否真正减少了服务端压力
- **方案**：服务端统计 `X-SW-Source: cache` 比例，写入审计日志

## 六、方向 5：构建与包体积优化（P2）

### 5.1 组件级懒加载缺失 `[NEW]`

- 路由级 `React.lazy()` 已实现（17 个路由）
- 但**组件级**未做：巨型面板（PerformanceAnalysisPanel、GeoRiskPanel）仍打入父页面 chunk
- **方案**：面板级 lazy + Suspense，结合 §2.1 拆分一起做

### 5.2 ECharts 实例未池化 `[NEW]`

- 每个图表 `echarts.init()` 创建新实例
- 页面切换 → 旧实例销毁 + 新实例创建 → GC 压力
- **方案**：实例池（回收 + 复用 canvas DOM），页面切换时 `setOption` 而非重建
- **优先级**：P3（收益中等，实现复杂）

### 5.3 地图数据同步加载 `[NEW]`

- `geo-map-loader.ts` 省级地图在 GeoRiskPanel 首次渲染时同步加载
- 阻塞首次渲染
- **方案**：动态 `import()` + Suspense 包裹

### 5.4 TypeScript 严格性缺口 `[NEW]`

- 缺少 `exactOptionalPropertyTypes` — 可选属性允许 `undefined` 赋值
- 缺少 `forceConsistentCasingInFileNames` — 文件名大小写不一致不报错
- **方案**：逐步启用（先 CI warning → 再升级为 error）

## 七、方向 6：安全加固（P2）

### 6.1 CSP unsafe-eval（已登记 B320）

- `app.ts:54` 保留 `'unsafe-eval'`，ECharts 纯数据驱动渲染不需要
- **方案**：移除 + E2E 验证全站功能无异常

### 6.2 SQL 注入边界 `[NEW]`

- `sql-permission-injector.ts` 基于正则注入 WHERE — 不支持嵌套子查询（`SELECT * FROM (SELECT ...) AS t`）
- `cache-warmer.ts` 有 14+ 处日期字符串插值
- **方案**：
  1. SQL 注入器增加嵌套子查询检测（或迁移到 AST 解析）
  2. 集中日期转义到 `escapeDateForSql()` 工具函数

### 6.3 明文密钥（已登记 B322）

- claude-to-im 飞书 `app_secret` 明文存于配置文件
- **方案**：迁移到环境变量 / 密钥管理器

### 6.4 权限过滤假阳性（关联 B326）

- RepairDim 缺少 `branch_code` → 联邦查询合法用户被 403
- **方案**：按域配置权限过滤字段（非统一 `branch_code`）
