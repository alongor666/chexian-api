## 一、评审范围与方法

本文是对**理赔模块**（claims/理赔/赔款/满期赔付率/理赔热力图/未决盯盘）的一次架构评审，覆盖六层链路：源 Excel → ETL/CDC → DuckDB 视图层 → 后端 SQL 生成器 → 路由缓存 → 前端消费。

评审方法：三层并行只读勘察 + 关键承重发现的**逐文件亲眼核验**（口径正确性不接受二手转述）。核验过的事实在下文以「证据」标注文件:行号。

### 1.1 架构全景

```
源 Excel(05_理赔明细)
   │ daily.mjs claims_detail
   ▼
convert_claims_detail.py ── 去重(claim_no,keep=last) + JOIN PolicyFact 取保单年度 + schema 补齐
   │ _incoming.parquet
   ▼
claims_partition_manager.py ── 年度分区 CDC(6 类变更) + 冻结/自动解冻 + _partition_meta.json

   ▼

   ▼

```

## 二、总体评价

理赔模块**分层清晰、职责边界明确、成熟度较高**，不存在结构性崩坏。做得好的方面：

| 优点 | 证据 |
|------|------|
| 赔款口径设定了单一事实源（SSOT） | ClaimsAgg.reported_claims 用一段 CASE 收口"已决取已决金额、未决取未决金额，剔除无责/零结/注销/拒赔"，并与电子表格周报对账校准（赔付率偏差 +7.3%→+1.28%）。证据：`server/src/services/duckdb-domain-loaders.ts:467-484` |
| 批改副本去重系统化 | B252 把"保单原单+批改导致赔款虚增 5.52%"用共享 helper + 反向 JOIN 子查询全覆盖，非逐点打补丁。证据：`server/src/sql/shared/policy-dedup.ts`、`claims-detail.ts:30-60` |
| 分域解耦降内存 | 8 域重构把赔付从保单事实表剥离为独立 ClaimsAgg，生产 VPS（2 核 4G）内存减半、查询提速 10-50 倍（B240/B241） |
| CDC 分区有冻结语义 | 年度分区 + 已结案冻结 + 迟到数据自动解冻 + 重开（reopened）计数，正确处理"理赔金额动态演变" |
| 跨视图对账锚点已文档化 | 赔付率发展末列 ≡ 理赔热力图末列（B248 已修同期保单组错配） |

## 三、核心架构判断（主结论）

> **理赔模块的精算正确性，依赖一个未在类型层声明、仅靠运维习惯维持的隐藏时间不变量：`查询截止日(cutoff) ≡ 最新数据日`。这一不变量是 B299、赔款谓词多处重复、保单去重粒度不一致三个问题的共同根因。**

推理链：

1. ClaimsAgg 把赔款预聚合成 `policy_no → reported_claims` **单一标量**，**塌缩了时间维**——它只回答"截至最新数据，这张保单累计赔了多少"。
2. `cost-ratios.ts` 的满期赔付率 = `SUM(ClaimsAgg.reported_claims) / 满期保费`。**满期保费侧按 cutoff 严格截断**（满期天数 = MIN(cutoff−起保+1, 保单期限)），**赔款侧不截断**（全库快照）。证据：`server/src/sql/cost/cost-ratios.ts:74` 的 `LEFT JOIN ClaimsAgg c ON p.policy_no = c.policy_no` 无任何时间条件。
3. 当 `cutoff = 最新数据日`，分子分母同窗口 → 无偏差，**当前所有看板正确**。
4. 一旦引入「历史 cutoff / 多窗口 / 年初至今回溯 / 时间机器」，早期窗口立即变成"未来赔款 ÷ 过去满期保费"，虚高数倍——与 2026-05-30 四川诊断脚本暴露的同根因缺陷一致（变动成本率 183.6%→81.0%），当时只修了 5 个诊断脚本，生产 SQL 按红线未动（B299，状态 PROPOSED）。

**为什么这同时解释了另外两个问题**：

- **赔款谓词多处重复**：需要 cutoff 感知的查询（热力图、赔付率发展）**用不了** ClaimsAgg（它没时间维），只能在各自 SQL 里**重新内联**那段赔款 CASE 谓词。实测共 **4 处内联**：ClaimsAgg（`duckdb-domain-loaders.ts:472`）、热力图两处（`claims-heatmap.ts:405`、`:461`）、赔付率发展（`claims-detail.ts:521`）。SSOT 是"一张表"，但表的形态满足不了消费方的时间需求 → DRY 被迫破坏，且**靠人工对齐维持、非机制**。频度同比展示列 `reserve_wan`（`claims-detail.ts:586`）已用裸 `SUM(reserve_amount)` 未带过滤，是**漂移已经发生**的证据。
- **保单去重粒度不一致**：ClaimsAgg 按 `policy_no`，热力图合格保单按 `(policy_no, insurance_start_date)`。同一保单跨年批改在前者被合并、在后者被拆分，两套"保单去重粒度"并存。

## 五、架构决策记录（ADR）

### ADR-1：理赔聚合层是否引入 `asOfDate` 时间维

**状态**: 提议（Proposed）· 对应问题 #1/#2/#3

**背景**：ClaimsAgg 塌缩时间维，满期赔付率正确性绑定隐藏不变量 `cutoff=最新数据日`；该不变量未在类型层声明，是 B299、谓词重复、粒度不一致的共同根因。

**决策（建议）**：分两步走。

- **半步（零行为变更，建议先行，详见附录 A）**：把赔款金额 CASE 谓词提取为单一函数 `buildClaimsAmountExpr(opts)`，4 处内联全部改为消费它，生成 SQL 逐字不变。先收口 DRY（关闭 #2），并为完整步铺接口。
- **完整步（候选 A，按需启动）**：给 `createClaimsAggFromDetail` 加可选 `asOfDate` 参数，默认 `NULL=最新`（**不改现有看板行为**），消费方需 cutoff 感知时传入。把 #1 隐患转为可控、#3 粒度统一到 `(policy_no, insurance_start_date)`。

**后果**：

- 正面：赔款规则单点生效；历史/多窗口查询变为可安全支持；为"时间机器"特性解锁前置依赖。
- 负面：完整步需多窗口单测 + Parquet 直查对账，回归面较大；应**绑定真实历史回溯需求排期**，否则只做半步。

**备选方案**：

- **候选 B（推迟全部）**：维持现状，等真出现历史查询需求再改。代价：隐藏不变量长期裸奔，下一个写历史 cutoff 查询的人大概率踩坑（已有四川诊断前车之鉴）。**至少应先做半步关闭 #2。**

---

### ADR-2：未决盯盘的缓存分层

**状态**: 提议（Proposed）· 对应问题 #4

**背景**：未决盯盘（pending-overview/by-org/aging）是业务员实时操作面，赔案在"未决→已决"间秒级变化；当前 11 个 claims 端点全部回落 `hotspotMedium=2 小时` TTL（`shared.ts:40`），看板滞后于真实理赔状态最长 2 小时。

**决策（建议）**：新增 `hotspotRealtime` 档（建议 5-15 分钟），仅 pending-* 三端点改用；赔付率发展/热力图/成本/地理/时效/原因等冷分析保持 `hotspotMedium`。

**后果**：

- 正面：决策依据新鲜度显著提升；改动局部、低风险，可独立落地。
- 负面：短 TTL 增加未命中→SQL 执行频次；但未决查询基数小、维度浅，开销可控。

**备选方案**：维持统一 2 小时（现状）——代价是未决面数据陈旧；或全局缩短 TTL——代价是冷分析白白增加 SQL 负载，不可取。

---

### ADR-3：数据加载层定时炸弹防御

**状态**: 提议（Proposed）· 对应问题 #5/#6

**背景**：claims 域 11 处裸 `read_parquet`（无 `union_by_name=true`，与 policy 加载器不一致），一旦分区出现兼容 schema 漂移会令生产首次惰性加载崩（B340，潜伏）；保单事实表冗余理赔列上游已停发但未清（#6）。

**决策（建议）**：打包为一次"数据加载层硬化"。给各域读法统一加 `union_by_name=true`（回归 8 端点惰性加载 + 真实 schema 漂移场景），同步把发布前闸门 B339 的 claims 模板改回带 `union_by_name`；冗余列清理需先 grep 确认零消费方再删。

**后果**：

- 正面：消除定时炸弹，加载层与 policy 域一致。
- 负面：纯防御性、无业务收益，宜在低优先级窗口推进；改动需同步发布前闸门模板（否则破坏金丝雀探测力）。

**备选方案**：维持裸读 + 依赖 ETL 侧 schema 契约（exit(1) 拦未知字段）作为第一道闸——可接受作为现状，但首次加载崩的兜底缺失。

## 六、风险与缓解

| 风险 | 触发条件 | 缓解 |
|------|---------|------|
| 历史满期赔付率虚高数倍 | 引入时间机器/年初至今回溯/历史快照对比 | ADR-1 候选 A；在此之前**禁止**新增非"最新 cutoff"的赔付率查询，并在 `cost-ratios.ts` 文件头加显式不变量注释 |
| 赔款口径多处漂移 | 任一处单独修改赔款规则 | ADR-1 半步（谓词提取）；补"4 处赔款谓词字节级一致"快照单测 |
| schema 漂移致生产首次加载崩 | claims 分区新增/重排字段 | ADR-3（B340）；ETL 侧 schema 契约已 exit(1) 拦未知字段，是第一道闸 |
| 未决盯盘误导决策 | 业务员看 2 小时前的未决池 | ADR-2 缓存分层 |

## 七、结论与下一步

**评审结论**：理赔模块架构分层健康、SSOT 意识与去重治理到位，是全项目较成熟的模块。**唯一的高严重度结构性风险**是 ClaimsAgg 塌缩时间维带来的隐藏不变量（B299），它同时是赔款谓词重复（#2）与粒度不一致（#3）的共同根因——这三者应作为**一组**来决策，而非分别打补丁。其余均为中低风险、可独立排期。

**建议执行顺序**：

1. **ADR-1 半步**（零行为变更，最稳，附录 A 有可执行清单）→ 立即可做，先关闭 #2。
2. **ADR-2 缓存分层** → 局部低风险，独立可做。
3. **ADR-1 完整步 / ADR-3** → 绑定真实需求（历史回溯 / 加载层硬化窗口）排期，需用户决策点拍板。
