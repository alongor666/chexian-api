## 🔴 P0：续保专项SQL错误（紧急）

### 问题定位

**Console错误日志**：
```
[ERROR] 14:48:48.324 [RenewalAnalysisPanel][ERROR] Renewal Trend error
Error: Binder Error: Ambiguous reference to column name "time_period"
(use: "ep.time_period" or "rp.time_period")

[ERROR] 14:48:48.615 [RenewalAnalysisPanel][ERROR] Renewal Not Renewed error
Error: Binder Error: Referenced table "rp" not found!
Candidate tables: "renewal_detail"
LINE 55: WHERE rp.policy_no IS NULL
```

### 根因分析

**错误1：列名歧义**
- **原因**：SQL中两个表（ep和rp）都有`time_period`字段
- **触发条件**：JOIN查询时未明确指定表前缀
- **影响**：续保趋势图无法加载

**错误2：表别名不存在**
- **原因**：SQL中使用了别名`rp`，但该别名未定义
- **候选表**：`renewal_detail`（提示应该使用这个表）
- **影响**：未续保分析无法执行

### 文件定位（需要修复）

**推测位置**：
```
src/shared/sql/renewal.ts          # 续保相关SQL生成逻辑
src/features/dashboard/RenewalAnalysisPanel.tsx  # 调用续保查询的组件
```

**关键SQL模板搜索关键词**：
- `time_period`
- `renewal_detail`
- `rp.policy_no`

### 修复方案

**方案1（推荐）：修复表别名和列名前缀**
```sql
-- 错误的SQL（推测）
SELECT time_period, ...
FROM expected_policies ep
LEFT JOIN renewal_policies rp ON ...
WHERE rp.policy_no IS NULL

-- 修复后的SQL
SELECT ep.time_period, ...
FROM expected_policies ep
LEFT JOIN renewal_detail rp ON ...  -- 修正表名
WHERE rp.policy_no IS NULL
```

**方案2：使用CTE简化查询**
```sql
WITH renewal_data AS (
  SELECT policy_no, time_period FROM renewal_detail
)
SELECT rd.time_period, ...
FROM renewal_data rd
WHERE rd.policy_no IS NULL
```

## 🟡 P1：图表加载性能问题（高优先级）

### 问题现象

**受影响组件**：
- KPI卡片（6个）：Loading超过10秒
- 占比分析图表（3个）：Loading超过15秒
- 保费趋势图：Loading超过12秒
- 营业货车专项图：Loading超过15秒

**数据量**：607,455条记录

### 性能分析

**瓶颈1：全表扫描**
```sql
-- 当前查询（推测）
SELECT SUM(premium) FROM PolicyFact  -- 60万+行全扫描
```

**瓶颈2：缺少WHERE过滤**
- 用户已选择日期范围：`2026-01-01 ~ 2026-01-11`
- 但查询可能未应用此过滤条件
- 导致查询整个2023-2026年的数据

**瓶颈3：浏览器内存限制**
- DuckDB-WASM运行在浏览器中
- 内存和CPU受限
- 大数据量传输慢（Worker ↔ Main Thread）

### 优化方案

**方案1：强制日期范围过滤（立即见效）**
```typescript
// src/shared/sql/kpi.ts 或 trend.ts
const sql = `
  SELECT SUM(premium) as total_premium
  FROM PolicyFact
  WHERE policy_date >= '${filters.policy_date_start}'  -- ✅ 添加过滤
    AND policy_date <= '${filters.policy_date_end}'
`;
```

**预期效果**：
- 查询范围：2026-01-01 ~ 2026-01-11（11天）
- 预计记录数：从60万降至 ~2000条
- 性能提升：~300倍

**方案2：添加LIMIT保护**
```sql
-- 业务员明细查询
SELECT * FROM PolicyFact
WHERE ...
LIMIT 100  -- ✅ 限制返回行数
```

**方案3：增量加载（后续优化）**
- 首次加载：仅查询KPI汇总数据
- 用户交互：按需加载详细数据
- 实现方式：使用React.lazy + Suspense

### 实施优先级

1. **立即执行**：方案1（添加WHERE过滤）
   - 影响范围：所有SQL查询
   - 风险：低（只是添加过滤条件）
   - 预计工时：1-2小时

2. **短期执行**：方案2（添加LIMIT）
   - 影响范围：表格组件
   - 风险：低
   - 预计工时：30分钟

3. **中期执行**：方案3（增量加载）
   - 影响范围：全局架构
   - 风险：中（需要重构组件）
   - 预计工时：4-6小时

## 🟠 P2：SQL查询超时（中优先级）

### 问题现象

**位置**：SQL查询Tab
**错误消息**：`查询超时 (超过 30 秒)`
**触发条件**：执行自定义SQL查询

### 根因分析

**当前超时配置**：
```typescript
// 推测位置：src/shared/duckdb/client.ts 或 worker.ts
const QUERY_TIMEOUT = 30000; // 30秒
```

**问题SQL示例**（从UI智能查询助手）：
```sql
SELECT CAST(policy_date AS DATE) as date,
       SUM(premium) as value
FROM PolicyFact
WHERE policy_date >= CURRENT_DATE - INTERVAL 30 DAY
GROUP BY CAST(policy_date AS DATE)
ORDER BY date DESC
```

**瓶颈分析**：
- `CURRENT_DATE - INTERVAL 30 DAY`：计算开销
- `CAST(policy_date AS DATE)`：类型转换（执行2次）
- 无索引支持（DuckDB-WASM限制）

### 优化方案

**方案1：增加超时时间**
```typescript
const QUERY_TIMEOUT = 60000; // 30秒 → 60秒
```
- 优点：简单直接
- 缺点：治标不治本

**方案2：优化SQL模板**
```sql
-- 优化后（避免重复CAST）
WITH daily_data AS (
  SELECT CAST(policy_date AS DATE) as date, premium
  FROM PolicyFact
  WHERE policy_date >= '2025-12-12'  -- 预计算日期
    AND policy_date < '2026-01-11'
)
SELECT date, SUM(premium) as value
FROM daily_data
GROUP BY date
ORDER BY date DESC
```

**方案3：添加查询进度提示**
```typescript
// 用户体验优化
const [queryProgress, setQueryProgress] = useState(0);
// 每5秒更新一次进度条
```
