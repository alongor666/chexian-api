# 路由权限矩阵

> **版本**：v1.0（2026-06-04，多分公司前置 Phase 0A）
> **由 `scripts/check-permission-coverage.mjs` 静态校验，governance 强制**
> **维护协议**：新增顶层 router 必须同步登记到本表 + 在 router 内挂权限中间件，否则 governance 失败

## 一、权限中间件三件套

| 中间件 | 职责 | 强制性 |
|--------|------|--------|
| `authMiddleware` | JWT / PAT 鉴权，注入 `req.user` | 全部 `/api/*` 必挂（除 `/api/auth`/`/api/auth/wecom` 公开） |
| `permissionMiddleware` | 基于 `req.user.role` 注入 `req.permissionFilter` SQL 行级过滤 | 数据型路由必挂 |
| `requireRole(...)` | 限制只允许特定角色访问 | 写操作或管理员路由必挂 |
| `readonlyMiddleware` | PAT 拦截 POST/PUT/DELETE | `/api/data`/`/api/query` 等共用 |

**新规则（Phase 0A 立项）**：每个顶层 router **要么挂 permissionMiddleware（行级隔离），要么挂 requireRole（角色隔离）**，**两者必有其一**。

## 四、fail-closed 加固进度

`|| '1=1'` fallback 在 SQL WHERE 上下文（**非 cache key 上下文**）应改为 `'1=0'`（fail-closed），防止上游路由未挂权限时 SQL 默默返回全量数据。

## 五、操作 SOP

### 新增顶层路由

1. 文件创建后必须 `router.use(authMiddleware)`（除非 `// public-route`）
2. 数据型路由 `router.use(permissionMiddleware)`；管理员型路由 `requireRole(UserRole.BRANCH_ADMIN)`
3. SQL WHERE 上下文用 `req.permissionFilter || '1=0'`（fail-closed）；cache key 上下文保留 `|| '1=1'`
4. 本文件追加一行，标注权限矩阵
5. 跑 `bun run governance:permissions` 验证

### 修复 KNOWN_GAPS

1. 检查"修复条件"是否达成
2. 实现权限隔离
3. 从 `scripts/check-permission-coverage.mjs` 的 `KNOWN_GAPS` 移除条目
4. 本文件移除 KNOWN_GAPS 行
5. 跑 `bun run governance:permissions` 验证
