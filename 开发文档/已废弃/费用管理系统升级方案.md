# 费用分析板块升级方案：费用管理系统（v1）

## 1. 目标与范围

将现有“费用分析”升级为独立的“费用管理系统”，从“看数”升级为“规则制定 + 费用核算 + 费用核对 + 预算跟踪 + 跨系统协同”的闭环平台。

- **当前定位**：分析看板（偏只读）
- **目标定位**：管理系统（可配置、可审批、可追踪、可导出、可对接）
- **建设策略**：先落地可用 MVP，再分阶段扩展企业微信智能表格与业务员 CRM 集成

---

## 2. 角色与层级设计

### 2.1 角色（权限维度）

1. **规则制定者**
   - 制定费用计算规则、预算口径、审批门槛
   - 发起规则版本发布
2. **规则管理者**
   - 维护规则库（生效时间、适用机构、险种范围）
   - 管理规则版本回滚、停用、归档
3. **费用核算者**
   - 按规则执行费用计算
   - 生成机构/业务员费用明细清单
4. **费用核对者**
   - 对核算结果复核、差异标注、确认通过/驳回
5. **分层级查看角色**
   - 按组织层级查看范围内费用、预算执行与异常

### 2.2 层级（数据可见范围）

1. **分公司管理层**
   - 查看成本、预算执行、审批规则
2. **车险部经理**
   - 制定规则、审批规则、查看全量执行效果
3. **费用管理岗**
   - 导入规则、执行核算、复核核算结果
4. **三级机构管理岗**
   - 查看本机构费用、导出清单
5. **业务员（企业微信端）**
   - 查看本人费用、核对状态、预算占用、处理进度

---

## 3. 核心业务能力蓝图

### 3.1 规则中心（Rule Center）

- 规则模板管理（按险种/渠道/机构）
- 规则版本管理（草稿、待审批、生效、停用）
- 生效窗口（起止时间）与冲突校验
- 审批流（提交→审批→发布）

### 3.2 费用核算中心（Calculation Center）

- 批量核算（按日/周/月）
- 重算能力（规则变更后回溯重算）
- 核算日志（输入、规则版本、输出、耗时、异常）

### 3.3 费用核对中心（Reconciliation Center）

- 核算结果复核（通过/驳回）
- 差异对账（上期 vs 本期、规则A vs 规则B）
- 差异原因分类与处理状态跟踪

### 3.4 清单与导出中心（List & Export）

- 三级机构按权限导出费用清单
- 导出模板（机构版、业务员版、财务对账版）
- 导出审计日志（谁在何时导出什么数据）

### 3.5 预算跟踪中心（Budget Tracking）

- 预算目标录入与版本化
- 实际费用 vs 预算的偏差监控
- 超预算预警与规则联动（触发审批）

### 3.6 外部集成中心（Integration Hub）

- 企业微信智能表格：同步费用明细、核对状态、待办任务
- 业务员 CRM：回传费用状态，支持客户维度联动分析（扩展阶段）

---

## 4. 分阶段交付路线图（立即实现 / 扩展实现）

## Phase 1（立即实现，4~6周）：MVP 管理闭环

**目标**：先把“规则→核算→复核→导出”跑通。

- 角色权限：分公司管理层、车险部经理、费用管理岗、三级机构管理岗
- 能力范围：
  - 规则导入（Excel/CSV）+ 基础校验
  - 规则版本发布（手工审批）
  - 月度费用核算 + 核算任务日志
  - 复核状态流转（待复核/已通过/已驳回）
  - 费用清单查询与 CSV 导出

**验收标准**：
- 规则变更可追溯（版本号 + 操作人 + 时间）
- 每月核算任务可重复执行且结果可复核
- 三级机构可按权限导出本机构清单

## Phase 2（近期扩展，6~10周）：企业微信协同

**目标**：让业务员“能看、能核对、能追踪预算”。

- 对接企业微信智能表格：
  - 自动同步个人费用明细
  - 同步核对状态与驳回原因
  - 同步预算占用和预警标识
- 增加业务员视角：
  - 个人费用看板
  - 核对反馈入口

**验收标准**：
- 智能表格字段映射稳定（失败重试 + 异常告警）
- 业务员可在企微端查看和反馈本人费用

## Phase 3（中期扩展，10周+）：CRM 联动与智能化

**目标**：实现“费用管理 + 客户经营”联动。

- CRM 联动：按客户/保单维度关联费用明细
- 智能策略：预算预测、超支预警、规则推荐
- 管理驾驶舱：机构层级多维成本效率分析

**验收标准**：
- 费用数据可回流 CRM，支持客户维度查询
- 预算预测与超支预警准确率达到既定阈值（另行定义）

---

## 5. 推荐系统架构（与现有 API 架构兼容）

- **前端**：新增“费用管理系统”模块（规则中心/核算中心/核对中心/预算中心）
- **后端**：在 `server/src/routes/` 下新增费用管理专用路由簇（保持现有查询路由兼容）
- **数据层**：保留 DuckDB 分析能力；新增规则与核算任务表（建议 PostgreSQL 或同等事务型存储）
- **集成层**：新增企业微信与 CRM 适配器（异步任务 + 重试 + 审计）
- **安全层**：沿用现有 JWT + 权限中间件；扩展到角色+层级的组合授权模型

## 5.1 是否应独立于当前项目？（结论）

**结论：应“业务独立、代码分阶段独立”，不建议第一期物理拆仓。**

### 为什么不是立即拆成新仓

1. 当前平台已具备可复用的认证、组织权限、筛选口径与 API 基建，立即拆仓会重复建设并拉长 MVP 周期。
2. 费用管理第一阶段核心是流程闭环（规则→核算→复核→导出），优先在现有体系内实现可最快验证价值。
3. 企微/CRM 集成初期变更频繁，先在单仓内演进更利于联调和快速迭代。

### 为什么最终仍应独立

1. 费用管理具备独立业务域（规则生命周期、任务调度、审计、审批流），长期会形成独立发布节奏。
2. 权限模型从“看板查询”升级到“规则变更+审批操作”，审计与风控边界更适合独立服务治理。
3. 对外集成（企微、CRM）会引入独立可靠性需求（重试、补偿、限流、告警），建议与分析查询服务解耦。

### 推荐拆分策略（两步走）

- **Step 1（当前~Phase 2）**：
  - 逻辑独立：在当前仓内建立 `expense-management` 领域模块（前后端分层 + 独立路由前缀 + 独立数据表）。
  - 运维不独立：沿用现有部署与认证体系，降低迁移成本。
- **Step 2（Phase 3 起）**：
  - 服务独立：拆分为独立服务（可同仓多服务或新仓），通过 API 网关与 SSO 对接主系统。
  - 数据独立：规则库、核算任务、审计日志独立存储；分析看板通过只读接口消费费用管理结果。

### 触发“正式独立”的客观门槛

满足以下任意 3 条，建议进入独立服务实施：

1. 费用管理迭代节奏与主看板迭代明显不同（连续 2 个迭代周期）。
2. 外部集成（企微/CRM）任务失败重试与补偿逻辑占比显著上升。
3. 审计/合规要求需要独立发布审批或独立数据留存策略。
4. 核算任务对资源占用持续影响主分析查询 SLA。
5. 需要独立对接更多下游系统（>2 个）且变更频率高。

## 6. 关键风险与控制

1. **规则冲突风险**：多版本规则叠加导致核算偏差
   - 控制：规则冲突检测 + 生效窗口校验 + 灰度发布
2. **权限越权风险**：层级隔离不严格
   - 控制：统一权限中间件 + 数据范围过滤 + 导出审计
3. **外部系统不稳定风险**：企微/CRM 同步失败
   - 控制：异步队列、重试机制、失败补偿与告警
4. **口径不一致风险**：管理系统与分析看板口径不一致
   - 控制：统一指标定义与口径字典，变更走审批

---

## 7. 本次结论（供立项决策）

可以按“**MVP先行 + 集成扩展**”推进：
- 先做内部可控闭环（规则、核算、复核、导出、预算跟踪）
- 再打通企业微信智能表格
- 最后联动业务员 CRM，形成费用管理与业务经营一体化平台

该路径兼顾“**尽快可用**”与“**长期扩展**”，能匹配你提出的“有些立即实现，有些扩展”的目标。
