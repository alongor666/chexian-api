# 技术栈审计（tech-stack-audit）

## 0. 定位与铁律

| 铁律 | 说明 |
|------|------|
| **只读审计** | 本 skill 不改任何依赖、锁文件、配置。治理执行永远走"逐 PR + 人工闸门"，本 skill 的第二产物就是那份治理热启动 prompt |
| **实测不背诵** | 所有版本号与安全通告必须来自当场执行的命令输出（outdated / audit），禁止用模型记忆里的"最新版本"或凭印象引用漏洞编号 |
| **暴露面优先于通告等级** | 排期按"生产运行时 / 仅开发机 / 仅安装时"三类暴露面 × 修复成本双轴裁定，不按 critical/high 标签照搬 |
| **报告中文全称** | 报告与回复禁止英文术语缩写堆砌（拒绝服务 / 路径穿越 / 正则拒绝服务 / 跨站脚本等一律中文全称，首次出现可括注原文） |

## 1. 第一步：探测项目形态（约定优于配置，零参数可跑）

1. **包管理器探测**（按锁文件判定，可多种并存）：

   | 锁文件 | 生态 | 取证命令 |
   |--------|------|---------|
   | `bun.lock` / `bun.lockb` | Bun | `bun outdated` · `bun audit` |
   | `pnpm-lock.yaml` | pnpm | `pnpm outdated` · `pnpm audit` |
   | `yarn.lock` | Yarn | `yarn outdated` · `yarn npm audit` |
   | `package-lock.json` | npm | `npm outdated` · `npm audit` |
   | `uv.lock` / `poetry.lock` / `requirements*.txt` | Python | `pip list --outdated` · `pip-audit`（缺工具则登记为缺口，不猜） |
   | `Cargo.lock` | Rust | `cargo outdated` · `cargo audit`（同上） |
   | `go.sum` | Go | `go list -m -u all` · `govulncheck ./...`（同上） |

2. **枚举全部包根**：`find . -name package.json -not -path '*/node_modules/*'`——workspaces 成员与**不在 workspaces 内的独立子包**（各有独立锁文件）都要分别取证，漏掉独立子包是最常见的审计盲区。
3. **读项目协作文档**（CLAUDE.md / AGENTS.md / .claude/rules/）提取三件事：验证命令（构建/测试/治理校验/健康检查）、**升级红线**（如"某依赖须与本地 CLI 小版本兼容""限流配置禁止降低"）、报告语言与归档位置惯例。
4. **运行时一致性盘点**：CI 工作流里的运行时版本声明（setup-node / setup-bun / setup-python 等）↔ 本地实测版本 ↔ package.json `engines` / `.nvmrc`，三方对比。
5. **确定文档漂移核对目标**：用户用 `--doc <路径>` 指定；未指定则按候选顺序探测——`TECH_STACK.md`（任意目录下）→ `ARCHITECTURE.md` → README 的技术栈章节。找不到则在报告中记"无技术栈声明文档"本身作为缺口。

## 2. 第二步：取证（每个包根分别执行）

- 对每个包根跑对应生态的 outdated + audit，**原始输出留存**（报告附录要引用）。
- 锁文件 vs 声明核对：
  - 同名包**多版本共存**（如 zod 4.3.5 与 4.3.6 并存）→ 去重项
  - 声明过松：`^0.x` 在语义化版本下被锁死不浮动、`^1.0.x` 浮动跨度过大导致不同时间安装行为差异 → 收紧项
- 对 audit 命中的每条通告，**grep 找到实际使用点**（哪个源文件 import），没有使用点的传递依赖标注链路。

## 3. 第三步：分级规则（本 skill 的核心方法论）

按**暴露面 × 修复成本**双轴裁定优先级：

| 优先级 | 判定 | 典型 |
|--------|------|------|
| **P0** | 生产运行时暴露 **且** 语义化版本区间内即可修（一句 update） | Web 框架补丁版修高危 |
| **P1** | 严重/高危但仅开发环境；或所用版本线**已被上游弃用不再发补丁**（需小型迁移） | 测试框架严重级通告、弃用的 1.x 上传库 |
| **P2** | 大版本落差，需专项回归；**逐条标注项目红线约束**（从第一步第 3 点提取） | 构建工具跨 3 个大版本、数据库驱动小版本兼容约束 |
| **P3** | 治理修补：engines 声明、lint 决策、文档漂移修订、脚本语言依赖声明文件 | 随任意 PR 顺带 |

暴露面三分法：
- **生产运行时**：进生产进程的依赖 → 最高权重
- **仅开发机**：测试/构建/热重载工具链 → 降一档，但严重级仍须排期
- **仅安装时**：原生模块预编译下载链（node-pre-gyp 类）→ 再降一档，随宿主包大版本一并消除

特别规则：前端打包产物中经由导出类库（表格/PDF 库）传入的 **Node 侧才可利用**的漏洞（临时文件路径穿越、通配符正则拒绝服务等），实际风险降级标注；若上游已停滞无法靠升级根治，给出"替换 / 接受现状"二选一并陈述理由，不悬置。

## 4. 第四步：文档漂移核对

对第一步锁定的技术栈声明文档逐条核对：

1. 文档声称的每个依赖 + 版本 ↔ package.json 与锁文件实测——**声称存在但全仓库 grep 零命中的依赖 = 高严重度（虚构依赖）**
2. 文档中的目录结构 / 模块数量清单 ↔ `find ... | wc -l` 实测，给出失真比例
3. 该文档与项目其他必读文档（CLAUDE.md 等）**互相矛盾**之处
4. 修订建议优先"指针 + 生成"模式：版本以 package.json 为唯一事实源、结构清单指向代码索引，消除再次靠人手同步的根因

## 5. 第五步：工具链缺口清单

- lint / format 工具存在性；**指向不存在目标的占位脚本**（如子包 lint 脚本写"用根级 lint"而根包没有）
- `engines` / `.nvmrc` / CI 运行时三方一致性（接第一步第 4 点）
- 脚本语言（Python 等）依赖声明文件缺失 → 新环境不可复现
- 锁文件是否全部提交、各包根 typescript 等共享工具版本是否一致

## 6. 第六步：产出（两件）

### 产物一：审计报告（中文）

- 归档位置按项目惯例（如 `开发文档/reviews/` 或 `docs/reviews/`），命名 `YYYY-MM-DD-技术栈审计.md`；项目有文档索引的**当场登记**
- 模板结构（与正文各步对应）：
  1. **结论先行**：总体健康判断一句话 + 问题表（优先级/问题/影响）
  2. 运行时与包管理（一致性盘点表）
  3. 依赖版本现状：每包一张三列表（实际安装 → 最新 → 落差 + 备注），备注里写使用点与红线约束
  4. 安全审计：按暴露面分组的处置表，给总数（X 条：严重/高危/中危/低危）
  5. 文档漂移：声称 vs 实际对照表 + 严重度
  6. 工具链缺口表
  7. 升级路线 P0-P3（每级给验证命令）
  8. 附录：取证命令原文

### 产物二：治理热启动 prompt

生成一份可直接粘贴到新会话的执行 prompt，固定骨架：

- **唯一事实源**：指向刚产出的审计报告路径，要求执行前 outdated/audit 复测防漂移
- **开工第一步**：worktree 隔离命令（若项目有 worktree 纪律则按其规则）
- **分 PR 拆分**：P0 一个 PR；P1 每项独立 PR（标注使用点文件与回归要求）；P3 可并入 P0
- **每个 PR 的验证命令**：从项目协作文档提取的真实命令（构建/测试/治理/健康检查 + 至少一个真实接口请求取证）
- **遇 peer 依赖冲突停下裁决条款**：禁止顺手把 P2 大版本带进 P0/P1 的 PR
- **禁止清单**：全部 P2 项 + 项目红线逐条列出

## 7. 禁止

- ❌ 执行任何 `update` / `install` / 改锁文件——那是治理会话的事，本 skill 只读
- ❌ 用模型记忆中的版本号或漏洞编号替代实测输出
- ❌ 按通告等级照搬排期而不做暴露面分析
- ❌ 把大版本升级塞进 P0/P1
- ❌ 报告里英文术语缩写堆砌
