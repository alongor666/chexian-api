## 一、SQL 生成层：模板去重与可组合化（P0）

### 问题

SQL 模板字符串大量重复，同一逻辑散布在多个文件中：

| 重复模式 | 涉及文件数 | 示例 |
|---------|-----------|------|
| 已赚天数计算（earned days `DATEDIFF`） | 4 文件 | `kpi.ts`, `cost-ratios.ts`, `claims-detail.ts`, `forecast/baseline.ts` |
| `LEFT JOIN ClaimsAgg` 赔款聚合 | 12 文件 | 几乎所有涉及赔付率的生成器 |
| 保单去重 CTE | 多文件独立实现 | 已有 `shared/policy-dedup.ts` 但未被一致调用 |
| 业务条件 WHERE 子句 | 广泛 | `shared/business-conditions.ts` 已抽取但覆盖不完整 |

**风险**：口径修改时需同步改 4-12 个文件，遗漏即产生数据不一致。BACKLOG B299（赔款分子窗口对齐）即此类债务。

### 优化方向

1. **CTE 片段库**（`server/src/sql/shared/cte-fragments.ts`）
   - 抽取 `earnedDaysCTE(cutoffDate)` / `claimsAggCTE(windowConfig)` / `policyBaseCTE(dedup, filters)` 等可组合片段
   - 各生成器通过 `import { earnedDaysCTE } from './shared/cte-fragments'` 引用，不再内联
   - 单一修改点 = 口径变更只改一处

2. **SQL Builder 增强**
   - 现有 `sql-builder.ts` 仅做基础拼接；升级为支持 CTE 链式组合：
     ```typescript
     buildQuery()
       .withCTE(policyBaseCTE(filters))
       .withCTE(claimsAggCTE({ window: 'cutoff' }))
       .select(getMetricSql('loss_ratio'))
       .groupBy(dimensionKey)
       .build()
     ```
   - 编译期校验：CTE 依赖关系（ClaimsAgg 依赖 PolicyBase）

3. **测试覆盖提升**
   - 当前 38%（23/60）→ 目标 70%+
   - 优先覆盖：有赔款口径的生成器（涉及 B299 对齐）

### 工作量估算

| 子任务 | 估时 |
|--------|------|
| CTE 片段抽取（earned days + claims agg + policy base） | 2-3 天 |
| 各生成器迁移到片段引用（12 文件） | 2 天 |
| SQL Builder 增强 | 3 天 |
| 补充测试 | 2 天 |

## 二、路由处理器：统一执行管线（P1）

### 问题

23 个路由文件存在三种不一致的执行模式：

```
模式 A：直接查询    → duckdbService.query(sql) → res.json()
模式 B：带缓存      → withRouteCache() → sendWithEtag()
模式 C：带立方体回退 → tryCostCube() ?? duckdbService.query()
```

每个路由独立处理：参数校验（有 Zod / 无 Zod 混杂）、缓存键构建、权限检查（`requireBranchAdmin` / `resolveBranchRlsCode` 重复调用）、错误处理。

### 优化方向

1. **QueryRouteBuilder 模式**
   ```typescript
   // 声明式路由定义，替代命令式重复代码
   router.get('/kpi', queryRoute({
     cacheKey: 'kpi',
     cacheTTL: hotspotMedium,
     params: KpiParamsSchema,       // Zod schema（统一校验）
     rls: 'auto',                   // 自动注入 permissionFilter
     cubeFirst: 'kpi-cost-cube',    // 可选：立方体优先策略
     query: (params, rls) => generateKpiQuery(rls, params),
   }))
   ```

2. **参数校验标准化**
   - 所有路由统一使用 `route-param-contracts.ts` 中的 Zod schema
   - 消除内联校验逻辑

3. **4 域 RLS 漏洞修复**（当前 P0 安全事项）
   - `customer-flow`、`quote-conversion`、`claims-detail`、`repair` 路由缺少 `permissionFilter` 注入
   - 当前用 `requireBranchAdmin()` 临时门控，需正式注入 WHERE 子句

### 工作量估算

| 子任务 | 估时 |
|--------|------|
| QueryRouteBuilder 基础设施 | 2 天 |
| 迁移高频路由（kpi/trend/cost/growth） | 2 天 |
| 迁移剩余路由 | 3 天 |
| 4 域 RLS 修复 | 2 天 |

## 三、前端大组件拆分与数据获取标准化（P2）

### 问题

**大组件**（>500 行，15+ 个）：

| 组件 | 行数 | 混合职责 |
|------|------|---------|
| `GeoRiskPanel.tsx` | 951 | 地图渲染 + 数据获取 + 状态管理 |
| `CrossSellAnalysisPanel.tsx` | 891 | 下钻状态 + 表格渲染 + 筛选逻辑 |
| `NewEarnedPremiumTable.tsx` | 891 | 数据转换 + UI 渲染 + 排序逻辑 |
| `PerformanceAnalysisPanel.tsx` | 882 | 多子面板 + 下钻状态 |
| `AdvancedFilterPanel.tsx` | 728 | 筛选 UI + 状态 + 复杂依赖链 |

**数据获取不统一**：43 个文件直接调用 `apiClient`（101 处），仅 27 个文件使用 React Query 的 `useQuery`（71 处）。缺乏统一的加载/错误/重试模式。

### 优化方向

1. **大组件拆分**（逐个治理，每个拆为 3-4 个子组件）
   - 分离关注点：数据获取 hook / 展示组件 / 状态管理
   - 优先拆 `GeoRiskPanel`（951 行，最大）和 `CrossSellAnalysisPanel`（891 行）

2. **数据获取标准化**
   - 将直调 `apiClient` 的组件迁移到 `useQuery` 模式
   - 建立 `createQueryHook(apiMethod, queryKeyFn)` 工厂函数，统一：
     - 加载状态管理
     - 错误重试策略
     - 缓存失效（ETL 更新时）
     - TypeScript 类型推导

3. **路由级懒加载**
   - 当前仅 3 处 `React.lazy`；21 个 feature 模块全部同步导入
   - 按 feature 拆分路由级 chunk，减少首屏加载体积

### 工作量估算

| 子任务 | 估时 |
|--------|------|
| 前 5 大组件拆分 | 5 天 |
| 数据获取 hook 工厂 + 迁移 | 3 天 |
| 路由级懒加载（21 个 feature） | 2 天 |

## 四、构建产物优化（P3）

### 问题

- **导出依赖加载过早**：`jspdf`（~280KB）+ `html2canvas`（~160KB）打入主 bundle，但仅导出功能使用
- **vendor chunk 粒度粗**：`vendor-data` 把 `date-fns`（40KB）和 `exceljs`（~800KB）打在一起
- **无 feature 级代码分割**：21 个 feature 模块全部打入同一 chunk

### 优化方向

1. **动态导入导出依赖**
   ```typescript
   // 现在：import jsPDF from 'jspdf'（同步，主 bundle）
   // 优化：const { default: jsPDF } = await import('jspdf')（按需）
   ```
   预计减少主 bundle ~440KB

2. **细化 vendor chunk**
   ```typescript
   manualChunks: {
     'vendor-react': ['react', 'react-dom', 'react-router-dom'],
     'vendor-echarts': ['echarts', 'echarts-for-react'],
     'vendor-date': ['date-fns'],
     'vendor-excel': ['exceljs'],      // 仅导出时加载
     'vendor-export': ['jspdf', 'html2canvas'],  // 仅导出时加载
     'vendor-ui': ['lucide-react'],
   }
   ```

3. **feature 级 code splitting**（与第三节路由懒加载配合）

### 工作量估算

| 子任务 | 估时 |
|--------|------|
| 导出依赖动态导入 | 0.5 天 |
| vendor chunk 细化 | 0.5 天 |
| feature 级 code splitting | 2 天（与 §三 合并） |

## 五、配置治理（P3）

### 问题

- `query-routes-metadata.ts`（864 行）单文件编码路由→SQL 映射、字段访问策略、缓存 TTL 配置，职责过重
- 访问控制相关配置分散在 `sql-federation-policy.ts`（12.4 KB）、`organizations.ts`（12.1 KB）、`preset-users.ts`（15.9 KB）三个文件

### 优化方向

1. **按域拆分路由元数据**
   - `query-routes-metadata/kpi.ts`、`query-routes-metadata/cost.ts` 等
   - 各域文件与对应 SQL 生成器同目录，降低认知距离

2. **访问控制聚合**
   - 将 "谁能看什么" 的逻辑从三个文件收敛到一个 `access-control-config.ts`
   - 或至少建立 barrel export 让调用方只需一个 import

### 工作量估算

| 子任务 | 估时 |
|--------|------|
| 路由元数据按域拆分 | 1.5 天 |
| 访问控制配置聚合 | 1 天 |

## 执行优先级总览

```
          影响
    高 ┌───────────────────────┐
       │  P0 SQL去重    P1 路由管线 │
       │                       │
       │  P2 组件拆分   P2 测试补全 │
       │                       │
       │  P3 构建优化   P3 配置治理 │
    低 └───────────────────────┘
         低 ──── 工作量 ──── 高
```

| 优先级 | 方向 | 核心收益 | 估时合计 |
|--------|------|---------|---------|
| **P0** | SQL 模板去重与可组合化 | 消除口径不一致风险，降低维护成本 | ~9 天 |
| **P1** | 路由执行管线统一 | 消除样板代码 + 修复 RLS 漏洞 | ~9 天 |
| **P2** | 前端组件拆分 + 测试补全 | 提升可测试性和首屏性能 | ~10 天 |
| **P3** | 构建产物 + 配置治理 | 减少 bundle 体积 ~440KB，降低认知负荷 | ~5 天 |

**建议执行顺序**：P0 → P1 → P3（低成本快赢） → P2（工作量最大但可渐进）

## 不建议做的事

| 不建议 | 原因 |
|--------|------|
| 引入 ORM 替代 SQL 生成器 | DuckDB 分析型查询（CTE 链/窗口函数）不适合 ORM 抽象 |
| 前端迁移到 SSR/RSC | 内部分析平台，无 SEO 需求，CSR 足够 |
| 微服务拆分 | 单体 Express + DuckDB 是当前规模的最佳选择 |
| 替换 React Query 为其他状态管理 | 已安装且正确方向，问题是使用不充分而非工具选择错误 |
