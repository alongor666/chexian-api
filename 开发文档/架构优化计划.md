## 一、现状总览

| 维度 | 现状 | 成熟度 |
|------|------|--------|
| **前端架构** | React + TS + Vite，21 个 features 模块，lazy loading 已覆盖页面级 | ★★★★☆ |

| **数据层** | Parquet 仓库 + 惰性域加载 + 物化引擎 + 连接池 | ★★★★☆ |

## 二、优化方向与计划

### P0：高优先级（直接影响用户体验或开发效率）

#### 2.1 前端大组件拆分

**现状**：6 个组件超过 700 行，最大 1,401 行（PerformanceAnalysisPanel.tsx），维护成本高、渲染粒度粗。

| 文件 | 行数 | 问题 |
|------|------|------|
| `src/features/dashboard/PerformanceAnalysisPanel.tsx` | 1,401 | God Component，re-render 爆炸半径大 |
| `src/features/claims-detail/components/GeoRiskPanel.tsx` | 951 | 地图+表格+筛选混合 |
| `src/features/dashboard/CrossSellAnalysisPanel.tsx` | 891 | 图表+热力图+趋势混合 |
| `src/features/cost/components/NewEarnedPremiumTable.tsx` | 891 | 复杂表格逻辑 |
| `src/features/admin/AccessControlPage.tsx` | 770 | 权限管理多表单 |
| `src/features/filters/AdvancedFilterPanel.tsx` | 728 | 筛选器逻辑密集 |

**优化方案**：
1. 将 >800 行组件拆为 container + 多个 memo 化子组件（图表区/表格区/筛选区/汇总区）
2. 子组件用 `React.memo()` 包裹，隔离 re-render
3. 提取 unstable 引用到 `useMemo` / `useCallback`，稳定化 props
4. 预期收益：筛选器变化时 re-render 减少 30-50%

**验证方式**：React DevTools Profiler 对比拆分前后的渲染次数和耗时

---

#### 2.2 ECharts / ExcelJS 按需加载

**现状**：`echarts`（~600KB gzip）和 `exceljs`（~400KB gzip）通过 `manualChunks` 打入独立 vendor chunk，但**所有页面都会加载**，即使用户从不访问图表或导出功能。

**优化方案**：
1. 将 `vendor-echarts` 从 manualChunks 移除，改为路由级 `React.lazy()` 按需加载
2. `exceljs` 已在 vendor-data chunk 中——改为动态 `import()`（参考 jspdf 已有的延迟导入模式 `src/shared/export/pdfExporter.ts:38-46`）
3. 仅在用户点击「导出 Excel」或进入含图表页面时才加载

**预期收益**：首屏 JS 体积减少 ~1MB（gzip 后约 200-300KB），首屏加载提速 0.5-1s

**验证方式**：`bun run build` 后对比 chunk 分布；Lighthouse Performance 评分

---

#### 2.3 Workspace 统一 server 子包

**现状**：`package.json` workspaces 仅含 `["cli", "mcp"]`，`server/` 未纳入。导致 `typescript`、`vitest`、`zod` 等在 root 和 server 各装一份。

**优化方案**：
1. 将 `"server"` 加入 root `package.json` 的 `workspaces` 数组
2. 将 server 与 root 共享的 devDependencies（typescript, vitest）提升到 root
3. 清理 `vite.config.ts` 中 `modules: ['node_modules', 'server/node_modules']` 的兼容路径

**预期收益**：node_modules 体积减少 ~15-20%，`bun install` 提速

**验证方式**：`du -sh node_modules server/node_modules` 前后对比

---

#### 2.4 CSP 安全强化

**现状**：`server/src/app.ts:54` 的 `scriptSrc` 同时允许 `'unsafe-inline'` 和 `'unsafe-eval'`，大幅削弱 CSP 对 XSS 的防护。ECharts 5.x 不再需要 eval。

**优化方案**：
1. 验证去掉 `unsafe-eval` 后前端功能是否正常
2. 用 nonce-based CSP 替代 `unsafe-inline`（Vite 支持生成 nonce）
3. 在 `permissionMiddleware` 中增加 organization 值白名单校验（对比 PRESET_USERS）

**验证方式**：E2E 测试全路径通过 + 浏览器控制台无 CSP violation

---

#### 2.5 cost/growth 页面 Bundle 路由

**现状**：已有 3 个 bundle 路由（dashboard/performance/cross-sell），但 cost 和 growth 页面仍发 6-10 个独立请求。高延迟网络下体验明显。

**优化方案**：
1. 为 cost 页面创建 cost-bundle 路由，聚合成本率/费用率/综合成本率等查询
2. 为 growth 页面创建 growth-bundle 路由，聚合增长趋势/环比/同比等查询
3. 参考 `server/src/routes/query/bundles/dashboard.ts` 的 `Promise.all()` 并行模式

**预期收益**：相关页面首屏请求从 6-10 个减少到 1-2 个，高延迟网络下加载提速 1-3s

---

#### 2.6 RouteConcurrencyGate 推广

**现状**：`RouteConcurrencyGate`（`server/src/routes/query/kpi.ts:46-50`）仅在 KPI 路由使用。其他重查询（comprehensive bundle、cross-sell bundle）无并发门控，高并发时可能打满 DuckDB 连接池。

**优化方案**：
1. 将 `RouteConcurrencyGate` 提升为通用中间件工厂
2. 在 bundle 类路由和 cost 路由启用
3. 配合连接池的 `waitQueueLimit: 32` 形成双重保护

---

### P1：中优先级（提升工程质量和可维护性）

#### 2.7 构建护栏强化

**现状**：`chunkSizeWarningLimit: 1000`（1MB），是行业推荐值（250-500KB）的 2-4 倍。构建需 4GB 堆内存（`--max-old-space-size=4096`），暗示 tree-shaking 有提升空间。

**优化方案**：
1. 降低 `chunkSizeWarningLimit` 至 500KB，暴露过大 chunk
2. CI 中增加 bundle size 回归检测（对比 main 分支 chunk 大小，超 10% 警告）
3. 审查 `optimizeDeps.include` 列表，移除不再需要预打包的依赖

**验证方式**：`bun run build` 零警告

---

#### 2.8 SQL 生成器复杂度治理

**现状**：60 个 SQL 生成器文件，4 个超过 500 行（claims-detail 624 行、performance-analysis/shared 584 行、claims-heatmap 574 行、performance-heatmap 492 行），存在跨文件复制 CTE 片段的隐患。

**优化方案**：
1. 提取共享 CTE 模板到 `server/src/sql/shared/` 目录（已有 shared 目录，扩展使用）
2. 对 claims-heatmap 的多 cutoff 累计计算考虑物化快照（周/月粒度预聚合）
3. 给 sql-builder.ts 增加 SQL 模板组合能力，减少各生成器中重复的 WHERE/GROUP BY 拼接

**预期收益**：减少 SQL 生成器平均行数 15-20%，降低口径不一致风险

**验证方式**：`duckdb` 直查与 API 返回对比（CLAUDE.md §6 验证协议）

---

#### 2.9 可观测性增强

**现状**：已有慢查询阈值（`SLOW_QUERY_THRESHOLD_MS`）、路由缓存命中率统计、连接池状态快照。但缺乏系统性的指标聚合和趋势可视化。

**优化方案**：
1. 增加 `/api/metrics` 端点，聚合展示：
   - 查询 P50/P95/P99 耗时（按路由分组）
   - 缓存命中率趋势（route-cache + query-cache 双层）
   - 连接池饱和度（active/idle/waiting）
   - 内存使用趋势（heap + RSS）
2. 在现有 `/health` 端点中增加 `cacheHitRate` 和 `avgQueryLatencyMs` 字段
3. 对接现有的 `request-context.ts` 中的 `recordQueryMetric`，补充聚合逻辑

**预期收益**：运维可见性提升，问题定位从"看日志"到"看仪表盘"

---

#### 2.10 TypeScript 严格度提升

**现状**：root `tsconfig.json` 已开 `strict: true`，但缺少 `noImplicitAny` 和 `exactOptionalPropertyTypes`。

**优化方案**：
1. 启用 `noImplicitAny: true`（root tsconfig）
2. 逐步启用 `exactOptionalPropertyTypes: true`（先在 server tsconfig 试行）
3. 修复新增的类型错误（预计 50-100 处）

**预期收益**：在编译期捕获更多潜在 bug

**验证方式**：`bun run typecheck` 零错误

---

### P2：低优先级（锦上添花，可长期推进）

#### 2.11 app.ts 职责分离

**现状**：`server/src/app.ts`（418 行）混合了健康检查路由逻辑（118-166 行，含立方体状态/影子对账/连接池快照等）、内部数据指纹端点、12 条路由挂载、startServer 启动编排、gracefulShutdown 关闭编排。

**优化方案**：
1. 将 `/health`、`/internal/data-fingerprint`、`/api/health/detail` 提取到 `routes/health.ts`
2. 将 startServer / gracefulShutdown 提取到 `server/src/lifecycle.ts`
3. app.ts 只保留中间件链和路由挂载（目标 <150 行）

---

#### 2.12 双层缓存内存预算优化

**现状**：QueryCache（256MB 上限）+ RouteCache（400MB 上限，存三份 buffer：jsonBuffer + brotliBuffer + gzipBuffer），总计可占 900MB+。VPS 仅 4GB，DuckDB 自身需 1.5GB。

**优化方案**：
1. 降低 RouteCache 到 200MB（`ROUTE_CACHE_MAX_BYTES`），旧条目由 dataVersion 驱动失效后占据无效内存
2. 或在 dataVersion 变更时主动清空 route-cache（cache-warmer 已能覆盖关键路由预热，cold cliff 风险可控）

---

#### 2.13 Service Worker 增强

**现状**：SW 仅管理 `/api/query/*` GET 的 stale-while-revalidate 缓存 + ETL 版本检测。ETL 更新时 `queryClient.invalidateQueries()` 一刀切清空所有缓存。

**优化方案**：
1. ETL 更新时按域选择性失效（只清受影响的查询键），避免无关页面重新加载
2. SW 增加预取能力：用户登录后预取 dashboard bundle，减少首次导航延迟
3. 添加 `removeEventListener` 清理（`main.tsx:34`），避免 HMR 重复注册

---

#### 2.14 测试覆盖率提升

**现状**：117 个测试文件，但大型前端组件（dashboard panels、cost tables）覆盖不足。集成测试仅 4 文件。

**优化方案**：
1. 为 P0 拆分的大组件补充组件级测试
2. 为高频 SQL 生成器（kpi、cost、trend）补充快照测试（SQL 输出稳定性）
3. 在 CI 中增加覆盖率门禁（新增代码 ≥80% 行覆盖）

---

#### 2.15 路由层按域懒加载

**现状**：`query.ts` 聚合器在启动时同步 import 全部 23 个子路由模块，即使部分路由几乎无流量。

**优化方案**：
1. 低频路由（patrol、policy-geo、renewal-tracker、sql-passthrough）改为 `await import()` 延迟加载
2. 配合现有 `LazyDomainRegistry` 机制，路由+数据域双惰性

**预期收益**：冷启动时间减少 100-200ms

## 四、风险与约束

| 风险 | 缓解措施 |
|------|----------|
| 大组件拆分可能引入回归 | 拆分前跑 golden-baseline，拆分后 `--compare` 零差异 |
| ECharts 懒加载导致图表首次闪烁 | 添加 Skeleton 占位 + 预加载 hint |
| Workspace 变更影响 CI | 在 worktree 中验证 `bun install` + `bun run verify:full` |
| SQL 共享模板影响现有口径 | 按 CLAUDE.md §6 验证协议，`duckdb` 直查对比 |

## 五、已有优势（无需优化）

以下领域架构成熟，短期内不建议动：

- **三级缓存体系**：route-cache（预序列化+预压缩）→ query-cache（LRU+字节上限）→ dashboard 硬缓存。设计精良，命中率高
- **连接池**：10 连接上限 + 2s 超时 + 32 等待队列 + 饱和度检测，防御性设计完备
- **认证体系**：JWT + PAT + Cookie 三路认证 + 三级限流（基线 100/200 + PAT 60/min），安全基线高
- **数据启动编排**：惰性域注册 + 分批物化 + 降级回退到 VIEW，容错设计优秀
- **React Query 配置**：SW 活跃时 staleTime=Infinity、登出清缓存防跨用户残留，细节到位
