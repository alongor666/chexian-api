## 4. chexian-api 侧收口

### 4.1 决策⑤：软链是否进 git

**采用**：项目侧不在 git 里保留任何软链或实体文件，消费完全走 `sync-skills` 装到
`~/.claude/skills/` 的全局软链。

**理由**：
1. 与仓内既有全部共享技能（`chexian-daily-loop` / `chexian-ir-diagnosis` / `chexian-channel` 等）
   消费模式完全一致，不新增第二套机制。
2. 软链是绝对路径（`~/.claude/skills/<name>` → `<技能仓 clone 路径>/skills/<name>`），进 git 会
   把开发者本机路径写死进版本库，不可移植。
3. 新机器接入路径已是既有惯例：clone 技能仓 + 跑 `sync-skills.sh link`，无需为这 14 个技能引入
   新的引导步骤。

### 4.3 治理闸适配

`scripts/governance/skill-frontmatter.mjs` 原只扫描 `.claude/skills/*.md` 扁平文件。
经评审返工（P1-3：初版仍接受实体文件/目录，未兑现政策语义）升级为**实体必红灯**：

- `.claude/skills/` 为空或不存在 → 优雅通过（迁移后的预期常态）。
- **`fs.lstatSync` 区分实体与软链**：实体目录、实体 `.md` 文件一律报错（项目内禁实体技能，
  绕铁律的唯一入口直接拦）；实体非 `.md` 杂项文件（`.gitkeep` 等）跳过。
- 软链条目解析目标：悬空链报错；目标为 `.md` 文件（扁平）或含 `SKILL.md` 的目录
  （sync-skills 直连标准形态）→ 校验 frontmatter 契约（name=stem、description 含触发语义、
  可解析）；软链目录无 `SKILL.md` 视为非技能目录跳过。

配套红绿夹具测试 `scripts/__tests__/skill-frontmatter.test.mjs` 重构为 18 个用例：实体扁平/
实体目录/实体非技能目录必红（即使内容合规）、软链两种形态合规通过、frontmatter 五类违规经
软链继续被拦、悬空链红灯、空目录/无目录通过。
