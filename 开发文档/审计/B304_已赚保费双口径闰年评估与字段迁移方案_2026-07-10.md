## 二、闰年差异量化（DuckDB 直查 Parquet 实证）

数据源：`数据管理/warehouse/fact/policy/current/SC/*.parquet`（四川，263.5 万行，签单至 2026-07-09；
山西走同一套生成器代码，口径结论相同）。筛选：交强险+商业保险、起保日非空，与生成器一致。

### 2.1 闰年跨越保单体量

保险期限跨 2 月 29 日的保单（`policy_term = 366`）：**445,808 行 / 433,472 张保单 / 保费 38,615.4 万元**，
约占全账本保费（223,869.6 万元）的 **17.2%**。起保日全部在 2024-02-28 及之前（2024 与 2020 闰年批次）；
下一批产生于 2027-03-01 起（跨 2028-02-29）。

### 2.2 财务口径：364 天封顶 vs 1 年封顶（差异 = 时间分摊部分的最后 1 个在保日）

口径三要素——分子：两种终保日约定下滚动 12 月已赚保费之差；分母：同窗口 364 天约定的已赚保费总额；
时间区间：滚动 12 月窗口 `[统计日−364 天, 统计日]`。完整复算生成器公式（首日费用 P·F·α·I + 时间分摊
P·(1−F)·窗口内天数/policy_term），仅终保日封顶从 `起保+364 天` 换成 `起保+1 年−1 天`：

| 统计截止日 | 窗口内闰年保单行数 | 364 天约定已赚（万元） | 1 年约定已赚（万元） | 差额（万元） | 差额占比 |
|---|---:|---:|---:|---:|---:|
| 2024-06-30 | 445,788 | 36,101.79 | 36,137.66 | +35.87 | +0.099% |
| 2024-12-31 | 445,788 | 36,861.59 | 36,933.13 | +71.54 | +0.194% |
| 2026-07-09（当前生产） | **0** | 42,145.44 | 42,145.44 | **0** | **0%** |

- **当前生产数值零差异**：今日滚动窗口内（起保 ≥ 2024-07-12）没有任何闰年跨越保单。
- **闰年批次在保期峰值差异约 0.2%**（71.54 万元/窗口）。差异来源：闰年保单第 366 个在保日
  （起保+365 天）被 364 天终保日约定永久截掉。
- **终身欠计**：闰年保单时间分摊部分终身只赚到 365/366，欠计 = Σ P·(1−F)/366 =
  **84.41 万元**（SC 全部 43.3 万张闰年保单口径）。该部分收入在财务口径下任何统计日都不出现。

### 2.3 反方向统一（把时间分摊口径改成固定 365 天）的代价——证伪

若反向统一（满期保费退回固定 365 天分母），@统计日 2024-06-30 全账本满期保费从 117,566.47 万元
**虚高到** 117,604.63 万元（+38.16 万元 / +0.0325%，全部来自闰年保单）→ 满期赔付率分母虚大、
赔付率被同比例低估。与业务规则字典"旧版固定 365 偏高"的既有结论一致，**方向被证伪**。

### 2.4 建议（待用户拍板，未实施）

**统一到闰年感知（+1 年）**，即把 `earned-premium.ts` 两个滚动 12 月函数的终保日
`起保日 + INTERVAL 364 DAY` 改为 `起保日 + INTERVAL 1 YEAR − INTERVAL 1 DAY`（共 3 处表达式 + 2 处窗口过滤）。理由：

1. 业务规则字典与指标注册表（`metric-registry/categories/cost.ts` `earned_premium` v2.0.0）均以闰年感知为权威口径；
2. 同文件内部已自相矛盾（分母 `policy_term` 闰年感知、终保日却固定 364），统一后消除 84.41 万元终身欠计；
3. V3 矩阵（`sql-builder.ts`）已是 1 年约定，财务口径家族内部先自洽；
4. **改动窗口极佳**：当前至 2027-02 生产数值零变化（滚动窗口内无闰年保单），2027-03 前任何发布上线都无对账扰动，且改后 `.planning/golden-baseline/` 黄金基线对比应零差异（可作为回归证据）。

风险与保留意见：若"固定 365 天在保"是财务报表侧的有意约定（外部报表对齐），则应保留 364 并在
JSDoc 标注"有意为之"——这是**业务判断，代码侧无法裁决**，故本任务不直接改。

## 三、字段命名区分迁移方案（API 破坏性变更，供拍板）

### 3.1 现状：机器层无同名冲突，冲突在语义层

grep 实证（2026-07-10，worktree HEAD）两口径输出字段**并不重名**：

| 口径 | 输出字段 | 前端消费文件数 |
|---|---|---:|
| 财务·滚动12月 | `earned_premium_cum` `total_earned_premium` `earned_ratio` `first_day_part` `time_part` | 7（全部在 `src/features/cost/`） |
| 财务·V3矩阵/明细 | `earned_01..earned_12` `earned_total` `earned_from_prev` `earned_from_curr` `total_earned_premium` `earned_ratio` | 4（同上目录） |
| 时间分摊（满期保费） | `earned_premium`（指标注册表 id，另有 `earned_claim_ratio` 等衍生） | **40+ 文件**，横跨前端 cost/claims-detail/comprehensive/chart-ledger、agent 注册表、skills、cube、forecast |

真正的风险：同一路由 `/api/query/cost` 靠 `type` 参数区分两口径，返回体里两种"已赚保费"都无口径前缀，
人/大模型下游（PAT + CLI + MCP 直读 JSON 字段）容易混用。

### 3.2 方案 A：零破坏（类型层 + 元数据标注）——**推荐先行**

不改任何 JSON 字段名。做三件事：
1. 前端类型层已按文件分开（`earned-premium-basic.ts` / `new-earned-premium.ts` / `cost-data.ts`），补 JSDoc 口径标注；
2. `query-routes-metadata.ts` 给 `/cost` 路由的 `timeWindowNote` 补一句两口径区分提示（MCP tool description 自动透出给大模型消费方）；
3. 两生成器文件头 JSDoc 已有对照表（2026-05-31 完成），本次修正其中过时的"days/365"格。

成本≈0，收益=挡住人/大模型混用的主通道。**已发布字段不动，PAT/MCP/CLI 下游零影响。**

### 3.3 方案 B：字段重命名 + 兼容层（真破坏性变更）

仅财务口径改名（时间分摊口径 `earned_premium` 消费面 40+ 文件且与注册表 id、业务字典"满期保费"绑定，**不动**）：

| 旧字段 | 新字段 | 影响路由 |
|---|---|---|
| `earned_premium_cum` | `fin_earned_premium_cum` | `/cost?type=earned` |
| `total_earned_premium` | `fin_earned_premium_total` | `type=earned` / `earned-new` / `expense-forecast` |
| `earned_ratio` | `fin_earned_ratio` | 同上 |
| `earned_from_prev` / `earned_from_curr` | `fin_earned_from_prev` / `fin_earned_from_curr` | `type=earned-new` |
| `earned_01..12` / `earned_total` | `fin_earned_01..12` / `fin_earned_total` | `type=earned-new` |

迁移步骤（两个发布周期）：
1. **发布 1（双写）**：SQL 层同值双别名（或路由层响应后处理补新字段），旧新字段并存；前端全部切新字段
   （精确消费点清单见 §3.4，约 11 个文件、60+ 处引用）；route-catalog 描述标注"旧字段 YYYY-MM-DD 废弃"。
2. **观察期**：审计日志（`logs/audit.log` 的 PAT `token_id`）确认无外部消费方仍依赖旧字段。
3. **发布 2（删除）**：移除旧别名；governance 加防回归 grep（禁止旧字段名再出现在 `server/src/sql/cost/`）。

### 3.4 下游调用方清单（grep 实证，测试文件除外）

- `earned_premium_cum`：`EarnedPremiumCharts.tsx`(71,78)、`EarnedPremiumTable.tsx`(57,98,221)、`useExportHandlers.ts`(200)、`cost-data.ts`(147)、`table-columns.ts`(151)、`transformData.ts`(139,178)
- `total_earned_premium`：上述文件 + `ExpenseRatioForecastPanel.tsx`(46,63,112,216)、`NewEarnedPremiumTable.tsx`(89,148,200,433)、`cost-summary-calc.ts`(165,178,207,219)、`earned-premium-basic.ts`(36)、`expense-forecast.ts`(38)、`new-earned-premium.ts`(92)、`table-columns.ts`(165)
- `earned_ratio`：`EarnedPremiumCharts.tsx`(125,383)、`EarnedPremiumTable.tsx`(72,75,116,118,141,144,202)
- `earned_from_prev/curr`：`NewEarnedPremiumTable.tsx`、`expense-forecast.ts`、`new-earned-premium.ts`、`cost-summary-calc.ts`
- `earned_01..12`/`earned_total`：`NewEarnedPremiumTable.tsx`、`new-earned-premium.ts`
- 服务端外部面：PAT/CLI/MCP 直读 JSON 字段（无静态清单，需靠审计日志观察期兜底）

### 3.5 推荐结论

**方案 A 立即做（本 PR 已含第 3 点），方案 B 挂起**——机器层无重名冲突，重命名的收益是防语义混淆，
成本是 11 个前端文件 + 外部 PAT/MCP 消费方的破坏面。除非发生真实混用事故（届时按 §3.3 步骤执行），
不建议为预防性收益支付破坏性成本。若用户拍板做 B，§3.3-§3.4 即可直接执行。

## 四、复算命令存档（可复现）

```sql
-- 闰年保单体量
SELECT DATEDIFF('day', CAST(insurance_start_date AS DATE),
                CAST(insurance_start_date AS DATE) + INTERVAL 1 YEAR) AS policy_term,
       COUNT(*), COUNT(DISTINCT policy_no), ROUND(SUM(premium)/10000,1)
FROM '数据管理/warehouse/fact/policy/current/SC/*.parquet'
WHERE insurance_start_date IS NOT NULL AND insurance_type IN ('交强险','商业保险')
GROUP BY 1;
-- 364 vs 1年 双变体复算：完整 SQL 见本文档 git 提交对应会话；
-- 核心差异项 = P·(1-F)·(daysB-daysA)/policy_term，daysA 封顶 起保+364，daysB 封顶 起保+1年−1天。
```
