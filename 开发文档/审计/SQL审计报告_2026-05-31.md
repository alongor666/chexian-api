## 1. P0 — 安全（立即修，已人工复核）

### 🔴 S-1【严重】growth 自定义/日上下文日期未校验即拼入 SQL

- **注入点**：`server/src/sql/growth/custom.ts:49-50,58-59`（`AND ${df} >= '${config.currentPeriod.startDate}'` 裸单引号拼接，无转义）
- **漏洞根因**：`server/src/routes/query/growth.ts:55,108` 仅对 `baselineStart/baselineEnd` 调用 `isValidDateFormat`，**漏校验 `startDate/endDate`**；且 `growth.ts:60,92` 把这两个值从 WHERE 中剥离（设 `undefined`），从而**绕过** `buildWhereFromFilterParams` 内唯一的日期格式校验。
- **可控来源**：HTTP query `startDate`/`endDate`（`commonFilterSchema` 仅 `z.string().optional()`，无正则）。
- **利用示例**：`?growthType=custom&baselineStart=2025-01-01&baselineEnd=2025-12-31&startDate=2026-01-01' OR '1'='1&endDate=...`
- **后果**：注入任意布尔条件绕过筛选/越权读数；此生成路径**无** `validateSQL` 二次拦截（那只作用于 `/api/query/sql` 直通）。
- **✅ 已人工核实**：阅读 `growth.ts:40-123` + `custom.ts:43-64` 确认漏校验与裸拼接同时成立、攻击向量可达。
- **最小修复**：`growth.ts` 两个 custom 分支对 `startDate/endDate` 补 `isValidDateFormat`（与 baseline 对齐）。**根治**：`currentPeriod/baselinePeriod` 改用 `buildDateCondition`（已含字段名校验+格式校验+转义），消除裸 `'${}'`。

### 其余安全发现（健壮性/纵深，非紧急）

| # | 风险 | 严重 | 位置 | 可达性 |
|---|------|------|------|--------|
| S-2 | limit/offset 无上界（DoS） | 中 | `routes/query/salesman.ts:10`、`performance.ts:280` → 下游 `LIMIT ${}` | 可达 |
| S-3 | `generateQuoteDrilldownQuery` groupCol switch 无 default | 低 | `quote-conversion.ts:141-156` | 路由 enum 兜底 |
| S-4 | 死代码注入面（维度/metric/limit 无校验） | 低 | `kpi.ts:336-395`、`trend/dimension-queries.ts:30-40` | 无调用方 |
| S-5 | 黑名单维护风险（`/sql` 直通新增危险函数需同步登记） | 信息 | `utils/sql-validator.ts:29-75` | 已纵深兜底 |

> **正面**：动态字段名/维度/排序/分组一律走白名单；`buildInCondition`/`escapeSqlString` 转义到位；`/api/ai/*` **不执行 LLM 生成的 SQL**（无 NL2SQL→执行信任边界问题）。整体防御纵深良好，S-1 是唯一真实可利用注入。

## 2. P1 — 业务口径（红线：登记 BACKLOG，禁擅改）

### 🔴 D-1【高】优质业务定义三处分裂、语义相反（双 agent 命中）

- `kpi.ts:11-21` 与 `trend/shared.ts:21-31`：`非新能源 AND (非营业客车 OR 货车1-9吨)`（字面相同但独立定义）
- `salesman-ranking.ts:1-9`：`网约车 OR 出租车 OR 货车1-9吨` —— **覆盖营业性质，与上者相反**
- **影响**：业务员排名的"优质"与 KPI 面板的"优质"含义相反，横向对比得出错误结论。
- **方向**：收归 `shared/business-conditions.ts` 单一事实源；确认 salesman-ranking 是否刻意分化（若是须命名区分+注释）。

### 🔴 D-2【高】赔付率发展三角缺无责/零结/注销/拒赔过滤

- `claims-detail.ts:474-495`（`claimed` CTE）直接从 ClaimsDetail 聚合赔款，**缺** SSOT 要求的 `liability_ratio=0` 与 `case_type IN ('零结','注销','拒赔')` 金额过滤。
- **影响**：历史 cohort 赔付率系统性虚高（四川诊断实测虚高约 7%）。
- **方向**：CASE 内补过滤，或直接 JOIN ClaimsAgg 替代自聚合。

### D-3【中】出险率分母用签单件数而非已赚暴露（2 处违反铁律）

- `claims-detail.ts:519-555`（`freq_per_1000`，分母 `policy_count`）
- `comprehensive-analysis.ts:134-137`（`claim_frequency`，分母 `policy_count`）
- 铁律：出险率分母用已赚暴露。对照 `cost-ratios.ts` 的 `earned_loss_frequency`（`SUM(earned_days)/365`）即正确范本。

### D-4 ClaimsAgg 无 accident_time 窗口（B299 已登记，确认影响面）

- `duckdb-domain-loaders.ts:390-410`，被 `cost-ratios.ts`/`comprehensive-analysis.ts`/`kpi.ts`/`forecast/baseline.ts` 全部继承。当前被 "cutoff 恒最新" 掩盖；引入历史快照/跨期查询即虚高。

### D-5【中】earned-premium 双口径未文档化

- `cost/earned-premium.ts:76-93` 用 `+INTERVAL 364 DAY` + 险类系数 α(0.82/0.94)「财务口径」；`cost-ratios.ts:53,57-63` 用 `+INTERVAL 1 YEAR` 无系数「时间分摊口径」。两接口都叫"已赚保费"，下游混用会算错赔付率。需字段命名区分 + 文档说明。

### 其余口径（低）

- D-6 `claims-detail.ts:26-50` / D-7 `repair.ts:371-381` 去重仅按 `policy_no`（缺 `insurance_start_date`）—跨年续保潜在隐患。
- ✅ cross-sell.ts 推介率分母（主全+交三，排除单交）**口径正确**。

## 3. P2 — 性能（DuckDB，不动口径，收益高风险低）

| # | 问题 | 严重 | 位置 | 收益 |
|---|------|------|------|------|
| F-03 | `segmentCaseExpr()` 8 层 `LIKE '%..%'` + CAST/TRIM 逐行执行，所有业绩分析查询受累 | 高 | `performance-analysis/shared.ts:123-155` | CPU −20~40% |
| F-01 | `EXTRACT(YEAR FROM col)=N` 破坏 zonemap 剪枝 | 高 | `claims-heatmap.ts:331`、`cost/earned-premium-detail.ts:62+`、`customer-flow.ts:18+`、`sql-builder.ts:269,391` | 扫描 −30~60% |
| F-02 | YoY 双 CTE 各全量扫 PolicyFact 一次 | 高 | `growth/yoy.ts:32-62` | 扫描量减半 |
| F-08 | `eligible_policies × all_cutoffs` CROSS JOIN 先膨胀再过滤 | 中 | `claims-heatmap.ts:344-440` | 内存 −40~70% |
| F-06 | `achievement_cache` 聚合子查询 3 模块重复内联 | 中 | `performance-analysis/{summary:85,top-salesman:60,drilldown:89}` | 每请求省 3 次聚合 |
| F-05 | `filtered`/`filtered_base` 均 `SELECT *` 全列物化 + 双扫 | 中 | `kpi.ts:57-65` | 内存 −30~50% |
| F-07 | 周编号 `DAYOFYEAR/ISODOW/DATE_TRUNC` 逐行重算 + 60 行重复 | 中 | `trend/{total-trend:66,premium-trend:74}` | 每行计算减少 |
| F-04 | RepairDim `SELECT DISTINCT SUBSTR` 子查询多次内联 | 中 | `repair.ts:207,363-364,417` | RepairDim 扫描 −2~3 次 |
| F-09 | 同请求 4 个生成器各扫 ClaimsDetail 一次 | 中 | `repair.ts`(4 函数) | ClaimsDetail 扫描 −75% |
| F-10 | `LIKE '%华安%'` 前置通配符逐行扫 | 低 | `customer-flow.ts:66-88` | ETL 预算布尔列 |
| F-11 | `policy_term` DATEDIFF 在 13~24 列重算 | 低 | `cost/earned-premium-detail.ts:31-157` | 加速 10~20% |

> **首选关键路径**：F-03（物化时预算 `segment_tag` 整数列）+ F-01（YEAR 函数 → `col BETWEEN DATE 'Y-01-01' AND 'Y-12-31'` 范围比较）。集中在物化层/SQL 生成层，不碰口径，风险最低收益最高。
>
> **架构正面**：`PolicyFactRealtime` 按 `policy_date` 有序物化（zonemap 友好）、`CrossSellDailyAgg` 预聚合、`loadMultipleParquet` mtime 增量加载、`union_by_name` schema 容错、route-cache 覆盖率高 —— 均为正确决策。
