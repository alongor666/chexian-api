## 2. 当前归属的局限（为何需要业务历史数据）

当前 `SX.json` 是**按机构名称的近似归属**，两处需业务历史数据精确化：

### 2.1 停用历史机构按「现名归现单元」— 需按存续时间对账

61 个原始机构含 **26 个「停用」历史机构**（机构改编码 / 撤并产生）。当前把停用机构按其营销服务部**名称**归到现在的在用单元，分布：

| 现归单元 | 停用机构数 |
|----------|:---:|
| 太原二部 | 12 |
| 经代、车商、重客 | 5 |
| 运城 | 3 |
| 太原一部 | 2 |
| 大同 / 长治 / 晋中 / 临汾 | 各 1 |

**风险**：若某营销服务部历史上曾跨多个单元、或撤并去向与「现名」不符，按名称归属会错配历史保费。**太原二部当前 4.13 亿即含大量停用历史**（在用太原盘子仅约 0.25 亿），其准确性依赖停用部门的真实存续与撤并去向。

### 2.2 「经代、车商、重客」未按人员细分

业务要求把 `01` 的 **经代、车商、重客**（当前合并为 1 个单元，1.15 亿）按**业务员人员**进一步拆为 经代 / 车商 / 重客 三类。当前机构列无此信息，须靠人员归属。

## 4. 数据到位后的处理流程

1. 营销管理部回填 §3.1 / §3.2 两张表。
2. 据此把 `SX.json` 从「按机构名」升级为「按 部门存续时间 + 业务员归属」的时序清分规则。
3. `transform.py:normalize_branch_org` 增强：支持按 **业务员 + 签单日期** 命中当时所属部门 → 经营单元（替代当前按机构名的近似），并对 `01` 按人员拆 经代/车商/重客。
4. 重跑 SX ETL → `validation/SX`，**与营销管理部口径逐单元对账**（保费/件数，误差应 ≤ 万分之一）。
5. 对账通过 **+** G5 其余口径签字（[口径对齐_山西.md](./口径对齐_山西.md) §1：客户分类 / NCD / 风险评级 / 续保窗口 / 满期系数）后，方可按 ADR D5 序列 GATED 上线（RLS-on → SX 进 `current/` → 发账号）。
