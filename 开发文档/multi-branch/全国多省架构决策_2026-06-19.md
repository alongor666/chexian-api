# 全国多省架构决策（ADR）— 山西接入触发的架构选型复评

> **状态**：✅ 架构方向已与用户拍板（2026-06-19）｜⏳ 实施计划见 `.claude/plans/`（先 0a + governance #13）
>
> **决策人**：项目主人（用户）｜**起草**：Claude（基于全程代码实证）
>
> **背景一句话**：原 ETL 只有四川（SC）单省数据，2026-06-19 拿到山西（SX）全套源数据（签单 485MB / 报价 113MB / 厂牌 43MB / 理赔 / 维修，共 ≈645MB），终局确认为**全国铺开多省的统一经营分析平台**。本文复评"该如何改造最好"，并固化结论与风险面。
>
> **维护协议**：append-only。已拍板章节禁止改写；新决策/新实证追加章节。

## 1. 决策（TL;DR）

| # | 决策 | 一句话理由 |
|---|------|-----------|
| D1 | **维持"单数据湖 + `branch_code` 省份列 + 行级安全（RLS）"架构，不换范式** | 终局是全国汇总 + 持续加省 + 一套运维，这正是单数据湖的主场；且本项目该架构脚手架已建成约 90% |
| D2 | 否决"各省独立部署/独立库" | 全国终局下没有跨省汇总能力、运维成本 ×N，仅适用于"某省法规要求物理彻底隔离"的特例 |
| D3 | 把"第 0 层"拆成 **0a（磁盘 Hive 分区，便宜低风险，现在做）** 与 **0b（内存随省份收缩，性能工程，归第 2 层）** 两块分开定价 | 二者收益与代价完全不同，混为一谈会低估风险（见 §6） |
| D4 | ClickHouse-分布式 / 存算分离作为**写进文档的扩容逃生口**，设明确触发阈值，现在不做（YAGNI） | 过早上 MPP 引擎是过度工程；但它是"物理隔离 + 跨省汇总"兼得的真正进化形态，须预留 |

> **行级安全（RLS, Row-Level Security）**：在查询 SQL 自动注入 `WHERE branch_code='<用户所属省>'`，让每个省的用户只能看到本省数据的机制。

## 2. 问题与终局

- **现状**：所有数据 `branch_code` 默认贴 `SC`（四川）；14 个三级机构均为四川分公司机构。
- **终局（用户 2026-06-19 确认）**：**未来全国铺开多省**，山西只是第二个省，目标是一个统一的多省经营分析平台，总部/用户需要看跨省合并视图。
- **核心矛盾**：现实现是按"四川 + 再加一个省"的体量做的，**不是按"全国 30 省平台"做的**；直接照现状铺全国，会在第 3、第 5 个省时撞上结构性扩展债（内存、源摄入、机构唯一性、派生域）。

## 3. 备选方案对比（评判标准＝全国多省终局）

| 维度 | A. 单数据湖 + branch_code + RLS（现方案） | B. 各省独立部署/独立库 | C. 分库存储 + 联邦查询层 | C+. ClickHouse-分布式（C 的进化形态） |
|---|---|---|---|---|
| 跨省合并视图 | ✅ 一条 `GROUP BY branch_code` | ❌ 跨 N 库拉数应用层拼，几乎做不动 | 🟡 联邦层能做但复杂 | ✅ 分布式表原生合并 |
| 新增一省成本 | ✅ ETL 贴标签 + 发账号 | ❌ 整套部署+运维 ×N | 🟡 加库 + 注册联邦 | ✅ 加一个分片 |
| 运维负担 | ✅ 一套代码一套部署 | ❌ N 套发版同步 | 🟡 1 套代码 + N 库连接 | 🟡 跑 DB 集群 |
| 数据隔离强度 | 🟡 逻辑隔离（RLS 列过滤，代码 bug 可能串读） | ✅ 物理隔离最强 | ✅ 物理隔离 | ✅ 库级物理隔离 |
| 小 VPS 扩展性 | 🟡 内存峰值随省线性涨（见 §6 0b） | ✅ 各省各机器天然分摊 | 🟡 看联邦层落点 | ✅ MPP 水平扩展 |
| 本项目现状 | **已建成 ~90%**（字段/RLS/缓存/SQL/SOP 全就位） | 从零另起 + 拆现有单仓逻辑 | 联邦雏形已有（`SQL_FEDERATION_ENABLED`）但面向派生域，非跨省 | 须迁移引擎 |

**结论**：选 **A**。B 的"物理隔离"优势在"需要全国汇总"面前直接变致命缺陷；C+ 是 A 撞到规模天花板后的升级路径（见 §7 第 2 层），不是现在的选项。

## 5. 缺口清单 — 真正要做的"改造"

> 这些是"全国可用"的前置，不是"换架构"。按性质分类。

### 5.1 ETL / 数据层
- **G1｜源文件省份化摄入**：`daily.mjs` 只认 `01_签单清单_*.xlsx` 等固定命名、从单目录 `数据管理/` 摄入；山西文件名为 `山西_签单清单_2021-2026.xlsx`，命名/路径对不上。需 `数据管理/staging/<省>/` 分目录 + daily.mjs 按省编排（或省份感知的源路径）。
- **G2｜磁盘按省分区**：见 §7 0a。
- **G3｜维度表省份限定**：业务员/厂牌/维修/计划维度表均为单 `latest.parquet`，无省份维度（`server/src/config/paths.ts`）；山西维度会与四川撞或漏。
- **G4｜派生域补 branch_code**：cross-sell / repair / customer_flow / new_energy_claims 等派生视图未携带 branch_code（BACKLOG `2026-06-19-claude-00bac8` / PR #676 follow-up）→ 省用户查这些一律 fail-closed 拒绝（安全无泄漏，但功能不可用）。

### 5.2 业务口径层
- **G5｜口径业务签字**：`口径对齐_山西.md` 中 11 类客户分类 / NCD 表 / 风险评级 / 续保窗口 / 满期系数 6 大口径全为 `⏳ 未签字`。**这是业务方的活**，山西口径未必与四川相同。RLS flag 启用前必须签字（Day-1 SOP §8 禁令）。

### 5.3 应用 / 治理层
- **G6｜机构白名单硬编码**：`kpi-detail.ts:85` 硬编码 14 个机构同城/异地白名单，新增/山西机构两边都不计入 → region 环形图分母静默缺失（BACKLOG `2026-06-11-claude-fdbba5`）。**到第 2 个省就破**。
- **G7｜山西账号**：`preset-users.ts` 尚未加 SX 超管 + 山西机构用户（Day-1 SOP §4）。
- **G8｜前端空态保护**：山西数据装载中的提示文案未做，禁止静默展示空 KPI（Day-1 SOP §5）。

## 6. 风险面 — 第 0 层 Hive 分区的改动面（实证）

**总判：风险小而集中，不是铺开。** 三道绝缘层把风险挡在存储层之外：

| 层 | 是否动 | 实证 | 风险 |
|---|--------|------|------|
| **71 端点 SQL** | ❌ **零** | `server/src/sql` 内联 `read_parquet` = **0 次**；全部 `FROM PolicyFact`(129) / `ClaimsDetail` / `RepairDim` 命名关系 | 极低 |
| RLS 权限注入 | ❌ 已就位 | `permission.ts:95` | 无 |
| route-cache key | ❌ 已 branch 隔离 | PR #501 + 压测脚本断言 | 无 |
| 装载层 parquet-loader | ✅ glob → `hive_partitioning=true` | `duckdb-parquet-loader.ts:141`；增量指纹逻辑(L90-131)须按分支适配"删文件"判断 | 中（集中 1 文件） |
| ETL 写出 daily.mjs | ✅ 输出路径加 `branch_code=` 分区目录 | `transform.py` 路径由 `-o` 决定，daily.mjs 编排 | 低 |
| sync-vps | ✅ 基本自动 | `scripts/sync-vps.mjs` 全目录 rsync，子目录分区自动带走 | 极低 |
| **governance #13 重叠检测** | ⚠️ **必须改 branch-aware** | `scripts/check-governance.mjs:789` `readdir(current)` 平铺查日期区间重叠；SC/SX 同为 2021-2026 → **不改会误报"数据翻倍"卡死 CI** | 中（最尖的雷） |
| governance 知识库行数 | ✅ 小改 | `check-governance.mjs:1190` 按 current 目录算行数/分片 | 低 |

### 6.1 关键修正 — "第 0 层"必须拆成 0a / 0b

数据最终 `CREATE TABLE PolicyFactRealtime AS ... ORDER BY policy_date`，**把全省的行物化进一张内存表**，并建 4 个索引（policy_date / org_level_3 / salesman / policy_no）+ zonemap 剪枝（`server/src/services/duckdb-materialization.ts:176-208`）。71 端点性能全靠这张内存索引表。因此：

- **0a｜磁盘 Hive 分区（便宜、低风险，真正的 Tier-0）**
  - 收益：**ETL/运维隔离**——单省刷新只重写自己分区目录、单省 sync、省份干净分离、`branch_code` 成真分区列。
  - 改动：装载层 glob（1 文件）+ daily.mjs 输出路径 + governance #13 改 branch-aware。
  - 不变：SQL / cache / RLS 全不动。
  - **局限：不省内存**——全省照样物化进 `PolicyFactRealtime`。
- **0b｜内存随被查省份收缩（不便宜，归第 2 层）**
  - 要省内存须**放弃"全量物化进内存"**，把 `PolicyFact` 改成直查 hive parquet 的视图。
  - 代价：**丢掉 4 个索引 + zonemap**，等于动所有查询的性能底座，必须用 `golden-baseline.mjs`（71 端点零差异）+ `bench-universal-cube.mjs` 全量回归。
  - 定性：这是性能工程，不是 Tier-0。

## 7. 扩展性分层与触发阈值

| 层 | 内容 | 何时做 | 触发阈值 |
|---|------|--------|---------|
| **0a** | 磁盘 Hive 分区 + 源摄入省份化（G1/G2）+ governance #13 | **现在**（山西接入前置） | 接第 2 个省即做 |
| **0b** | 物化层改造，内存随被查省份收缩 | 内存吃紧才做 | 单机 RSS / DuckDB peak 接近上限 |
| **第 2 层 · 存算分离** | parquet 上腾讯云 COS，查询节点独立于 web 层 | 数据撑爆单 2核4G web 机时 | 单机内存/磁盘扛不住 |
| **第 2 层 · ClickHouse / Doris** | MPP 列存 + 原生行策略 + 分布式表（每省一分片 + 总部合并视图）；物理隔离 + 跨省汇总兼得 | 真到全国规模 | 行数千万-亿级 / 高并发 / 监管要求物理隔离 |
