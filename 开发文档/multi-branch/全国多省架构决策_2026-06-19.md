> **状态**：✅ 架构方向已与用户拍板（2026-06-19）｜🟡 0a 实施方案经对抗式评审校准（见 §11）｜⏳ 实施计划见 `.claude/plans/`
>
> **决策人**：项目主人（用户）｜**起草**：Claude（基于全程代码实证 + 双模型对抗式评审）
>
> **背景一句话**：原 ETL 只有四川（SC）单省数据，2026-06-19 拿到山西（SX）全套源数据（签单 485MB / 报价 113MB / 厂牌 43MB / 理赔 / 维修，共 ≈645MB），终局确认为**全国铺开多省的统一经营分析平台**。本文复评"该如何改造最好"，并固化结论与风险面。
>
> **维护协议**：§1-§10 经一次对抗式评审校准（2026-06-19，见 §11 日志）。后续新决策/新实证追加章节或追加 §11 日志条目，不静默改写已校准结论。

## 1. 决策（TL;DR）

| # | 决策 | 一句话理由 |
|---|------|-----------|
| D1 | **维持"单数据湖 + `branch_code` 省份列 + 行级安全（RLS）"架构，不换范式** | 终局是全国汇总 + 持续加省 + 一套运维，这正是单数据湖的主场；且本项目该架构脚手架已建成约 90%（双轮对抗评审未动摇此决策） |
| D2 | 否决"各省独立部署/独立库" | 全国终局下没有跨省汇总能力、运维成本 ×N，仅适用于"某省法规要求物理彻底隔离"的特例 |
| D3 | 把"第 0 层"拆成 **0a（省份前缀文件名 + 扁平目录 + branch-aware 管道，现在做）** 与 **0b（内存随省份收缩，性能工程，归第 2 层）** 两块分开定价 | 0a 不采用 Hive 子目录（评审证实代码库多处假设扁平目录，子目录 blast radius 超 0a 预算）；二者收益与代价不同，混为一谈会低估风险（见 §6） |
| D4 | ClickHouse-分布式 / 存算分离作为**写进文档的扩容逃生口**，设明确触发阈值，现在不做（YAGNI） | 过早上 MPP 引擎是过度工程；但它是"物理隔离 + 跨省汇总"兼得的真正进化形态，须预留 |
| **D5** | **【评审校准】0a 不把 SX 数据载入共享 runtime；口径签字（G5）+ RLS-on 是 SX 进共享 runtime 的硬前置（对齐 Day-1 SOP）** | 装载器把 `current/*.parquet` 全量物化进同一内存表；RLS 默认关时 SX 行会让四川用户看到混省数据。详见 §6.2、§11 |

> **行级安全（RLS, Row-Level Security）**：在查询 SQL 自动注入 `WHERE branch_code='<用户所属省>'`，让每个省的用户只能看到本省数据的机制。本项目由 `BRANCH_RLS_ENABLED` 开关控制，**默认关闭**（`server/src/config/env.ts:127`）。

## 2. 问题与终局

- **现状**：所有数据 `branch_code` 默认贴 `SC`（四川）；14 个三级机构均为四川分公司机构。
- **终局（用户 2026-06-19 确认）**：**未来全国铺开多省**，山西只是第二个省，目标是一个统一的多省经营分析平台，总部/用户需要看跨省合并视图。
- **核心矛盾**：现实现是按"四川 + 再加一个省"的体量做的，**不是按"全国 30 省平台"做的**；直接照现状铺全国，会在第 3、第 5 个省时撞上结构性扩展债（内存、源摄入、机构唯一性、派生域、扁平目录假设）。

## 3. 备选方案对比（评判标准＝全国多省终局）

| 维度 | A. 单数据湖 + branch_code + RLS（现方案） | B. 各省独立部署/独立库 | C. 分库存储 + 联邦查询层 | C+. ClickHouse-分布式（C 的进化形态） |
|---|---|---|---|---|
| 跨省合并视图 | ✅ 一条 `GROUP BY branch_code` | ❌ 跨 N 库拉数应用层拼，几乎做不动 | 🟡 联邦层能做但复杂 | ✅ 分布式表原生合并 |
| 新增一省成本 | ✅ ETL 贴标签 + 发账号 | ❌ 整套部署+运维 ×N | 🟡 加库 + 注册联邦 | ✅ 加一个分片 |
| 运维负担 | ✅ 一套代码一套部署 | ❌ N 套发版同步 | 🟡 1 套代码 + N 库连接 | 🟡 跑 DB 集群 |
| 数据隔离强度 | 🟡 逻辑隔离（RLS 列过滤，代码 bug 可能串读） | ✅ 物理隔离最强 | ✅ 物理隔离 | ✅ 库级物理隔离 |
| 小 VPS 扩展性 | 🟡 内存峰值随省线性涨（见 §6 0b） | ✅ 各省各机器天然分摊 | 🟡 看联邦层落点 | ✅ MPP 水平扩展 |
| 本项目现状 | **已建成 ~90%**（字段/RLS/缓存/SQL/SOP 全就位） | 从零另起 + 拆现有单仓逻辑 | 联邦雏形已有（`SQL_FEDERATION_ENABLED`）但面向派生域，非跨省 | 须迁移引擎 |

**结论**：选 **A**。B 的"物理隔离"优势在"需要全国汇总"面前直接变致命缺陷；C+ 是 A 撞到规模天花板后的升级路径（见 §7 第 2 层），不是现在的选项。

## 5. 缺口清单 — 真正要做的"改造"

> 这些是"全国可用"的前置，不是"换架构"。按性质分类。

### 5.1 ETL / 数据层
- **G1｜源文件省份化摄入**：`daily.mjs` 只认 `01_签单清单_*.xlsx` 等固定命名、从单目录 `数据管理/` 摄入；山西文件名为 `山西_签单清单_2021-2026.xlsx`，命名/路径对不上。需 `数据管理/staging/<省>/` 分目录 + daily.mjs 按省编排。
- **G2｜省份物理隔离**：见 §7 0a（省份前缀文件名 + branch-aware 管道）。
- **G3｜维度/派生域省份限定**：业务员/厂牌/维修/计划维度表 + `RepairDim`/`QuoteConversion`/`RenewalTrackerFact` 等派生域均为单 `latest.parquet`/全局表，无省份维度（`paths.ts`、`duckdb-domain-loaders.ts:408-560`）。**多省共存不只是 policy 分片问题**——这些表不处理则 SX 用户拿到混省/全局数据。
- **G4｜派生域补 branch_code**：cross-sell / repair / customer_flow / new_energy_claims 等派生视图未携带 branch_code（BACKLOG `2026-06-19-claude-00bac8` / PR #676 follow-up）→ 省用户查这些一律 fail-closed 拒绝（安全无泄漏，但功能不可用）。

### 5.2 业务口径层
- **G5｜口径业务签字**：`口径对齐_山西.md` 中 11 类客户分类 / NCD 表 / 风险评级 / 续保窗口 / 满期系数 6 大口径全为 `⏳ 未签字`。**这是业务方的活**，山西口径未必与四川相同。**RLS flag 启用前必须签字（Day-1 SOP §8 + D5）。**

### 5.3 应用 / 治理层
- **G6｜机构白名单硬编码**：`kpi-detail.ts:85` 硬编码 14 个机构同城/异地白名单，新增/山西机构两边都不计入 → region 环形图分母静默缺失（BACKLOG `2026-06-11-claude-fdbba5`）。**到第 2 个省就破**。
- **G7｜山西账号**：`preset-users.ts` 尚未加 SX 超管 + 山西机构用户（Day-1 SOP §4）。
- **G8｜前端空态保护**：山西数据装载中的提示文案未做，禁止静默展示空 KPI（Day-1 SOP §5）。

## 6. 风险面 — 0a 的改动面（实证 + 评审校准）

> **校准说明**：本节原结论"风险小而集中，71 端点零改动，最尖雷=governance #13"经双模型对抗评审**部分被证伪**，已重写。校准日志见 §11。

### 6.1 哪些真不动 / 哪些会动

| 层 | 评审后结论 | 实证 |
|---|------|------|
| **71/78 端点 SQL 文本** | ✅ 文本零改动 | `server/src/sql` 内联 `read_parquet`＝0；全走 `FROM PolicyFact`(129) 等命名关系 |
| 端点**结果** | ⚠️ **会变**（措辞校准） | 端点结果取决于"装载的文件集合 + RLS 开关"——0a 一旦载入 SX 行且 RLS 关，四川结果即变（见 §6.2）。"SQL 不改 ≠ 结果不变" |
| 装载/物化层 | ⚠️ 与文件布局耦合 | `data-bootstrapper.ts:147/321` 非递归读 `current/*.parquet` → `loadMultipleParquet` → `duckdb-parquet-loader.ts:139` `read_parquet` → `materialization.ts:176` 全量物化进 `PolicyFactRealtime` |
| **扁平目录假设** | ❌ **远不止 2 处**（证伪） | 见下表 ~6 处 |
| RLS / cache key | ✅ 已就位 | `permission.ts:88`、PR #501 |

**扁平目录假设站点（0a 必须逐一 branch-aware，否则误报/误删/CI 卡死）：**

| 站点 | 问题 | 后果 |
|---|---|---|
| `scripts/lib/parquet-overlap-check.mjs:43` | 非递归 readdir + 不分省两两比对 | SC/SX 同期分片误报"数据翻倍" |
| `数据管理/pipelines/quick_reference.mjs:96` | `shardCount`＝目录文件总数 | governance #15 报"分片 4 vs 8"→ **CI exit 1** |
| `数据管理/daily.mjs:1449` | 范围互斥守卫 `RANGE_RE` 不分省 + 归档正则只匹配 `^YYYYMMDD-` | SX 前缀分片不受保护 / 归档误判 |
| `数据管理/daily.mjs:~1651` | `data-sources.json` 行数汇总所有 current 分片 | 跨省行数混计 |
| `scripts/sync-vps.mjs:368/459/595` | `--delete` + 新鲜度闸按全目录总行数 + 整目录同步粒度 | 单省同步风险 / 新鲜度误判 |
| `server/src/routes/data.ts:367` | web 上传归档正则把 `SX_` 吃进 prefix 段 | 仅 web 上传路径触发（0a 走 ETL+rsync 不触发，记为已知限制） |

### 6.2 最关键发现 — 0a 不能把 SX 载入共享 runtime（D5 来源）

装载器把 `current/*.parquet` **全量物化进同一张内存表** `PolicyFactRealtime`（`materialization.ts:176`）。0a 若把 SX 放进 `current/`、而 RLS **默认关**（`env.ts:127`），则 admin（四川）查询无 branch 过滤 → **看到 SC+SX 合并数据**。这不仅让 golden-baseline"四川零差异"必然失败，更是**生产安全隐患**：SX 数据一进共享 runtime + RLS 没开，四川线上用户立刻看到混省数据。

**纠正**：本项目 Day-1 SOP 早已在 T-3（载 SX 到生产、发账号之前）就 `BRANCH_RLS_ENABLED=true`。故正确序列 = **口径签字（G5）+ RLS-on 是 SX 数据进共享 runtime 的硬前置**；0a 期 SX 数据只在隔离环境（独立目录/duckdb 直查）验证，绝不进服务端载入的 `current/`。

### 6.3 0a 拆分

- **0a｜省份前缀文件名 + 扁平目录 + branch-aware 管道（现在做）**
  - 收益：**ETL/运维隔离 + governance 不误报**——单省刷新只动自己的前缀文件、单省 sync、`branch_code` 作数据列。
  - 改动：§6.1 表中 ~6 处扁平假设逐一 branch-aware（含测试）。
  - 不变：71/78 端点 SQL 文本、RLS、cache。
  - 局限：**不省内存**（全省物化进 `PolicyFactRealtime`）；**不把 SX 载入共享 runtime**（见 6.2）。
- **0b｜内存随被查省份收缩 + 真 Hive 子目录分区（不便宜，归第 2 层）**
  - 要省内存须放弃"全量物化进内存"，改 `read_parquet(hive)` 视图，**丢掉 4 个索引 + zonemap**，须 `golden-baseline` + `bench-universal-cube` 全量回归。
  - branch_code 双源风险（同名数据列 + hive 路径分区键）在此层才触发（评审论点⑤，0b 的雷，非 0a）。
