# UI端数据载入性能 10000倍 提升架构设计方案

## 目标与挑战
**核心目标**：一切需求都要能响应（支持各维度灵活下钻、下钻深度无限制），且“又快又好”（追求 0.05ms 级别的点击即呈现，也就是消除约 500ms 的网络与计算延迟）。

**挑战矛盾**：
1. **预计算方案**（快）无法穷举所有维度的交叉组合（支持不了无限下钻）。
2. **实时计算方案**（全）必须经过前端请求、网络传输、后端解析、DuckDB OLAP 聚合计算的固定开销路线（快不起来，通常下限在 100ms-500ms）。

---

## 破局之道：全链路混合分层极速架构 (Hybrid Tiered Ultra-fast Architecture)

要实现这“既要全又要极速”的“不可能三角”，必须彻底改变目前的“请求-响应”心智模型，采用 **多级命中 + 预判缓存 + 计算前置** 的混合架构。

### 第 1 层：前端内存微立方体 (Frontend Micro-Cube) —— 解决 95% 高频基础操作
**核心指标：响应时间 < 1ms**

既然光跨越网络需要时间，我们就把高频数据的全集**塞进浏览器的内存里**。

*   **数据模型降维**：将每天生成的超大 Parquet（百兆以上），在夜间通过 Node 脚本或 Python 压缩成一个极小、高度字典编码的 JSON/Arrow 压缩包（通常只有几 MB）。这个小压缩包只包含：机构维度、保费、车险件数、险种分布这些**最高频、聚合程度最高**的数据组合。
*   **首屏与高频操作**：当用户第二天早上打开系统，这几 MB 的数据包静默下载到了浏览器的 IndexedDB 或内存中。
*   **UI 交互**：当用户点击 `月度趋势`、`各个分公司 KPI`、`简单的车驾意占比` 等基础报表时，**UI 组件完全断开与后端的网络请求**，直接在浏览器端对本地内存中的 Micro-Cube 执行 JS 过滤和聚合计算。由于数据全在本地内存，响应只需 0.1ms 级别，页面交互如丝般顺滑。

### 第 2 层：后端内存位图索引 (Backend In-Memory Bitmap Index) —— 解决常规下钻
**核心指标：响应时间 ~10ms - 50ms**

当用户进行常规组合筛选（比如：“成都分公司” 下的 “营运货车” 且 “交三险”）时，前端的数据包可能没有这么细的粒度，必须请求后端。

*   **现状问题**：请求打到后端后，后端会去拼 SQL，然后交给 DuckDB 重新扫描。
*   **极致优化（脱离传统 DB）**：对于常见的枚举类维度（如客户类别、险别组合、机构、渠道），在 Node.js 内存里构建 **Bitmap Index (位图索引)** 或 **倒排索引**。
*   **查询过程**：请求进来，Node.js 根本不经过 SQL 解析器，直接在内存中做几个 Bitmap 的按位与（AND）和按位或（OR）操作（极其高效的 CPU 指令级操作），瞬间得出结果并返回。在 10-50ms 内即可穿透网络并完成计算。

### 第 3 层：预判式极速预热 (Predictive Prefetching) —— 解决最后 5% 的深层下钻
**核心指标：感知时间 = 0ms（隐式等待）**

就算后端的极限只能压到 50ms，或者需要 fallback 到真实的 DuckDB SQL 执行（比如极其冷门的维度组合下钻），我们依然可以在**前端体感上做到 0 延迟**。

*   **用户的每一次 Hover（悬停）都是计算指令**：当用户的鼠标光标移动到“营业货车”这个图表区块，或者移动到下拉菜单的“德阳中心支公司”选项上方（此时还未点击），前端系统就立刻（提前了 100-300ms）在后台向服务器发起了该维度组合的 API 预读请求。
*   **时间差魔术**：等用户按下鼠标（Click）并且 UI 开始切换动画的时候（通常需要 200ms），其实数据在刚才 Hover 的那一瞬间已经被服务器算好并送回前端缓存里了。
*   **用户体感**：点击图表的瞬间，数据立刻呈现，没有任何 Loading 圈，从用户视角看就是“万倍速度”的 0 延迟下钻。

## 总结：如何落地？

如果按照这个目标，我们的改造路径（Roadmap）可以分为三步走：

1.  **基础设施：前端预读与状态合并（成本最低，收益最大）**：接入 React Query 或 SWR 这类极致的数据预取和内存缓存框架。在现有的过滤面板上加入 `onMouseEnter` 预请求逻辑。利用用户操作的时间差掩盖网络延迟。
2.  **核心减负：首屏数据打包下发 (Micro-Cube)**：在每天夜间的 `daily.sh` 脚本中，用鸭子数据库（DuckDB）跑出一个包含了所有全局/二级机构的基础 KPI 与趋势的极简 JSON 集，客户端初始化时作为资源文件拉取。
3.  **终极形态：后端计算引擎内存化**：针对深层下钻的高开销图表，在 Node.js 层建立倒排缓存，旁路掉耗时的实时 SQL 解析。

**您认为这个“前端本地内存计算 + 用户行为预判拉取 + 后端全内存倒排索引”的混合三层架构，是否符合您对极速体验的期望？** 我们马上就可以进行第一阶段的落地！
