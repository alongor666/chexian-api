# 2026-02-17 安全深度研究报告（chexian-api）

## 1. 研究范围与方法

### 1.1 范围
- 后端认证与权限链路（`server/src/routes/auth.ts`, `server/src/middleware/auth.ts`, `server/src/middleware/permission.ts`）
- 数据上传与文件访问面（`server/src/routes/data.ts`, `server/src/utils/security.ts`）
- 前端认证状态与 Token 存储（`src/shared/contexts/AuthContext.tsx`, `src/shared/api/client.ts`）
- 全局安全基线（`server/src/app.ts`, `server/src/config/auth.ts`, `server/src/config/cors.ts`）

### 1.2 方法
- 静态代码审计（配置、路由、中间件、工具函数）
- 关键风险模式检索（认证、口令、注入、日志、令牌管理）
- 现有单元测试与治理脚本复核

---

## 2. 结论概览（Executive Summary）

当前项目的**输入校验与文件上传安全做得较扎实**（文件名安全、路径约束、Parquet 魔数校验、基础上传频率限制均已具备），但在**认证体系与会话治理**层面存在明显短板。

**总体评级：中高风险（High-Medium）**。

- **P0（需立即处理）**：默认弱口令/可预测口令链路、缺失登录限流与暴力破解防护
- **P1（建议本周处理）**：Token 存储在 `localStorage`、刷新接口信任客户端传入 token、无统一 API 级速率限制
- **P2（建议本月处理）**：CORS 生产回退策略过宽、安全文档与代码现实存在漂移

---

## 3. 关键发现（按优先级）

## 3.1 P0-1：默认口令体系可预测，存在横向撞库/弱口令风险

### 现象
- 代码中存在“管理员 `admin123`、机构用户 `<拼音>123`”口令策略与注释。
- 同时存在预置用户账户配置，说明系统具备“默认账户体系”路径。

### 风险
- 一旦服务暴露在可探测网络（含内网横向渗透场景），攻击者可通过低成本字典攻击获取有效账户。
- 即便密码哈希存储，若密码策略弱，依然无法抵抗在线爆破。

### 建议
1. 立即禁用默认口令策略，强制部署时通过环境变量注入随机高熵口令。
2. 首次登录强制改密（管理员与机构用户）。
3. 增加密码复杂度与轮换策略（最短长度、历史密码限制、过期提醒）。

---

## 3.2 P0-2：登录接口无限流/锁定机制，可被暴力破解

### 现象
- 登录路由有参数校验，但未见账号/IP 维度登录限流、失败阈值、账户锁定或挑战机制。
- 项目中仅对“文件上传”做了简易 rate limit，未覆盖认证入口。

### 风险
- 攻击者可持续尝试用户名+口令组合进行在线暴力破解。
- 与“默认/可预测口令”组合后，风险从理论升为高概率可利用。

### 建议
1. 对 `/api/auth/login` 增加 `IP + username` 双键限流（如 5 次/15 分钟）。
2. 增加指数退避与临时锁定（如 10 次失败锁定 30 分钟）。
3. 登录失败与异常认证事件接入审计日志与告警。

---

## 3.3 P1-1：JWT 置于 localStorage，XSS 成功后可直接窃取会话

### 现象
- 前端将 token 持久化在 `localStorage`，并在应用启动时解析使用。

### 风险
- 任意 XSS 一旦成立，可直接读取 token 并进行会话劫持。
- 鉴于前端存在用户输入展示场景，该风险应按“高可能+高影响”处理。

### 建议
1. 迁移为 `HttpOnly + Secure + SameSite` Cookie 会话。
2. 前端移除 token 读写逻辑，改为后端 session/cookie 验证。
3. 对高敏 API 增加二次校验（短时 token / step-up auth）。

---

## 3.4 P1-2：刷新接口接受请求体中的 token，缺少绑定校验语义

### 现象
- `/api/auth/refresh` 直接读取 body 中的 token 并刷新。
- 未体现与当前会话上下文绑定（设备指纹/refresh token 白名单/jti 撤销列表）。

### 风险
- 若 token 泄露，攻击者可在过期前持续续期（取决于后续策略），扩大攻击窗口。

### 建议
1. 改为 refresh token（长）+ access token（短）双令牌模型。
2. refresh token 存储服务端（或至少维护 jti 黑名单），支持注销与撤销。
3. refresh 接口不再接受任意 body token，改为仅信任 HttpOnly Cookie。

---

## 3.5 P1-3：缺少统一 API 速率限制与安全事件审计基线

### 现象
- 目前速率限制仅出现在上传路由。
- 对查询接口、认证刷新接口、AI 接口未见统一限流与审计策略。

### 风险
- 容易被批量请求拖垮（资源滥用、DoS 前置阶段）。
- 异常行为追溯能力不足，事件响应滞后。

### 建议
1. 全局接入 `express-rate-limit`（按路由分层策略）。
2. 对敏感接口加入审计日志（登录、刷新、上传、SQL自定义查询）。
3. 建立最小告警集（暴力登录、异常 429 峰值、token 刷新异常）。

---

## 3.6 P2-1：生产环境 CORS 回退策略偏宽松

### 现象
- 生产环境若未配置 `CORS_ORIGIN`，会回退到本地开发 Origin 列表。

### 风险
- 在某些部署拓扑中可能造成跨域策略非预期放宽。

### 建议
1. 生产环境要求 `CORS_ORIGIN` 必填，不允许默认回退。
2. 启动时校验并在缺失时 fail-fast。

---

## 3.7 P2-2：安全文档与架构现状有漂移

### 现象
- 现有安全文档中存在“数据不离开浏览器/无服务器存储”等历史表述，与当前纯 API 架构不一致。

### 风险
- 误导后续安全评审与合规判断，导致错误控制措施被接受。

### 建议
1. 更新安全约束文档为“API 模式”版本。
2. 为每条安全控制补充“代码锚点 + 验证命令 + 责任人”。

---

## 4. 现有安全优势（保留并加固）

1. 文件上传链路具备多层防护：扩展名检查、路径验证、Parquet 魔数验证、最大大小限制。  
2. 查询路由已统一经过 `authMiddleware + permissionMiddleware`。  
3. SQL 自定义查询具备只读/关键词/访问边界/隐私字段等约束。  
4. 服务已接入 `helmet`，具备基础 HTTP 头防护能力。
