### 架构亮点（值得保持的优势）

1. **注册表驱动**：指标 + 字段 + 数据域 + 路由参数 四大注册表，codegen 派生保一致性
2. **四层缓存**：SW → HTTP ETag → 路由 LRU（预压缩 brotli） → DuckDB 查询缓存，版本化自然失效
3. **治理自动化**：30+ 项 governance 检查 + git hooks + CI，红线编码为可执行闸
4. **DuckDB 查询去重**：inflight coalescing（同 SQL 并发只执行一次）+ 连接池 + 并发门控
5. **渐进式重构**：API 客户端、SQL 生成器、DuckDB 服务的拆分都用 barrel re-export 保接口稳定
6. **安全纵深**：三级限流 + RLS 行级安全 + PAT 只读 + 审计 + SQL 注入防护 + 错误不泄漏

## 二、六大优化方向

### 方向 1：前端大组件拆分（影响面：用户体验 + 可维护性）

**问题**：多个页面组件超过 700 行，职责混杂（数据获取 + 状态管理 + 渲染 + 交互逻辑耦合在单文件中）。

| 文件 | 行数 | 问题 |
|------|------|------|
| `GeoRiskPanel.tsx` | 951 | 地图 + 表格 + 筛选逻辑混合 |
| `CrossSellAnalysisPanel.tsx` | 891 | 热力图 + 趋势 + KPI + 下钻全在一个组件 |
| `NewEarnedPremiumTable.tsx` | 891 | 大表格组件包含排序/分页/格式化/导出 |
| `PerformanceAnalysisPanel.tsx` | 882 | 已登记 BACKLOG `2026-06-22-claude-03f6f0` |
| `AccessControlPage.tsx` | 761 | 用户管理 + PAT 管理 + 权限矩阵 |
| `AdvancedFilterPanel.tsx` | 728 | 复杂筛选器 |
| `ForecastBaselinePanel.tsx` | 705 | AI 预测面板 |
| `FilterLayoutV2.tsx` | 690 | 布局 + 响应式 + 筛选条件 |

**优化方案**：
1. 抽取自定义 Hook（`useXxxData` / `useXxxState`）分离数据获取和状态逻辑
2. 将渲染子区域拆为独立子组件（如 `GeoRiskPanel` → `GeoMap` + `GeoTable` + `GeoFilters`）
3. 优先处理 `PerformanceAnalysisPanel`（已有 BACKLOG 项）和 `CrossSellAnalysisPanel`

**预期收益**：组件渲染隔离、测试可单独覆盖、代码审查效率提升

**关联 BACKLOG**：`2026-06-22-claude-03f6f0`（PerformanceAnalysisPanel）、`2026-06-11-claude-3093a3`（重复组件收拢）

### 方向 2：SQL 生成器公式统一走注册表（影响面：数据准确性 + 口径一致性）

**问题**：指标注册表（`metric-registry`）已建立但未充分执行——多个 SQL 生成器硬编码公式绕过注册表，导致同一指标在不同路由有不同计算口径。

| 具体问题 | 涉及文件 | BACKLOG |
|---------|---------|---------|
| `comprehensive-analysis.ts` 硬编码 4 个已注册指标 | `earned_premium` × 9 处、`expense_ratio` × 5 处 | B305 |
| `cost-ratios.ts` 别名与注册表分裂 | `comprehensive_cost_ratio` vs 注册表 `comprehensive_expense_ratio` | B310 |
| `kpi.ts:229-242` 硬编码 `variable_cost_ratio` | 注册表 + cost-ratios + kpi 三处复刻 | B305/B310 |
| 双重舍入（SQL ROUND + 前端 decimals） | `earned_claim_ratio` 等 4 个指标 | B277 |
| `total_expense` 附加税率 1.6% vs 注册表 1.5% | `earned-premium-detail.ts:292` | B274 |
| 硬编码阈值（7% 四象限、14 机构白名单） | `drilldown.ts`、`kpi-detail.ts` | `2026-06-11-claude-fdbba5` |

**优化方案**：
1. **Phase 1**（B305 + B310 合并）：`comprehensive-analysis.ts` 和 `cost-ratios.ts` 接入 `getMetricSql()` 替换硬编码公式，统一别名
2. **Phase 2**（B277）：去除所有 SQL 层 `ROUND()`，统一由前端 `formatPercent` 单次舍入
3. **Phase 3**（B274）：用户确认附加税率后统一系数，补字典定义
4. **Phase 4**：阈值从硬编码迁入注册表或配置文件

**预期收益**：消除口径分裂风险，新增/修改指标只需改注册表一处

### 方向 3：DuckDB 查询性能优化（影响面：响应速度）

**问题**：已识别多处 DuckDB 性能反模式，部分查询存在不必要的全表扫描。

| 编号 | 问题 | 预估优化幅度 |
|------|------|------------|
| F-03 | `segmentCaseExpr` 8 层 LIKE + CAST/TRIM 逐行计算 | 物化 `segment_tag` 整数列后 CPU −20~40% |
| F-01 | `EXTRACT(YEAR FROM col)=N` 破坏 DuckDB zonemap | 改 `BETWEEN DATE` 范围扫描后 −30~60% |
| F-02 | `growth/yoy.ts` 双 CTE 各全量扫 PolicyFact | 合并单扫后 −50% |
| B311 | version bump 早于视图物化（竞态） | 消除预热风暴 |
| B340 | claims 域缺 `union_by_name`（定时炸弹） | 防 schema 漂移崩溃 |

**优化方案**：
1. **F-01 快速胜利**：全局替换 `EXTRACT(YEAR FROM ...)=N` 为 `col BETWEEN DATE 'Y-01-01' AND DATE 'Y-12-31'`，用 benchmark 脚本回归
2. **F-02**：合并 `yoy.ts` 双 CTE 为条件聚合单扫
3. **F-03 中期**：在物化层（`duckdb-materialization.ts`）预算 `segment_tag` 列，SQL 生成器改为直接引用整数列
4. **B311**：`setDataVersion` 移到 `createPolicyFactView` 完成后统一调用
5. **B340**：`duckdb-domain-loaders.ts` 9 处 `read_parquet` 统一加 `union_by_name=true`

**关联 BACKLOG**：B306（DuckDB 性能三件套）、B311、B340

### 方向 4：重复代码收拢（影响面：开发效率 + 一致性）

**问题**：全站存在大量重复实现，同一类型的组件/逻辑在多个 feature 模块中独立编写。

| 重复类型 | 实例数 | 具体位置 |
|---------|--------|---------|
| 机构×维度热力图 | 4 套 | performance-org / cross-sell / claims-detail / quote-conversion |
| 机构→团队→业务员下钻表 | 5 处 | 各 feature 模块 |
| KPI 卡片组件 | 5 套 | dashboard / cost / growth / quote / claims |
| 趋势折线图封装 | 5 套 | 各 feature 模块 |
| 导出对话框 | 2 个 + 1 处重写 | `ExportDialog` vs `ExportModal` vs `crossSellExport` |
| 格式化函数本地重写 | 3+ 处 | renewal-tracker / expense-development / growth 各自重写 `shared/utils/formatters` |
| Context value 未 memoize | 2 处 | `StableContext.tsx:181` / `ExportContext.tsx:45` |

**优化方案**：
1. **Phase 1**：提取 `<SharedHeatmap>` / `<SharedDrillTable>` 到 `src/widgets/`，参数化差异
2. **Phase 2**：合并导出组件（`ExportDialog` 胜出，删 `ExportModal`）
3. **Phase 3**：清理格式化函数本地副本，统一从 `shared/utils/formatters` 导入
4. **Phase 4**：memoize Context value（`StableContext` / `ExportContext`），消除无效重渲染

**关联 BACKLOG**：`2026-06-11-claude-3093a3`（重复组件收拢）、`2026-06-11-claude-7dca99`（Context 未 memoize）
