# 架构优化方向与计划

> **审计日期**：2026-06-27 · **审计范围**：全栈（前端 + 后端 + 数据管道 + CI/CD + 治理）

## 二、优化方向（按投入产出比排序）

### 方向 1：生产冷启动 502 根治（P1 · 高影响 · 中投入）

**现状**：PM2 reload 期间全站 502 持续数分钟（BACKLOG B294022），用户可感知。

**优化路径**：
1. 实现 DuckDB 预热完成后再切流（graceful reload）—— PM2 `wait_ready` + `process.send('ready')`
2. 在 `data-bootstrapper.ts`（713 行）中分离"加载"与"就绪"状态
3. `cache-warmer.ts`（852 行）预热完成后才通知 PM2 接流
4. 添加 `/health` 端点的 `ready` 状态（区分 alive vs ready）

**验收标准**：reload 期间无 5xx；`curl /health` 返回 `ready: true` 后才接流。

---

### 方向 2：前端状态管理优化（P2 · 高影响 · 中投入）

**现状**：
- `FilterContext` 任意字段变化触发 48 个消费者全量重渲染
- `StableContext`（6968 行）/ `PermissionContext`（8819 行）/ `BranchContext`（4299 行）体量庞大
- BACKLOG B324 明确指出 StableContext/ExportContext 值未 memoize

**优化路径**：
1. **字段级订阅**：新增 `useFilterField(fieldName)` hook，仅返回单字段 + setter；底层用 `useSyncExternalStore` 或 `useRef` + 选择性通知
2. **Context 拆分**：将 StableContext 按职责拆为 `DateMetaContext` + `FilterOptionsContext` + `TeamMappingContext`
3. **大面板 memo 加固**：对 `PerformanceAnalysisPanel`（882 行）、`CrossSellAnalysisPanel`（891 行）内部派生状态补 `useMemo` 包裹

**验收标准**：React DevTools Profiler 确认筛选器切换时非相关面板零重渲染。

---

### 方向 3：SQL 生成层治理与瘦身（P2 · 中影响 · 中投入）

**现状**：
- `server/src/sql/` 共 55 文件 ~18000 行；6 个文件 > 500 行
- 最大文件：`claims-detail.ts`（624 行）、`performance-analysis/shared.ts`（590 行）、`claims-heatmap.ts`（574 行）
- B305/B310：指标公式硬编码绕过 metric-registry

**优化路径**：
1. **指标注册表对齐**：审计 SQL 生成器中的硬编码公式，迁移到 `metric-registry/categories/*.ts`
2. **SQL Builder 抽象层强化**：`sql-builder.ts`（382 行）已有基础，将重复的 WHERE/GROUP BY/ORDER BY 拼装模式抽取到 builder
3. **拆分大文件**：`claims-detail.ts` 按查询类型拆分（列表 / 聚合 / 热力图 / 地理）
4. **参数化查询**：将正则 sanitizer 替换为参数化绑定（安全 + 性能）

**验收标准**：`bun run governance` 通过 + `golden-baseline.mjs --compare` 零差异 + 无 > 500 行 SQL 生成器。

**现状**：24 个文件 > 400 行，6 个 > 700 行。

| 文件 | 行数 | 拆分建议 |
|------|------|---------|
| `GeoRiskPanel.tsx` | 951 | → GeoNarrativeBanner + GeoKpiGrid + GeoHeatmapDual + GeoInsightCards |

**验收标准**：拆分后每个文件 < 400 行；`bun run build` 零 TS 错误；页面交互行为不变。

### 方向 5：DuckDB 服务集群重构（P2 · 高影响 · 高投入）

**现状**：
- 9 文件 2784 行，`duckdb-domain-loaders.ts` 独占 1077 行
- 立方体（Cube）层首批切流中（趋势/增长/业务员），成本/KPI 立方体生产不可服务（B65f495）
- 物化视图竞态（B311）：版本号更新先于视图物化完成

**优化路径**：
1. **domain-loaders 拆分**：按数据域拆为独立 loader 模块（policy / claims / cross-sell / quotes / renewal / customer-flow）
2. **物化视图生命周期管理**：引入 `MaterializationState` 状态机（pending → building → ready → stale），版本号仅在 ready 时更新
3. **立方体服务化**：将 `duckdb-cube.ts` 的 build/query 分离，build 异步 + query 读最近成功快照
4. **连接池优化**：评估 DuckDB 读写分离（write connection 独占 + read connection 池化）

**验收标准**：冷启动预热 < 10s；物化视图无竞态；cube build 失败不影响查询。

---

### 方向 6：数据管道可靠性加固（P2 · 高影响 · 中投入）

**现状**：
- 84 个 Python ETL 脚本无单元测试
- 同步前无数据质量门禁；赔案新鲜度检查仅治理阶段（事后）
- Parquet 重叠曾导致 31 万行重复（10.8% 膨胀）

**优化路径**：
1. **同步前门禁链**：在 `sync-vps.mjs` 执行前串联检查：赔案新鲜度 + Parquet 重叠 + 省份隔离 + 行数异常
2. **Python ETL 测试骨架**：为 `convert_*.py` 核心转换逻辑添加 pytest 单测（优先覆盖保费/赔案/交叉销售 3 域）
3. **ETL 幂等性保证**：`daily.mjs` 每步产出写入 `ledger/` 事件日志，支持断点续跑
4. **Excel 多 Sheet 加载强制校验**：在 governance 检查中验证 Python 脚本使用 `load_excel_all_sheets()`

**验收标准**：`node scripts/check-data-readiness.mjs` 在同步前跑通全部检查；Python 核心域 3 个转换脚本有 pytest 覆盖。

---

### 方向 7：前端测试覆盖提升（P2 · 中影响 · 中投入）

**现状**：233 源文件仅 26 测试文件（11%），几乎全在 copilot 模块。

**优化路径（分阶段）**：
1. **Phase 1（快赢）**：为 `src/shared/hooks/` 10 个 hook 补齐单测（纯逻辑，不依赖 DOM）
2. **Phase 2（核心路径）**：为 dashboard / cost / growth 3 个高频页面添加组件集成测试
3. **Phase 3（防回归）**：E2E 覆盖权限分级场景（admin vs org_user）
4. **持续机制**：CI 添加覆盖率报告 + 门禁（新代码 > 60% 覆盖率）

**验收标准**：前端测试文件 > 60 个；`bun run test --run` 全过。

---

### 方向 8：ECharts 包体优化（P3 · 低影响 · 低投入）

**现状**：
- 39 个文件引用 echarts，打包约 700KB（已单独 chunk）
- 未做 tree-shaking（echarts 整包引入）

**优化路径**：
1. 将 `import * as echarts from 'echarts'` 改为按需引入 `import { LineChart, BarChart, ... } from 'echarts/charts'`
2. 注册器模式：`echarts.use([LineChart, BarChart, PieChart, MapChart, HeatmapChart, ...])`
3. 预估可削减 30-40% echarts 包体

**验收标准**：`bundle-baseline.html` 中 echarts chunk 体积下降 > 25%。

---

### 方向 9：可观测性补齐（P3 · 中影响 · 中投入）

**现状**：
- 无 API 延迟百分位（p50/p95/p99）
- 无查询缓存命中率监控
- 无前端错误边界上报

**优化路径**：
1. **API 遥测**：在路由中间件添加 histogram 采集（响应时间 + 缓存命中/未命中），写入 `logs/metrics.log`
2. **前端 ErrorBoundary**：全局包裹 + 按 feature 模块分区，错误上报到 `/api/telemetry/error`
3. **哨兵增强**：将 API 延迟 p95 纳入 `etl-anomaly-sentinel` 检查维度

**验收标准**：`/api/telemetry/stats` 返回近 1h 各路由 p50/p95/p99 + 缓存命中率。
