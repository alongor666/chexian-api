# 生产 502 事故根因分析（RCA）— 真因是内核升级重启触发 pm2 resurrect，非缺 CORS_ORIGIN

> **卡片**：backlog uid `2026-07-12-claude-4d85f8`（P1 · Ops/生产事故）
> **RCA 认领方**：502-RCA（只读取证 + 仓内结构性根治）
> **取证时点**：2026-07-12（生产 VPS `chexian-vps` / `162.14.113.44`，`ssh` root 只读逐条勘查）
> **性质**：**推翻事故卡原判定** —— 卡片将根因标为「已证实：缺 CORS_ORIGIN，`cors.ts:42` 旧 fail-fast」，但生产日志证据表明**从未发生 CORS 崩溃**；真因是一次**内核升级重启 → `pm2 resurrect` 对着缺 `express` 的 node_modules 拉起 → 崩溃循环**。

---

## 0. 结论摘要（先行）

1. **原判定被证伪**：生产错误日志（12MB，回溯至 2026-02-14，未轮转）中 `CORS_ORIGIN` / `必须设置 CORS` / `生产环境必需变量` 命中数**全为 0**。事故卡「已证实：`[CORS] 生产环境必须设置 CORS_ORIGIN`」缺乏日志支撑，系**未核日志的推断**。
2. **真实崩溃签名**：`Error: Cannot find package 'express' imported from /var/www/chexian/server/dist/app.js`（`code: ERR_MODULE_NOT_FOUND`），**8 次**，2026-07-12 **21:55:08 → 22:04:38（CST）**。即 node_modules 缺 `express`，启动即崩。
3. **触发事件 = 一次内核升级重启**：VPS 于 **21:54–21:55（CST）重启**（新核 `6.6.117-45.1.oc9`，此前自 4-14 已连跑 89 天）。这正是**卡 10e467 于 33 分钟前（21:22）刚装上的 `pm2-root.service` 开机自启**首次真实触发。
4. **完整因果链（journal 铁证）**：内核重启 → `pm2-root.service` 于 21:55:34 执行 `pm2 resurrect`（从 `dump.pm2` 恢复进程）→ `chexian-api` 以 `node dist/app.js` 拉起，但此刻 node_modules 缺 `express` → 崩溃 → **PM2 层（`max_restarts:5`）与 systemd 层（`pm2-root.service` 21:57:03 `start operation timed out` 后被 systemd 重启再 resurrect）双重循环** → PM2 放弃 → 全站 502。
5. **三候选逐一定性**：① 部分沾边（是"重启触发"，但触发者是**开机 resurrect**而非部署 reload，且失败是**缺 node_modules**而非缺 env）；② 方向对、载荷错（10e467 的 `pm2 save`+`startup` 确是 resurrect 路径的成因，但 `dump.pm2` 存的**不是坏 env**，而是一个 resurrect 时撞上残缺 node_modules 的进程定义）；③ **完全证伪**（生产 dist 是当前版，含 CORS 兜底，非落后版）。
6. **原卡「结构性根治=部署 cors.ts 兜底版」已无意义**：CORS 兜底修复（commit `57907862`）**自 2026-03-25 起已在 main 主干、且已部署到生产**（生产 `dist/config/cors.js` 实测含新兜底 `WARNING: CORS_ORIGIN`）。「缺 CORS_ORIGIN 即硬崩」这一失败模式在代码层**早已消除**。
7. **真正该根治的失败模式**：**「进程被自动拉起（开机 resurrect / 部署 reload）时若 node_modules 残缺，即崩溃循环，且因 `max_restarts:5` 导致 PM2 永久放弃 → 瞬时故障恶化为持续宕机」**。根治方向见 §5。

## 5. 结构性根治（让失败模式不可表达 / 不可持续）

原卡设想的「部署 cors.ts 兜底版」已完成且与本次事故无关（§0.6）。真正该根治的是 §0.7 的失败模式。分三层，按风险/收益排序：

### 根治 A（本 PR 落地 · 最高杠杆 · 最低风险）— 让瞬时崩溃自愈，杜绝"永久宕机"

**问题**：`ecosystem.config.cjs` 的 `max_restarts: 5` 使 PM2 在 5 次快速崩溃后**永久放弃**。node_modules 残缺是**秒级瞬时**（部署 `npm ci` 几十秒内自我修复），但 PM2 已放弃 → 需人工介入 → 瞬时故障恶化为**持续 502**。

**修复**：为 `ecosystem.config.cjs` 增加 `exp_backoff_restart_delay`（PM2 原生指数退避重启）。启用后 PM2 以递增延迟**无限重试**（不再因 `max_restarts` 永久放弃），一旦 node_modules 补全，下一次退避重试即成功 → **自愈**；退避上限约 15s，同样不会像定长重启那样紧密 thrash（保留原 `max_restarts` 抑制 OOM 空转的意图）。

- 生效范围覆盖**开机 resurrect** 与**部署 reload** 两条路径（进程定义级）。
- 验收：`bun run build` + `bun run governance` + 新增配置单测断言 `exp_backoff_restart_delay` 存在；生产验收需人工在维护窗 reload 后观察一次 `pm2 describe`。

### 根治 B（follow-up · 需人工维护窗）— 开机 resurrect 前置依赖完整性闸

**问题**：`pm2 resurrect` 直接 `node dist/app.js`，**绕过** wrapper 的原生模块自愈（`heal_natives_or_exit` 仅在 install/reload/restart 触发）。开机路径对 node_modules 完整性零校验。

**修复方向**：给 `deploy-chexian-api.sh` 增一个受保护的 `resurrect-guarded` 子命令（先校验 + 自愈依赖，再 `pm2 resurrect`），并把 `pm2-root.service` 的 `ExecStart` 指向它。**改 systemd 单元属生产 ops 变更**（需 root + 维护窗），本 PR 只提供 wrapper 子命令与操作说明，**不擅自改生产**。同时把 `express` 纳入 `verify-natives` canary 清单（现仅覆盖 3 个原生模块）。

### 根治 C（follow-up · 部署链 RED LINE · 需人工窗）— 消除 node_modules 空窗

**问题**：部署 `install` 用 `npm ci`（先 `rm -rf node_modules`）本质制造"半装窗口"。deploy.yml 已对**前端 dist** 用 `dist.new` 原子 rename 换轨消除空窗，但 **node_modules 仍是原地擦除重装**。

**修复方向**：node_modules 也走 staging 目录构建 + 原子 rename 换入，消除"擦空到装好"之间任何进程能看到残缺 node_modules 的时间窗。属部署链改造（`.claude/rules/deploy-chain-sop.md` §2 RED LINE，禁 auto-merge、须人工窗），登记为独立 BACKLOG，不并入本 PR。

## 6. 事故卡的过程教训

- **违反真实性协议**：原卡把根因标为「已证实」，但该结论**未核对生产日志**——日志里根本没有 CORS 崩溃。信号（生产 502 + 记得 cors.ts 曾有 fail-fast）被模式匹配到一个**已被 3.5 个月前修复**的旧故障，未做证伪。（呼应 `CLAUDE.md §0` 验证不声称、全局真实性协议 §6「高代价操作前先证伪自己的假设」。）
- **"已止血"需独立复验**：本 RCA 复验时一度观测到外网 502（#1093 部署 reload 预热空窗），数十秒后恢复 200——"止血"是真的，但每次主干推送（含纯 docs PR）都会触发一次 build+reload 的 30–120s 502 空窗，是独立的可用性课题。
