## 三、关键发现（逐条附证据）

### 3.1 「是否报价」不在「04 报价清单」，而在历史"签单清单"宽表

- **04 报价清单**（报价转化的原始源，ETL = `数据管理/pipelines/quote_etl.py`）实测为 **33 列**：`报价时间 / 车架号 / 险类 / 三级机构 / 险别组合 / 客户类别 / … / 续转保 / 是否过户车 / 是否新能源车 / 是否电销 / 是否承保 / … / 最终报价`。**不含「是否报价」，也不含「续保单号」**。它表达报价是否成交用的是「是否承保」（值为"承保"/"未承保"）与「续转保」。
- 对应产物 `warehouse/fact/quotes_conversion/latest.parquet`（88 万行）同样**不含**这两列——印证 B255 描述里"处理后 parquet 不含是否报价/续保单号"的观察，但其归因（"它们是 04 报价清单的 ETL 前源字段"）**不成立**。
- 「是否报价」的真实历史归属是"签单清单"宽表：`数据管理/字段字典.md` 第 261 行（第 27 号字段）记录「是否报价」为字符串、历史非空 681,210 条、唯一值 2 个（即"是 / 否"）；同表还有「批改类型」「案件数」「赔款合计」「总费用金额」等——都是早期签单清单宽表的字段。

> **教训**：B255 账本与上一轮排查把源文件指向了「04 报价清单」，但「是否报价」从未在该文件中。动手前用 `DESCRIBE` / 表头直读亲眼核实列归属，可一次纠偏。

### 3.2 现行全部签单清单源数据已移除「是否报价」

用流式解析（`zipfile` + `iterparse`，规避超大文件全量加载）直读 4 个签单清单文件首行表头，结果一致：

| 签单清单文件（按文件名时间范围） | 列数 | 含「是否报价」 | 含「续保单号」 |
|---|---|---|---|
| `20210101-20231231_01_签单清单_定稿.xlsx` | 44 | 否 | 是 |
| `20240101-20250531_01_签单清单_定稿.xlsx` | 44 | 否 | 是 |
| `20250601-20260531_01_签单清单_定稿.xlsx` | 44 | 否 | 是 |
| `20260601-20260613_01_签单清单_定稿.xlsx` | 44 | 否 | 是 |

- 四个文件覆盖 **2021-01-01 ~ 2026-06-13**，**无一含「是否报价」**，**全部含「续保单号」**。
- 该移除是**有意为之**：`数据管理/shard-config.json` 第 22 行把「是否报价」列入"显式忽略字段"（`explicitly_ignored_fields`），备注原文「口径不可靠，待用户修正」。即上游已主动放弃该字段，ETL Schema 契约不再要求它。
- `字段字典.md` 第 27 号「是否报价」条目（681,210 行、2 个值）是**早期宽表快照的陈旧字典残留**，不代表现行数据。

### 3.3 现行续保判定 `is_renewal` 即"续保单号非空"，260 万行 0 不一致

派生链（全部可在代码中核对）：

1. 源签单清单：`续保单号`（保单号格式字符串，非空即为续保）；
2. `数据管理/pipelines/transform.py` 第 383 行：`df['是否续保'] = df['续保单号'].notna()`（"是否续保"由"续保单号非空"直接派生）；
3. 字段映射：`续保单号 → renewal_policy_no`、`是否续保 → is_renewal`；
4. 落入保单事实表 `warehouse/fact/policy/current/*.parquet`。

对该表（**2,600,421 行 / 2,530,856 张唯一保单 / 起期 2020-01-06 ~ 2026-09-09**）做交叉核对：

| 续保单号口径 | 是否续保口径（is_renewal） | 行数 | 占比 |
|---|---|---|---|
| 续保单号非空 | is_renewal = 真 | 947,069 | 36.42% |
| 续保单号空 | is_renewal = 假 | 1,653,352 | 63.58% |
| 续保单号非空 | is_renewal = 假 | **0** | 0.00% |
| 续保单号空 | is_renewal = 真 | **0** | 0.00% |

- 显式核对"两口径不一致行数 = **0**"。即现行 `is_renewal` 与「续保单号非空」是**结构性 1:1 等价**，不存在任何方向的差异。
- 近 12 个月起期（2025-07-01 ~ 2026-06-30）切片同样自洽：续保（续保单号非空）205,548 件（39.26%）、新保（续保单号空）318,068 件（60.74%）。

**这正是 B255 要求的"以续保单号非空判定"——它已经是现行系统的实现，而非待迁移的目标。**

### 3.4 「是否报价」无任何现行 SQL / ETL 消费方

- **报价转化率 / 承保率**（`server/src/sql/quote-conversion.ts` 第 114-116 行）：口径为 `承保率 = 已承保件数 ÷ 总报价件数`，其中"已承保"判定用 `is_underwritten = '承保'`（来自 04 报价清单的「是否承保」），续保 / 转保拆分用 `renewal_status`（「续转保」）。**全程不引用「是否报价」。**
- **交叉销售**（`server/src/sql/cross-sell.ts`）：基于保单事实表的交叉销售标识（`is_cross_sell`）。**不引用「是否报价」。**
- **续保追踪**（`server/src/sql/renewal-tracker.ts`）：用 `is_quoted` / `first_quote_time` / `quote_count`（由保单与 04 报价清单匹配派生），以及 `is_renewal`（即续保单号非空）。注意 `is_quoted`（是否报过价）≠ `is_quote`（旧"是否报价"字段）——前者来自报价清单匹配，**不是被弃用的「是否报价」**。
- 退役证据：历史上消费「是否报价」的是 `transform.py` 第 1013-1024 行的 quotes 域（"只保留 是否报价=True 且 续保单号非空"，产物 `warehouse/fact/quotes/`）。该产物目录**已不存在**，quotes 域已退役；账本所列 `convert_quotes_v2.py` 文件也**已不存在**。`transform.py` 第 588 行的「是否报价」处理分支（`if '是否报价' in df.columns`）因现行签单清单无此列而**永不触发**。

## 五、对报价转化率与交叉销售的影响估算，及"不可计算项"说明

### 5.1 影响估算

**对现行报价转化率、交叉销售两条口径的影响均为 0。** 原因是这两条口径在现行实现中**从未引用「是否报价」**（见 3.4）。即便历史上「是否报价」曾被某个已退役口径使用，现行链路中也已不存在该依赖，因此切换 / 移除「是否报价」对现行对外指标**无任何数值影响**。

若 owner 关心的是"用续保单号判定续保 / 报价"本身的口径质量：现行续保识别（`is_renewal`）已等同"续保单号非空"，全样本 0 误差（3.3），口径稳健。

### 5.2 不可计算项（诚实声明）

B255 最初设想的"原始口径（是否报价"是/否"）× 新口径（续保单号"空/非空"）"**逐方向差异交叉表无法在现行数据上产出**，因为：

- 现行全部在盘签单清单与报价清单均**不含「是否报价」列**（3.1、3.2 实测）；
- 归档目录（`.archive` / `.xlsx-archive` / `staging`）中亦无保留「是否报价」的旧格式签单清单；
- 字段字典里"非空 681,210、唯一值 2"的旧统计来自早期宽表快照，**其原始明细数据已不在盘**，无法回放逐行交叉。

因此"历史上是否报价误标了多少、误标方向如何"这一问题，**在现有数据资产下不可量化**。如确需量化历史误差，需 owner 提供仍含「是否报价」列的旧版签单清单原始导出（或上游 BI 重新导出该列）。
