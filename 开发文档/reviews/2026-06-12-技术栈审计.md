## 一、结论先行

**总体判断：技术栈选型健康、核心框架（React 19 / TypeScript 5.9 / DuckDB / Express）处于活跃维护线，但存在 4 个需要排期处理的问题：**

| 优先级 | 问题 | 影响 |
|--------|------|------|
| **P0** | `server/` 安全审计共 **19 条通告（1 条严重、12 条高危）**；其中 Express 4.22.1 自带的路由解析库（path-to-regexp）高危正则拒绝服务，**升级到区间内的 4.22.2 即可修复** | 生产后端直接暴露面 |
| **P0** | Vitest 2.1.9 命中**严重级**通告（Vitest 界面服务器监听时可被任意读取并执行文件）——仅开发环境受影响，生产不受影响，但修复要求升到 3.2.6 以上（跨大版本） | 开发机 |
| **P1** | 文件上传库 multer 停留在 1.x（`1.4.5-lts.2`），该版本线已被上游官方弃用，2.1.1 为当前维护线；使用点仅 `server/src/routes/data.ts` | 上传接口长期无补丁 |
| **P1** | `开发文档/TECH_STACK.md` 与实际代码**多处漂移**（详见第五节）：文档声称的 Apache Arrow 17.0.0 依赖在仓库中不存在、目录结构清单约为实际规模的 1/7、API 子客户端数量与 CLAUDE.md 矛盾 | 该文档自称"唯一事实来源、所有开发任务前必读"，漂移会误导后续协作 |

另有一组**工具链治理缺口**（第六节）：无任何 lint 工具但 server 的 lint 脚本假装存在根级 lint、无 engines/Node 版本声明、CI 各工作流 Node 版本不一致、Python ETL 无依赖声明文件。

## 四、安全审计（bun audit，2026-06-12）

### 4.1 server/（生产后端，最需要关注）

**共 19 条通告：1 条严重、12 条高危、5 条中危、1 条低危。** 按暴露面分组：

| 组 | 通告 | 暴露面 | 处置 |
|----|------|--------|------|
| **生产运行时** | express → path-to-regexp <0.1.13 **高危**（多路由参数正则拒绝服务）；express → body-parser → qs 中危（拒绝服务） | 生产 API 直接暴露 | **`bun update` 升 express 4.22.2 区间内修复，最优先** |
| **开发依赖** | vitest <3.2.6 **严重**（Vitest 界面服务器监听时任意文件读取与执行）；其 vite/rollup/esbuild 传递链另有 1 高危 3 中危 | 仅开发机，且需主动开 Vitest 界面模式 | 随 vitest 大版本升级修复（P1） |
| **安装时工具链** | bcrypt → node-pre-gyp → tar 6 条高危（解包路径穿越）、minimatch 3 条高危、brace-expansion 中危 | 仅 `bun install` 安装瞬间，不进运行时 | 随 bcrypt 6 升级消除（P2），短期风险可接受 |
| **开发热重载** | tsx → esbuild 中危（开发服务器可被任意网站读响应） | 仅开发机 | tsx 区间内更新可缓解部分 |

## 五、文档漂移：`开发文档/TECH_STACK.md` vs 实际

该文档自称"唯一事实来源、所有开发任务前必读"，最后一次实质对齐是 2026-04-24。本次核对发现：

| 文档声称 | 实际 | 严重度 |
|----------|------|--------|
| §1.2 后端依赖含 "Apache Arrow 17.0.0 - 内存数据格式" | 全仓库 grep `apache-arrow` **零命中**，server/package.json 无此依赖 | **高**（虚构依赖） |
| §1.2 "Zod 4.0.0" | 实际 4.3.6 | 低 |
| §2 API 客户端 "10 个 `*-api.ts` 命名空间子客户端" | CLAUDE.md §4 已是 **13 域**（新增 premium/geo/patrol） | 中（两份必读文档互相矛盾） |
| §7 后端目录结构：`sql/` 列 12 个平铺文件、`routes/` 列 5 个文件 | 实际 `server/src/sql/` 共 **83 个 .ts 文件**（31 顶层 + 23 子目录拆分），`routes/query/` 下 **25 个子路由文件**；`config/` 清单缺 metric-registry、field-registry、env.ts、api-routes.ts、route-param-contracts.ts、preset-users.ts 等核心注册表 | **高**（清单约为实际规模 1/7，按此导航必然找错） |
| §1 未提及 | react-router-dom 7、@tanstack/react-query 5、better-sqlite3（PAT 存储）、lru-cache（路由缓存）、multer（上传）、Playwright（E2E）、`cli/`/`mcp/` 子包、Service Worker（CLAUDE.md §4 核心链路） | 中 |

**建议**：将 §1（版本清单）和 §7（目录结构）改为"指针 + 生成"模式——版本以 package.json 为唯一事实源、目录结构指向 `开发文档/00_index/CODE_INDEX.md`，避免再次靠人手同步。§3（DuckDB 约束）、§4-§6（验证协议与陷阱）仍然准确，保留。

## 六、工具链治理缺口

| 缺口 | 现状 | 建议 |
|------|------|------|
| **无 lint 工具** | 仓库无 ESLint / Prettier / Biome 任何配置；而 `server/package.json` 的 lint 脚本是 `echo 'Use root-level linting'`——**根包根本没有 lint 脚本**，指向不存在的东西 | 二选一：引入 Biome（与 Bun 生态契合、零依赖冲突）做 lint+format；或删除 server 的占位脚本，明确"本项目以 typecheck + governance 代替 lint"并写入 TECH_STACK.md |
| **无 Node 版本声明** | 无 engines / .nvmrc；CI 哨兵用 Node 20、有环境跑 Node 22 | 在根与 server 的 package.json 加 `engines`，并放一个 `.nvmrc`；统一 CI 工作流 Node 版本；与生产 VPS 实际 Node 版本核对后定基线 |
| **Python ETL 无依赖声明** | `数据管理/` 大量 Python 脚本（ETL、维度表、校验）依赖系统 python3 + duckdb 等模块，仓库内无 requirements.txt / pyproject.toml | 增加一份最小 requirements.txt（duckdb、openpyxl 等），否则新机器/新协作者无法复现 ETL 环境 |
| zod 双版本 | 根锁文件同时存在 zod 4.3.5 与 4.3.6 | `bun update zod` 去重 |
