### E03 | 报价/客户流向整页 400 的根因定位（BACKLOG 8f71c0，此前"根因待定位"）
- **动机**：山西 13 账号验证报告称 /quote-conversion/* 与 /customer-flow/* 报「列不存在：policy_date」整页不可用——这是"每个页面见山西数据"的最大缺口。
- **假设 H1**：SX Parquet 缺列（backlog 原猜测）。**证伪**：SQL 生成器静态不引用 policy_date。
- **假设 H2**：通用筛选构建器把 startDate/endDate 默认拼成 `policy_date >=`（filter-params.ts:114），而 QuoteConversion 视图（只有 quote_time）/ CustomerFlow（只有 insurance_start_date）无此列 → Binder Error。**证据**：
  1. buildConditionsFromFilterParams 默认 dateField='policy_date'；quote/customer-flow 路由全部直接 parseFiltersAndBuildWhere(req) 不做净化；
  2. chart-ledger 前端 hook（useChartLedgerData.ts:107-111）**注释明确记载了同一根因**并已在前端绕过（startDate→dateStart 翻译）；
  3. 前端报价页 hook 只发 dateStart/dateEnd → 浏览器内页面不触发；验证脚本/MCP/CLI 直传 startDate/endDate 才触发。
- **推论（修正认知）**：该 bug **不分省**——四川账号带 startDate/endDate 打这两组路由同样 500。"SX 专属"是验证只测了 SX 账号造成的归因偏差。
- **修法**：路由层净化（quote：startDate/endDate→dateStart/dateEnd 域内翻译；customer-flow：dateField 强制 insurance_start_date），用 parseFiltersAndBuildWhere 既有 queryOverride 注入口。
- **环境事实**：@duckdb/node-api 1.4.4 在本容器可用（bun -e 实测 SELECT 42 通过）→ 可跑 test:integration 真 SQL 验证。

## 阶段二：成本优化研究（≥25 实验 · Opus 统筹 · 假设先行）

### E06 | 研究对象判读（先于一切实验的方法论决策）
- **歧义**："成本优化研究"两种解读——(a) 车险业务成本口径研究（满期赔付率/费用率/综合成本率）；(b) AI 算力成本优化研究（哪类任务用哪档模型，即目标里"小任务交给便宜模型，重任务保留全力运行"的路由策略实证）。
- **判定 (b)**，理由：① 目标原文"让 **Opus** 负责统筹"+"自行判断选择**低算力模型**并在 subagent 中运行"——通篇在谈 AI 模型分工，(b) 是对该策略本身的实证研究；② 决定性约束：本容器**无 Parquet 数据、无生产凭据**，(a) 完全不可执行（若用户本意是 (a)，需在有数据的环境跑，此处如实登记为环境缺口）；③ (b) 的产出（模型路由策略）对本项目长期 AI 协作成本直接有价值。
- **研究问题**：在本仓库的真实任务类型上，haiku / sonnet / opus 三档模型的「质量 × 令牌成本」边界在哪里？哪类任务降档零损失，哪类必须保档？
- **方法**：7 类真实任务 × 3 档模型 = 21 核心实验 + 方差重复 + 评估方法修正轮 ≥ 25 次。每次实验先写假设/预期再启动；正确性用确定性判据（grep 基准/测试通过）优先，主观类用 Opus 评审员按量规打分；成本用 subagent 实际令牌数。
- **任务矩阵**（全部取自本仓库真实工作负载，均有基准答案）：
  | 任务 | 类型 | 判据 |
  |---|---|---|
  | A 符号检索 | 找 resolveBranchRlsCode 全部调用点 | grep 基准（确定性） |
  | B 业务口径问答 | 推介率分母/赔付率分子/渗透率定义 | 规则字典基准（Opus 评审） |
  | C 根因追踪 | 解释 quote 路由为何需要净化 startDate | 已知根因链（Opus 评审） |
  | D 单测编写 | 为 isBranchSummaryRow 补边界测试 | vitest 跑通 + 覆盖度（确定性+评审） |
  | E bug 修复 | worktree 内修复注入的缺陷（有失败测试） | 测试转绿（确定性） |
  | F 文档摘要 | 山西 cutover 452 行接力文档 → 10 行简报 | 要点命中率（Opus 评审，我持基准） |
  | G 多文件审计 | src/ 非测试文件 '四川' 硬编码清单 | grep 基准（确定性） |
- **先验假设（总）**：A/G（机械检索）haiku 零损失省 ~10×；B/F（读文档综合）sonnet 够用；C（因果链跨 3 层）与 E（改码验证）haiku 显著掉质量；D 分界在 sonnet。若实验推翻，按数据修正。

### E07 | Opus 统筹审查（设计修订轮）
- **动机**：按目标"让 Opus 负责统筹"，实验前把设计交 Opus 审查。
- **假设**：设计有可修的盲点，但框架能过。
- **实际结果**：Opus 抓出 3 个 P0——① **基准口径未固化**（A 题 grep 原始命中 85 处/30 文件，我的"27 处"只在一套没写进提示词的排除规则下成立；G 题 18 也依赖"是否算 .json 资产"的未声明规则——测的会是"猜口径"不是"能力"）；② n=1 无方差不能下结论；③ 成本只数总令牌不拆输入/输出/缓存，且漏掉低档模型多轮试错的往返成本。另有盲评（评审员自偏）、失败模式枚举、天花板效应、预注册等 6 项 P1/P2。
- **采纳的修订**：口径规则逐字写进提示词 + 集合相等评分（precision/recall）；40 格矩阵（判别力高的 C 每档 3 次、D/E/B/F 每档 2 次、A/G 只跑 haiku+sonnet、加 sanity 控制格）；提示词存盘记 sha256 三档同 hash；subagent_type 恒 general-purpose 只变 model；执行顺序随机化（seed=42）；worktree/答案文件用中性编号防模型身份泄露给被试与评审；B/C/F 盲评（答案匿名乱序）+ 量规写死扣分点；失败模式枚举记录。
- **声明的已知残差**（不假装消除）：harness 只回报 subagent 总令牌数（不拆输入/缓存/输出）→ 成本为代理指标，用"总令牌 × 档位单价比"估算并在报告声明口径；各档模型推理档位用默认值。

### E10 | 成本折算模型（定价来自官方缓存表 2026-06-24）
- 每百万令牌单价：haiku 4.5 = $1 输入/$5 输出；sonnet = $3/$15；opus 4.8 = $5/$25。
- **关键便利**：三档的输入与输出单价比率相同（1:3:5）→ 相对成本 = 总令牌数 × 档位系数{haiku 1, sonnet 3, opus 5}，与输入/输出构成无关。
- 声明残差：① harness 的 subagent_tokens 不拆缓存读（0.1×折扣），缓存命中率高的运行成本被高估——三档同样受影响，方向性结论稳健；② 固定开销 ~39-46k 令牌/次（系统提示词+工具清单，见 S 组 sanity：haiku 38.8k/sonnet 46.8k/opus 40.9k）主导小任务成本。

### E37 | 环境无效格的意外收获（失败路线的价值）
6 个拿到空路径的格子成为一次"意外的鲁棒性压力测试"：sonnet 4 格全部自救（搜索文件系统找到错位目录把活干完，代价 86 轮工具/15 分钟/12.5 万令牌级）；haiku 2 格中 1 格迷路后把测试写进他人工作区（交叉污染）、1 格声称完成但产物未落盘。**结论：环境异常下 sonnet 的"执拗自救"烧钱但不出错，haiku 会静默错交付——自动化流水线里 haiku 必须配确定性验收闸。** 6 格合计烧 60.5 万令牌（正常格的 2 倍均值），是"环境错误比模型降档更贵"的实证。
