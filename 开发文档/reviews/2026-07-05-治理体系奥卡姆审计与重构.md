## 3. 治理系统的"自我违例"（审计核心发现）

体系对外执法严格，对内曾三处破戒，本轮全部修复：

1. **违反自己的 SSOT 闸**：cube 三张清单在治理脚本内手工同步，而「5路由清单SSOT」闸恰恰禁止 inline 清单 → 批次三下沉 SSOT。
2. **违反自己的防漂移闸**：「SQL模块数一致」强制维护会漂移的计数快照 → 批次一删除改指针。
3. **违反自己的 expires 哲学**：pr-evolution 条目强制 expires，检查自身却无生命周期 → 批次三 retireWhen 字段 + 批次四 harness 棘轮（H5 主脚本 ≤4000 行）。

> 原则沉淀：**治理系统必须能治理自己**——新增任何"对外规则"前先问它对治理代码自身是否成立。

## 5. 有意保留的宽松点（诚实边界）

- `filter-params-mapping` 存量 16 处裸 marker（无理由字段）**不强制迁移**：语义受限（单行单规则豁免），为形式一致制造 16 处噪音 diff 不符合奥卡姆；新词根已被 H6 拦住。
- `DC-002 Exception` 旧上下文豁免文法保留（allowContext 机制），历史 SQL 生成器沿用。
- pattern 引擎是逐行/内容正则，不是语义分析——跨行拆写可绕过（与原实现同等强度，规则注释已诚实标注）；语义级方案（semgrep/ESLint AST）经 B321 判定为"防假想敌"，维持自研轻量路线。
- Claude push 钩子删除后，若用户绕过 git hook（`--no-verify`）且 CI 关闭，治理无本地拦截——接受此残余风险（CI 是强制层）。

## 6. 外部对标要点

- **架构适应度函数**（fitness functions，《Building Evolutionary Architectures》）：每条治理检查应像单元测试一样管理——有守护对象、触发时机、**退役条件**。本项目"每闸溯源真实事故"优于多数团队，缺的正是生命周期管理（已补 retireWhen/H5/H6）。
- **dependency-cruiser / eslint-plugin-boundaries**：分层依赖闸的业界标准替代，但 B321 已判引入 ESLint 全家桶为过度工程；自研 AST 闸维持，若边界规则超 ~15 条再重估。
- **semgrep**：模式闸的业界标准（YAML 声明式规则 + 统一豁免注释 + 规则测试框架）——本轮的 pattern-rules.mjs + fixture 完备性断言即其零依赖等价物。
- **棘轮（betterer 模式）**：遗留额度只降不升——已用于 H5 行数上限；DarkMode 额度待其变动时同法处理。
