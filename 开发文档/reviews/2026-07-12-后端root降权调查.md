## 0. 结论摘要（先行）

1. **面向公网的后端 `node dist/app.js`（监听 127.0.0.1:3000）确实以 root 运行**，由 root 名下 PM2 God Daemon（v6.0.14，`/root/.pm2`）托管——实证无误。
2. **这是"设计的必然结果"，但根因是历史 bootstrap 惯性，不是任何安全或功能需求。** wrapper 硬编码探测 `/root/.nvm` 下的 PM2 且经 sudoers 以 root 调用 → PM2 与 app 必然是 root。整条 `npm ci`（root）→ `chown 回 deployer` 的机制正是这一身份错配的症状。
3. **CLAUDE.md §8「deployer 用户管 PM2」是措辞误导**：deployer 只是 SSH 登录 + sudo 调用者身份；wrapper 内部一切（PM2 守护、app 进程、`npm ci`）都以 root 执行。
4. **降权在工程上可行且能顺带净化架构**（消除 chown 回环），但**不是一条命令**：核心阻塞是 **deployer 侧完全没有 node/pm2 环境**（node 仅存在于 `/root/.nvm`），外加若干 root 属主运行时文件（`state.db`、日志）需归属迁移。有秒级停机窗口。
5. **:3000 是非特权端口（>1024）且绑 localhost，绑定不需要 root——端口不是阻塞项。**
6. 顺带发现**独立新缺口**：PM2 无 systemd startup 单元、无 `@reboot` cron、`dump.pm2` 停在 2026-05-23 —— **VPS 重启后应用不会自动拉起**（已由 04:34 会话另立新卡；见 §7）。
7. **建议**：维持 P2，采用**方案 A（systemd `User=deployer`）**作为终态——它同时根治「root 运行」与「无开机自启」两个缺口。**部署决策留给用户**；批准后再单独排期迁移（走部署链 RED LINE 流程，禁 auto-merge）。

## 2. 是不是有意设计？

**判定：机制层面刻意，根因是历史惯性——不是安全需求，可安全降权。**

### 支撑「刻意机制」的证据
- `deploy/vps-wrapper/deploy-chexian-api.sh` 的 `auto_detect_nvm()` **硬编码** `/root/.nvm`（第 16-25 行），只信任 root nvm 下的 PM2。
- `deployer-pm2.sudoers`：`deployer ALL=(root) NOPASSWD: /usr/local/bin/deploy-chexian-api` —— 明确让 deployer 以 root 跑 wrapper。
- deploy.yml 全程以 `sudo /usr/local/bin/deploy-chexian-api {install,reload,...}` 操作 → 必然 root。

### 支撑「根因是历史惯性、非需求」的证据
- **没有任何东西要求 root**：端口 3000 非特权（>1024）、绑 localhost（证据#1）；app 只读 parquet、写 `data/state.db`，无特权系统调用；server 代码无 `eval`/`child_process`（FIND-003 前提已确认）。
- wrapper 注释自称「最小权限原则」，但最小权限只落在 **deployer→wrapper 命令白名单**边界，**没落到运行时进程身份**——典型的"安全设计只做了一半"。
- **root 是副产品**：PM2 最初以 `sudo pm2 start` bootstrap → `PM2_HOME=/root/.pm2` → app 继承 root。wrapper 里 `npm ci`（root 跑）之后 `chown -R deployer:deployer node_modules`（第 121、138 行）的整套回环，正是"为了让 deployer 能在回滚时 `rm -rf` root 建出来的文件"——**这套复杂度本身就是 root 身份错配的症状**。若 app + npm ci 都以 deployer 跑，这套 chown 回环可**整体消失**（净简化）。

**结论**：现状是刻意机制的必然产物，但当初选 root 是 bootstrap 路径最省事，而非深思熟虑的决定。降权不违背任何设计意图，反而更贴合 wrapper 自述的「最小权限」目标。

## 3. 降权阻塞项清单（两方案共通）

| 阻塞 | 现状 | 迁移动作 |
|------|------|----------|
| B1 · deployer 无 node | node 仅在 `/root/.nvm`（证据#9/#10） | 为 deployer 装 node v22.x（独立 nvm 或系统级 `/usr/local/bin/node`），**版本须匹配以复用已编译原生模块，否则 rebuild** |
| B2 · state.db root 属主 | `data/state.db` 等 root:root（证据#12） | 一次性 `chown deployer:deployer data/state.db*`（含 `-wal`/`-shm`）+ `api_tokens.json` + `chexian.duckdb.wal` |
| B3 · 日志目录 root 属主 | `/var/log/chexian` root:root（证据#13） | `chown -R deployer:deployer /var/log/chexian`（或改 ecosystem 日志路径到 deployer 可写处） |
| B4 · PM2 实例是 root | `/root/.pm2` 活跃（证据#5） | 停 root PM2 的 chexian-api → 以 deployer 建新实例/或改 systemd（见方案） |
| B5 · 原生模块 ABI | 按 root node v22 编译（证据#14） | node 同为 v22.x 则可直接复用；否则 deployer 身份 `npm rebuild @duckdb/node-api bcrypt better-sqlite3` |
| B6 · 发布链 reload 路径 | deploy.yml 调 `sudo wrapper reload` | 相应改造（见各方案）；属部署链 RED LINE，禁 auto-merge |

**非阻塞**：端口绑定（3000>1024 且 localhost）、应用/数据目录属主（已 deployer）、reports 目录（已 deployer）、nginx 反代（与 app 身份无关）。

## 4. 方案 A（推荐）：systemd `User=deployer` 托管

用 systemd 直接以非特权用户跑 app，**替换 PM2**。一并根治 §7 的无开机自启缺口。

### 单元草案 `/etc/systemd/system/chexian-api.service`
```ini
[Unit]
Description=chexian-api backend
After=network.target

[Service]
Type=notify              # 复用 ecosystem 现有 wait_ready（app.ts 发 ready 信号）
User=deployer
Group=deployer
WorkingDirectory=/var/www/chexian/server
# 环境从 ecosystem.config.cjs 迁到 EnvironmentFile（见 §6 草案）
EnvironmentFile=/var/www/chexian/server/.env.production
ExecStart=/usr/local/bin/node --max-old-space-size=3072 dist/app.js
Restart=on-failure
RestartSec=5
StartLimitBurst=5
TimeoutStartSec=120
# 纵深加固（降权后可叠加）
NoNewPrivileges=true
ProtectSystem=strict
ReadWritePaths=/var/www/chexian/server/data /var/log/chexian
PrivateTmp=true

[Install]
WantedBy=multi-user.target
```
> `Type=notify` 需 app 用 `sd_notify`（或退化为 `Type=simple` + `TimeoutStartSec` 覆盖 30-60s 加载）。若不想动 app，先用 `Type=simple`。

### 发布链改造
- deploy.yml 的 `sudo wrapper reload` → `sudo systemctl restart chexian-api`。
- sudoers 收窄为：`deployer ALL=(root) NOPASSWD: /bin/systemctl restart chexian-api, /bin/systemctl status chexian-api`。
- wrapper 的原生模块自愈（`heal_natives_or_exit`）迁为 `ExecStartPre=` 脚本，或保留 wrapper 仅作 `install`（`npm ci` 也改 deployer 身份跑 → chown 回环消失）。

### 优点 / 风险
- ✅ 终态干净：非特权 + 标准 init 托管 + 开机自启 + systemd 沙盒加固（`ProtectSystem`/`NoNewPrivileges`）。
- ✅ 消除 root→deployer chown 回环整套复杂度。
- ⚠️ 改动面最大（新增 unit + 改 deploy.yml + 改 sudoers + 环境变量搬家 + 可能动 app 的 ready 信号）。属部署链 RED LINE。
- ⚠️ ecosystem 里的 `max_memory_restart`/`kill_timeout` 语义需用 systemd 对应项（`MemoryMax`/`TimeoutStopSec`）重表达。

## 5. 方案 B：deployer 自有 PM2 实例（改动最小）

保留 PM2 架构，只把实例从 root 换成 deployer。

### 动作
1. 为 deployer 装 nvm + node v22.22.0 + `npm i -g pm2`（`/home/deployer/.pm2`）。
2. wrapper `auto_detect_nvm()` 改指 `/home/deployer/.nvm`；因 PM2 由 deployer 自己跑，**去掉 sudoers/sudo**——deployer 直接 `pm2 ...`。
3. `pm2 startup systemd -u deployer --hp /home/deployer` 生成 `pm2-deployer.service`（补开机自启）+ `pm2 save`。
4. 停并删 root PM2 里的 chexian-api（`sudo /usr/local/bin/deploy-chexian-api stop` 后 root `pm2 delete`）。
5. B2/B3 属主迁移；B5 视 node 版本决定是否 rebuild。

### 优点 / 风险
- ✅ 沿用现有 PM2 心智模型与自愈逻辑，deploy.yml 改动较小（去 sudo）。
- ✅ `pm2 startup` 顺带补开机自启。
- ⚠️ 仍保留 chown 回环的部分复杂度（npm ci 改 deployer 跑才能消，需一并改）。
- ⚠️ deployer 需要一套完整 node 工具链，攻击面比方案 A 的 systemd 略大。
- ⚠️ 两个 PM2 实例（root 残留 + deployer 新）迁移窗口内需谨慎，避免双跑抢 :3000。

## 6. 迁移与回退步骤 + 验收标准

> 以下为**可执行草案**，供用户批准后按部署链 RED LINE（`.claude/rules/deploy-chain-sop.md`）单独排期执行。**本调查不执行任何一步。**

### 迁移（以方案 A 为例，选低峰窗口）
1. 备份：`sudo cp -r /root/.pm2 /root/.pm2.bak.$(date +%s)`；记录当前 `pm2 describe chexian-api`。
2. 装非特权 node：`/usr/local/bin/node` v22.22.0（deployer 与 root 共用；避免 ABI 漂移）。
3. 属主归一：`sudo chown deployer:deployer /var/www/chexian/server/data/state.db* .../api_tokens.json .../chexian.duckdb.wal`；`sudo chown -R deployer:deployer /var/log/chexian`。
4. 原生模块：以 deployer 身份 `require` 自检，失败则 `npm rebuild`（§3 B5）。
5. 落 systemd unit + `EnvironmentFile`（从 ecosystem env 块搬 §6 草案），`systemctl daemon-reload`。
6. 切换：`sudo /usr/local/bin/deploy-chexian-api stop`（停 root PM2 的 app）→ `sudo systemctl enable --now chexian-api`。
7. 改 deploy.yml + sudoers（新 PR，禁 auto-merge，人工盯 5 分钟）。

### 回退（任一步失败）
- 立即 `sudo systemctl disable --now chexian-api` → `sudo /usr/local/bin/deploy-chexian-api start`（root PM2 拉回旧 app）→ 属主可留 deployer（root 能读写，不影响回跑）。
- deploy.yml/sudoers 未合并则无需回退；已合并则 revert PR。

### 验收标准（全部满足才算 DONE）
1. **进程属主 = 非特权**：`ps -o user -p $(ss -tlnp|grep :3000|grep -oP 'pid=\K[0-9]+')` 返回 `deployer`。
2. **健康**：`curl -s localhost:3000/health | grep '"success":true'` 命中。
3. **核心 API**：至少一个 `/api/query/*` 返回 200 + 非空 JSON。
4. **发布链可 reload**：跑一次 `systemctl restart chexian-api`（或新 reload 路径）后 §验收 1-3 仍全绿。
5. **写路径正常**：登录一次（触发 state.db 写）成功，无权限报错。
6. **（方案 A 附赠）开机自启**：`systemctl is-enabled chexian-api` = enabled。

## 7. 顺带发现的独立缺口：无开机自启

取证#7 实证：**无 pm2 systemd 单元、无 `@reboot` cron、`dump.pm2` 停在 2026-05-23**。当前 PM2 守护仅因 double-fork 挂在 init 下存活，**VPS 一旦重启，chexian-api 不会自动拉起**（且 dump 过期，`pm2 resurrect` 也未接线）。

- 这是**独立于 root 降权**的可用性缺口，已由 2026-07-12 04:34 会话另立新卡。
- **两方案均顺带修复它**（方案 A 的 `WantedBy=multi-user.target`；方案 B 的 `pm2 startup`）——是"合并整改"的加分项。
- 注意任务背景所述「systemd 拉起」与实证不符：**当前并无 systemd 单元托管 PM2**，此点已更正。
