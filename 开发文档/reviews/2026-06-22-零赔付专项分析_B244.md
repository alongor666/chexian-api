# 零赔付专项分析（B244）—— 立案有准备金、结案零赔付的规模与准备金释放效应

> 任务编号：B244（账本 uid `2026-04-11-claude-b244`）
> 性质：**只读数据分析报告。未改动任何生产 SQL / ETL。** 全部结论由 DuckDB 直查 Parquet 得出。
> 出具日期：2026-06-22（v2，已纳入 codex 对抗审查 P1/P2 修订，见文末「审查修订记录」）
> 数据基线：`数据管理/warehouse/fact/claims_detail/claims_*.parquet`（赔案明细域，按保单起期年度分区 2019~2026，共 288,198 行，唯一赔付数据源）
> 两套口径必须区分（本报告 v1 曾混淆，v2 已修正）：
> - **通用赔付公式**（`claims_detail/SCHEMA.md` §4）：`CASE WHEN settlement_time IS NOT NULL THEN settled_amount ELSE reserve_amount END`（已结取已决、未结取未决，二选一不相加）。
> - **生产发展三角形/热力图金额口径**（`server/src/sql/claims-detail.ts:519-535`、`claims-heatmap.ts:405-412`，标记 B302）：在通用公式**外层再加一道过滤**——`liability_ratio > 0 AND case_type NOT IN ('零结','注销','拒赔')`，不满足则计 0。**生产口径比通用公式更严格。**

## 一、结论先行

1. **已结案 285,309 件中，27.0%（77,136 件）为零赔付**（已决金额 `settled_amount` = 0）。这一比例与 B244 立项观察（约 27%）一致。
2. **数据中确实存在"立案计提了准备金（`reserve_amount` > 0）、结案却零赔付"的子集 = 12,748 件，准备金合计 1,818.7 万元**。
3. **但这部分对"生产发展三角形/赔付率"的准备金释放效应≈0——因为现行生产 SQL 已主动过滤掉它们**。释放子集中 **12,745 件（1,818.2 万元，占 99.98%）的案件类型正是 `零结`/`注销`/`拒赔`**，而生产发展三角形（`claims-detail.ts:519-535`）与理赔热力图（`claims-heatmap.ts:405-412`）的金额分子**外层 `CASE` 已将这三类案件类型 + 无责案件计 0**。因此它们在生产口径下**从所有观察期（cutoff）起就是 0，根本不以"开口准备金"形态进入三角形，也就不存在"先计 reserve、后释放"的回撤**。
4. **所谓"1,818.7 万元准备金释放"仅在通用赔付公式（`SCHEMA.md` §4，无 case_type 过滤）下成立**；而生产口径比通用公式更严格，已剔除该效应。**这是 B244 最重要的修正：释放效应存在于"通用公式视角"，但现行生产赔付率/三角形不受影响。**
5. **释放子集主体是"零结"案件**：10,326 件、准备金 1,475.8 万元（占 81.2%），案均立案仅 1,429 元；"拒赔"仅 34 件但案均立案最高（5.48 万元）。
6. **零赔付率近年下行**：按保单年度从 2022 年 29.8% 降至 2025 年 21.3%（仅为已结案件口径的描述性观察，受保单成熟度与未结案结构影响，**不直接等同经营/结案质量改善**）。
7. **现行口径无需修正**：生产总赔付口径（已结取 `settled_amount`、未结取 `reserve_amount`）+ B302 case_type 过滤，已正确地把这些零赔付案件排除在赔付率分子之外。本分析为只读量化，**未改动 `server/src/sql/claims-detail.ts`**。这是又一个"疑似待修、实则现行已正确处理"的核实结论。

## 二、背景与口径

- **B244 立项问题**：有多少赔案立案时计提了准备金（`reserve_amount` > 0），但最终结案赔款为零（`settled_amount` = 0）？这部分对"发展三角形（赔付率随观察期成熟的曲线）"的"准备金释放"效应有多大？
- **关键字段**：`reserve_amount`（未决/立案估损金额）、`settled_amount`（已决/实际赔付）、`settlement_time`（结案时间，非空 ⇔ 已结案，与 `claim_status='已业务结案'` 一一对应）、`case_type`（案件类型：零结/注销/拒赔/普通赔付等）、`liability_ratio`（责任比例）。
- **`reserve_amount` 结案后不清零**（证据见 §3.3）：已结且 `reserve_amount > 0` 的 213,962 件，准备金合计 143,804.1 万元 vs 已决合计 136,141.0 万元——若结案清零则准备金合计应≈已决，实际更高，故保留立案估损值。
- **"准备金释放"机制（仅通用公式下成立）**：通用公式下，案件开口期间以 `reserve_amount` 计入"已发生赔款"、结案后改用 `settled_amount`；有准备金的案件零结，对该公式的贡献从 `reserve_amount` 跌到 0，形成负向发展（释放）。**但生产口径在此之上加了 case_type 过滤，使零结/注销/拒赔从一开始就计 0，无释放过程**（详见 §5）。

## 三、零赔付全景：27% 的精确拆解

对已结案 285,309 件按"是否零赔付 × 立案是否计提准备金"四象限拆分：

| 已结案件分类 | 件数 | 占已结案 | 立案金额合计 | 已决合计 |
|---|---:|---:|---:|---:|
| 有赔付、有准备金（正常赔付） | 201,214 | 70.5% | 141,985.4 万元 | 136,141.0 万元 |
| **零赔付、有准备金（"释放"子集，仅通用公式视角）** | **12,748** | **4.47%** | **1,818.7 万元** | **0** |
| 零赔付、无正准备金（快速零结/注销） | 64,388 | 22.57% | -9.1 万元（微小冲正） | 0 |
| 有赔付、无准备金（小额直赔） | 6,959 | 2.44% | -3.5 万元 | 570.4 万元 |
| **合计** | **285,309** | **100%** | — | — |

- **零赔付合计 = 12,748 + 64,388 = 77,136 件 = 27.0%**，与立项观察吻合。
- **"零赔付且有准备金"的 12,748 件（4.47%）才会在通用公式下产生释放**；占"曾计提准备金的已结案件"（201,214 + 12,748 = 213,962 件）的 5.96%（件数口径），其准备金占"曾计提准备金已结案"准备金总额（143,804.1 万元）的 **1.26%（金额口径）**。
- 64,388 件"零赔付且无正准备金"是报案后快速零结/注销、从未挂起估损的案件，准备金合计仅约 -9.1 万元（微小冲正），通用公式下也几乎无释放。
- **口径提醒（修订 v1 的措辞）**：若仅按"零赔付件数率"判断"释放规模"，会把真正有准备金的释放子集（4.47%）与无准备金零赔付（22.57%）**混为一谈、件数/比例高估约 5~6 倍**；但就**释放金额**而言，无准备金那批合计仅约 -9.1 万元，并不会把 1,818.7 万元放大 5 倍。故"虚高"只成立于件数/比例口径，不成立于金额口径。

### 3.3 `reserve_amount` 结案后不清零（证据）

| 已结且立案>0 的案件 | 件数 | 准备金合计 | 已决合计 |
|---|---:|---:|---:|
| 有赔付 + 零赔付（reserve>0） | 213,962 | 143,804.1 万元 | 136,141.0 万元 |

准备金合计（143,804.1 万）> 已决合计（136,141.0 万），差额即"零结释放子集 1,818.7 万 + 正常赔付案件的估损-实赔差"，证明结案后 `reserve_amount` 保留立案估损、未清零。

## 四、释放子集（12,748 件）特征

### 4.1 按案件类型（关键：99.98% 落在生产口径的过滤集内）

| 案件类型 | 件数 | 准备金 | 占释放子集 | 案均立案 | 生产 SQL 是否过滤 |
|---|---:|---:|---:|---:|:--|
| 零结 | 10,326 | 1,475.8 万元 | 81.2% | 1,429 元 | ✅ 过滤（计 0） |
| 注销 | 2,385 | 156.1 万元 | 8.6% | 654 元 | ✅ 过滤（计 0） |
| 拒赔 | 34 | 186.3 万元 | 10.2% | 54,781 元 | ✅ 过滤（计 0） |
| 未决（数据残留 3 件） | 3 | 0.6 万元 | 0.0% | 1,883 元 | ❌ 不过滤 |

- **12,745 件（1,818.2 万元，99.98%）的案件类型为零结/注销/拒赔**，被 `claims-detail.ts:525` / `claims-heatmap.ts:407` 的外层 `CASE` 计 0；仅 3 件"未决"（0.6 万元，疑似数据残留：已结案却标未决）未被该过滤覆盖。
- 故无论从件数还是金额，**释放子集几乎完全落在生产口径的过滤集内**——这是 §5 修正的实证基础。

### 4.2 结案时效

| 指标 | 值 |
|---|---|
| 平均结案时延（报案→结案） | 20.2 天 |
| 中位结案时延 | 2.0 天 |
| 30 天内结案占比 | 89.8%（11,453 / 12,748 件） |

释放子集"立而即释"：近九成在报案后 30 天内零结。**注意**：此为"报案→结案"自然时延，与生产发展三角形的"按保单 cohort 日历月窗口累计快照（`effective_cutoff`）"是不同的时间轴，不能直接推断"释放落在三角形第几个发展期"（见 §5 修订说明）。

## 五、对发展三角形"准备金释放"效应的量化（v2 重大修正）

**v1 结论（错误，已废弃）**：曾断言这 12,748 件在生产发展三角形早期发展期产生约 1.3% 的向下回撤（准备金释放）。

**v2 修正结论**：
- 生产发展三角形（`claims-detail.ts:519-535`）与理赔热力图（`claims-heatmap.ts:405-412`）的金额分子**外层 `CASE` 先排除 `liability_ratio = 0` 及 `case_type IN ('零结','注销','拒赔')`，再做 settled/reserve 二选一**。
- 释放子集 12,745/12,748 件（99.98%）正属这三类案件类型 → 在生产口径下**从所有 cutoff 起即计 0**，**从不以"开口准备金"形态进入三角形**，**因此对生产赔付率/三角形不产生任何"先计后释放"的回撤**。
- 所谓 1,818.7 万元释放，**仅在 `SCHEMA.md` §4 通用二选一公式（无 case_type 过滤）下成立**。生产口径比通用公式更严格，已主动剔除该效应。
- **时间轴澄清**：即便在通用公式下，"释放集中在最早发展期"也需谨慎——生产三角形按日历月窗口累计快照，中位 2 天即零结的案件可能在任一月末快照中从未以开口准备金形态出现。本报告**不再给出"应预期约 1.3% 向下回撤"的预测**。

**净结论**：B244 关心的"准备金释放对发展三角形的影响"，在**现行生产口径下≈0**（已被 case_type 过滤消除）；在通用公式视角下规模为 1,818.7 万元、占曾计提准备金已结案准备金的 1.26%（金额口径）。

## 附录 A：复现 SQL（需本地 claims_detail 数据；注意 `COALESCE(settled_amount,0)=0` 把 NULL 已决视作零赔付，为本分析口径假设而非生产口径）

```bash
cd 数据管理
P="warehouse/fact/claims_detail/claims_*.parquet"

# A1. 27% 全景四象限拆解
duckdb -c "WITH base AS (SELECT *, (settlement_time IS NOT NULL) AS is_closed FROM read_parquet('$P', union_by_name=true))
SELECT is_closed, (COALESCE(settled_amount,0)=0) AS 零已决, (COALESCE(reserve_amount,0)>0) AS 立案gt0,
  COUNT(*), ROUND(SUM(COALESCE(reserve_amount,0))/1e4,1) 立案万元, ROUND(SUM(COALESCE(settled_amount,0))/1e4,1) 已决万元
FROM base GROUP BY 1,2,3 ORDER BY 1 DESC,2,3;"

# A2. 释放子集 case_type + 是否被生产 SQL(B302)过滤 + 案均/时延（含 median 与 30 天占比）
duckdb -c "WITH r AS (SELECT * FROM read_parquet('$P', union_by_name=true)
  WHERE settlement_time IS NOT NULL AND COALESCE(settled_amount,0)=0 AND COALESCE(reserve_amount,0)>0)
SELECT case_type, (case_type IN ('零结','注销','拒赔')) AS 生产过滤, COUNT(*), ROUND(SUM(reserve_amount)/1e4,1) 准备金万元,
  ROUND(AVG(reserve_amount),0) 案均立案,
  ROUND(AVG(DATEDIFF('day',CAST(report_time AS DATE),CAST(settlement_time AS DATE))),1) 平均时延,
  ROUND(MEDIAN(DATEDIFF('day',CAST(report_time AS DATE),CAST(settlement_time AS DATE))),0) 中位时延,
  ROUND(100.0*COUNT(*) FILTER(WHERE DATEDIFF('day',CAST(report_time AS DATE),CAST(settlement_time AS DATE))<=30)/COUNT(*),1) 内30天占比
FROM r GROUP BY 1,2 ORDER BY 3 DESC;"

# A3. 全部已决赔款分母（正常赔付案件）+ 结案后 reserve 不清零证据
duckdb -c "SELECT ROUND(SUM(CASE WHEN settlement_time IS NOT NULL THEN COALESCE(settled_amount,0) END)/1e4,1) 已结已决万元,
  ROUND(SUM(CASE WHEN settlement_time IS NOT NULL AND COALESCE(reserve_amount,0)>0 THEN reserve_amount END)/1e4,1) 已结有准备金的准备金万元,
  COUNT(*) FILTER(WHERE settlement_time IS NOT NULL AND COALESCE(reserve_amount,0)>0) 已结有准备金件
FROM read_parquet('$P', union_by_name=true);"

# A4. 按保单年度时序
duckdb -c "WITH base AS (SELECT *, (settlement_time IS NOT NULL) AS is_closed FROM read_parquet('$P', union_by_name=true))
SELECT insurance_year, COUNT(*) FILTER(WHERE is_closed) 已结, COUNT(*) FILTER(WHERE is_closed AND COALESCE(settled_amount,0)=0) 零赔付,
  ROUND(100.0*COUNT(*) FILTER(WHERE is_closed AND COALESCE(settled_amount,0)=0)/NULLIF(COUNT(*) FILTER(WHERE is_closed),0),1) 零赔付率
FROM base GROUP BY 1 ORDER BY 1;"

# A5. 待结案在险池
duckdb -c "SELECT COUNT(*), ROUND(SUM(reserve_amount)/1e4,1) 在险万元 FROM read_parquet('$P', union_by_name=true)
  WHERE settlement_time IS NULL AND COALESCE(reserve_amount,0)>0;"
```
