# 发布链全链路环节地图与耗时分析（2026-07-11）

> 目的：把「从 myvps 取数 → ETL → 发布到生产 → 企微智能表」全流程的**每一个环节、每环节耗时、断点、问题**变成可持续记录、可回溯的数据资产——数据与感知是优化决策的前提。
> 数据口径：环节清单来自 `scripts/sync-and-reload.mjs`（发布编排唯一入口）实测代码；耗时来自 2026-07-11 三次真实发布的台账事件（`数据管理/ledger/etl-ledger.jsonl`）与 watcher 日志（`数据管理/logs/auto-release.log`）。标注【推断】的数字由相邻事件时间差推得，非直接打点。

## 三、感知缺口与本次补齐（PR #1065 同分支）

| 缺口 | 现状（2026-07-11 前） | 本次补齐 |
|------|---------------------|---------|
| 环节耗时 | 台账只记完成"时刻"，无 duration | `runCmd` 统一打点：每环节 `stage='pipeline'` 事件带 `duration_ms` |
| 失败留痕 | **只记成功**——当天 3 次失败台账零记录 | 失败/超时/spawn 错误全部入账（`status='failed'` + exit_code/note） |
| run 级终态 | 无"这次发布成没成、断在哪"的单条答案 | `stage='run'` start/end 事件对：总耗时、终态、断点 note |
| 触发方式 | 无法区分自动/AI/人工 | `trigger` 字段：watcher（launchd 注入）/ ai（Claude 会话）/ manual |
| watcher↔发布关联 | watcher 与 release 各自为政 | watcher 预生成 run_id 传给发布链，事件同 run_id 可串联 |
| 分析入口 | 只能人肉 grep jsonl | `bun run ledger:analyze`：逐 run 概览 + 环节耗时排行 + `--run` 单次时间线 |

## 四、简化与优化清单（按收益排序，均须等数据佐证后动手）

> 原则：先积累 2~4 周打点数据再动刀——本文件第二节的【推断】数字换成实测分布后，以下每项的真实收益才可判。

**全局项**

1. **断点续跑**：三次失败每次都从 Stage 0 全量重来（拉取+ETL 约 3.5 分钟纯浪费，且当天数据已最新）。台账现在能精确知道断在哪个环节，可加 `--resume-from <stage>`，重试只跑失败环节及其后。当天若有此能力，第 2 次重试成本从"全链"降为"仅 sync"。
2. **确定性失败不重试**：watcher 对 exit=1 一律重试，但重叠闸/漂移闸是确定性失败，重试必然同败、白烧重试上限。发布链可在确定性闸失败时输出特征码（如 exit 2），watcher 识别后直接告警等人工，保留重试预算给网络抖动类瞬时失败。
3. **watcher 触发延迟**：tick 间隔 15 分钟且相位随机，上游就绪到触发平均多等 ~7.5 分钟。窗口开始后前 30 分钟可加密到 5 分钟一探（探测本身仅 1~3 秒 ssh 只读，成本可忽略）。

**单点项**

4. **ETL 按源变更跳过**：brand（37 万行）/repair 等低频维表每天全量重转，源 xlsx 字节未变也跑。pull 阶段已有 manifest 字节对比，可把"源未变 → 跳过该域 ETL"打通（04 厂牌本就"很少增量"，用户已拍板过其可选表地位）。
5. **同步清单 sha256 全量重算**：`writeSyncManifest` 每次对全部文件（含 73MB/60MB 大分片）算 sha256，多数文件当天未变。可按 size+mtime 命中缓存跳过重算。
6. **governance 发布路径瘦身**：发布链跑全量 54 项，其中大量纯代码治理项（体积棘轮/文档一致性等）与数据发布无关。可拆 `governance --release-relevant` 子集。**需先用打点数据确认 E8 实际耗时占比再决定**——若只占十几秒则不值得拆。
7. **30 秒稳定性硬等待**：E14 固定 sleep 30s。若历史数据表明启动期崩溃极罕见，可缩短或改为 PM2 事件驱动。

**候选退役项（需数据佐证）**

8. **E11 post 漂移校验与 E10 的关系**：post 校验对账的是"sync 刚写完的清单 vs 本地"——正常情况下 sync 成功则必一致，它实际防的是"sync 部分失败但退出码 0"的静默半份。保留有价值，但其误报已两次拦停发布（07-08 dim、07-11 fact）。本次对称化修复后若长期零发现，可考虑降级为告警不阻断。
9. **E6 报告生成在关键路径**：period-trend 报告失败不阻断但占关键路径时长。可移到 E14 健康检查通过后异步跑（报告晚几分钟无业务影响）。

## 五、使用方式（沉淀为日常习惯）

```bash
bun run ledger:analyze                 # 最近 14 天逐 run 概览 + 环节耗时排行
bun run ledger:analyze -- --days 30    # 放宽窗口
bun run ledger:analyze -- --run <id>   # 单次 run 完整时间线（含事件间隔）
```

每次发布（watcher / AI / 人工）自动留痕，无需任何手工动作。积累 2~4 周后回到本文件第四节，用实测分布替换【推断】，逐项决策。
