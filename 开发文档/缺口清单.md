# 项目开发缺口清单 (AI工作记录表)

## 📖 文档价值与协作规范

### 核心价值
- **AI工作记录**: 记录AI在实际工作中发现的缺口
- **状态追踪**: 避免重复发现，追踪解决进度  
- **知识沉淀**: 累积项目隐性信息和边界案例

### 协作规范
- **AI发现**: 规划/开发中遇到信息缺失时登记
- **用户提供**: 根据缺口描述提供所需信息
- **AI验证**: 验证信息完整性和正确性
- **状态更新**: 验证通过后标记为已完成

## 📋 规划阶段检查清单

**规划任何功能前，必须对照以下类型检查信息是否完备**：

### 业务规则类
- [ ] 业务规则文档是否存在？（如保费目标、考核标准）
- [ ] 计算口径是否明确？（如续保率定义、时间范围）
- [ ] 业务流程是否清晰？（如数据审批流程、异常处理）

### 数据格式类
- [ ] 数据格式规范是否明确？（如Excel列名、字段类型）
- [ ] 必需字段是否全部列出？（如policy_no, premium等）
- [ ] 示例数据是否提供？（如测试用Parquet文件）

### 技术实现类
- [ ] 第三方API文档是否可访问？（如地图API、认证接口）
- [ ] 数据库Schema是否明确？（如表结构、索引规则）
- [ ] 性能要求是否量化？（如响应时间、并发量）

### 交付标准类
- [ ] 验收标准是否明确？（如KPI数值精度、图表样式）
- [ ] 测试数据是否提供？（如边界案例、异常数据）
- [ ] 部署环境是否说明？（如浏览器版本、数据库版本）

**⚠️ 发现任何缺口，立即在下方"缺口记录"章节登记，标记为DISCOVERED！**

## 🔄 缺口状态管理

### 状态定义
- **🔴 DISCOVERED**: AI发现的缺口，待用户响应

### 🟡 B003-2026-06-22 - 报告生产方未 emit 机构归属 sidecar（B328 phase-2 GATED 项）
**发现时间**: 2026-06-22
**发现场景**: 实现 B328 phase-2（org_user 读本机构报告）时核实前置依赖
**缺口描述**:
- handler 侧（`server/src/routes/reports.ts`）已实现 sidecar 归属解析 + org 行级安全（org_user 本机构 200 / 跨机构 403 / branch_admin 不变），并以 36 单测 + live HTTP 矩阵验证
- **但生产方未补 sidecar**：`数据管理/integrations/wecom_bot/push_html.py` 单文件命名为 `<YYYYMMDD>-<slug>-<8hex>.html`，不写 `.meta.json`；diagnose-* skills 也不 emit `{reportId}/{snapshot}/.report-meta.json`
- 故生产真实报告无归属 → `resolveReportOwner` 返回 null → org_user 对真实报告仍 fail-closed 403（**端到端 org_user 200 GATED**）

**约定（生产方需 emit）**:
- 单文件 `foo.html` → 旁路 `foo.html.meta.json`，内容 `{ "ownerOrg": "<org_level_3>", "ownerBranch": "<branch_code，如 SC>" }`
- 多文件 `{reportId}/{snapshot}/` → `{reportId}/{snapshot}/.report-meta.json`，同 schema
- ownerOrg 必须等于 `org_user.organization`（org_level_3 中文名）；ownerBranch 须匹配 `^[A-Z]{2}$`

**影响范围**:
- org_user 在生产仍无法读到本机构报告（功能就绪但未启用）
- branch_admin 不受影响（全放行）

**所需动作（生产方侧，本 loop 范围外）**:
- [ ] `push_html.py` 增 `--org`/`--branch` 参数，stage 时写同名 `.meta.json`
- [ ] diagnose-org-weekly 等机构专属报告 skill 在产物目录 emit `.report-meta.json`
- [ ] 补齐后用真实 org_user 凭据做生产端到端验收（200/403 矩阵）

**状态**: 🟡 PARTIAL（handler 就绪，生产方 emit 待补）
**最后更新**: 2026-06-22

### 🔴 B002-2026-06-08 - 报价未成交无法归因到具体原因
**发现时间**: 2026-06-08
**发现场景**: 续保诊断/精炼/PPT 三步链中，把"续保问题=两部分（未报价＋报价未成交）"的应知应会下沉到 refine 时
**缺口描述**:
- 续保缺口可由 Parquet 精确拆出两段：① 未报价（应续−已报价）② 报价未成交（已报价−已续保）
- 但 **②"报价未成交"无法继续归因到具体原因**（理赔不满意 / 服务不满意 / 价格高于同业 / 跟进不到位 / 公司主动剔除等）——当前 Parquet 无这些信号字段
- 现行处置：refine 方法论已将这些原因列为"待归因候选"、PPT 原样渲染"待归因"，**禁止武断断言单一原因**（见 `.claude/commands/diagnose-report-refine.md` 续保领域口径层）

**影响范围**:
- 续保诊断只能定位缺口"落在成交端"，无法定位"为何没成交"，行动指向受限
- 报价未成交段的根因分析需人工 `--context` 注入或补数据源

**所需信息**:
- [ ] 理赔满意度 / NPS 数据（按车架号或客户可关联）
- [ ] 同业报价 / 价格竞争力对照
- [ ] 跟进记录（电联次数 / 触达时点）
- [ ] 核保主动剔除标记（公司端拒保/限制承保）

**状态**: 🔴 DISCOVERED
**最后更新**: 2026-06-08
