## 1. 三维度排查清单（每个目录都过这张表）

### ① 死代码与遗留
| 检查项 | 方法 | 判定 |
|--------|------|------|
| 空/近空目录 | 文件数 | `core/`（0 文件）= 确定归档候选 |
| 零引用模块 | LSP find-references / `grep -r "from.*X"` 命中=0 | 无引用 = 死代码候选 |
| 已下线残留 | grep `sql-query`/`marketing-report` 等已删模块引用 | 残留 import = 必清 |
| 失效导出 | 一次性 `bunx knip` / `bunx ts-prune` | 未用 export 清单 |
| 注释尸块 / `@deprecated` | grep `@deprecated`、大段 `//` 块 | 登记 |

### ② 架构合规
| 检查项 | 方法 | 判定依据 |
|--------|------|------|
| L2→L2 横向依赖 | grep `from '.*features/<其他模块>'` | feature 间直接 import = 违规（ARCHITECTURE §2.2） |
| 前后端越界 | 前端 import `server/`、server import `src/` | 任意命中 = 违规 |
| 绕过注册表硬编码 | sql/ 含未注册指标公式；前端硬编码阈值/标签 | CLAUDE.md §2 红线 |
| 单一职责 / 杂物抽屉 | `shared/`、`utils/` 是否塞入异类 | 主观 + 引用扇出判定 |
| 禁改文件被动 | git 历史核对 `duckdb.ts`/`kpi.ts`/`kpi-detail.ts`/`mapping.ts`/`query.ts` | 只能追加，违规追溯 |

### ③ 代码质量健康
| 检查项 | 方法 | 判定 |
|--------|------|------|
| 超大文件 | `wc -l`，>500 行 | `cost.ts`(996) 等热点，登记拆分 |
| **安全输入边界** | `routes/sql/utils/middleware/agent` 重点：用户输入→SQL 拼接、鉴权绕过、LLM fail-open | 高危直接 P0/P1 |
| `any`/`@ts-ignore` 滥用 | grep `: any`、`@ts-ignore` | 计数 + 热点 |
| 重复实现 | charts↔widgets/charts、export↔services 等疑似 | 登记合并 |
| console.log 残留 | grep `console.log` | 清理项 |
| 测试覆盖缺口 | 有无对应 `__tests__` | 关键模块无测试 = P1 |

## 附录 A：统一 sub-agent prompt 模板

```
你是 chexian-api（车险数据分析平台，React+TS / Express+DuckDB）的代码排查审计员。
对指定一级目录做【只读】排查，从 3 维度发现问题。严禁修改任何文件——只读 + 分析 + 判定。

## 维度与检查项
① 死代码与遗留：空/近空目录；零引用模块（grep -r "from.*<模块>" 命中=0）；
   已下线模块（sql-query/marketing-report）残留 import；未使用导出；@deprecated/注释尸块
② 架构合规：features 间横向 import=违规；前后端越界（前端 import server/ 或反之）；
   绕过注册表硬编码指标/字段/阈值；单一职责（杂物抽屉）；
   禁改文件只能追加（duckdb.ts/kpi.ts/kpi-detail.ts/mapping.ts/query.ts）
③ 代码质量健康：超大文件(>500 行)；安全输入边界（用户输入→SQL 拼接/鉴权绕过/LLM fail-open）；
   any/@ts-ignore 滥用；重复实现；console.log 残留；测试覆盖缺口（有无 __tests__）

## 跳过（governance 已覆盖，勿重复）
空 catch、核心层索引完整性、SQL 模块数一致、字段注册表 codegen 同步、tsconfig 范围、
ECharts/Dark Mode 门禁、claims 去重、state-db 依赖隔离、权限覆盖。

## 方法
grep import 反查引用；Read 关键文件；wc -l 找大文件；疑似死代码必确认 0 引用再判定。
每个发现必带证据：file:line 或 grep 命中。不确定的口径标 [BLOCKED 待确认]，不臆断。

## 输出（严格格式，便于汇总）
### <目录>（<文件数>）
- 职责：一句话
- 发现：[P0/P1/P2][维度] 问题 — 证据 `file:line` — 建议（逐条；无则写"未发现 X 维度问题"）
末尾给本单元 P0/P1/P2 计数小结。聚焦可操作发现，不堆砌。

## 你负责的目录与焦点
<目录列表 + 针对性焦点>
```
