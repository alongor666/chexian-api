### 2.2 数据特点
- **多层结构**：机构 → 团队 → 业务员
- **分产品计划**：车险、财险、人险分别设定计划

- **缺失值**：约13.3%的业务员缺少团队信息

## 3. 技术集成

### 3.1 DuckDB 集成

**新增方法**：`src/shared/duckdb/client.ts`

```typescript
// 1. 加载业务员计划数据
await duckdbClient.loadSalesmanPlan(file);

// 2. 查询业务员计划信息
const planInfo = await duckdbClient.getSalesmanPlanInfo('200048468肖照耀', 2026);
```

**创建视图**：`SalesmanPlanFact`

```sql
CREATE OR REPLACE VIEW SalesmanPlanFact AS
SELECT
  salesman_name,
  salesman_id,
  team_name,
  org_name,
  CAST(entry_date AS DATE) as entry_date,
  plan_year,
  plan_vehicle,
  plan_property,
  plan_life,
  plan_total,
  actual_vehicle,
  actual_property,
  actual_life,
  actual_total,
  rate_vehicle,
  rate_property,
  rate_life,
  rate_total,
  months_in_service
FROM raw_salesman_plan
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY salesman_name, plan_year
  ORDER BY plan_year DESC
) = 1
```

### 3.2 SQL 查询示例

#### 场景1：补全业务员团队信息
```sql
SELECT
  p.policy_no,
  p.salesman_name,
  p.premium,
  COALESCE(p.org_level_3, s.org_name) as org_level_3,
  s.team_name
FROM PolicyFact p
LEFT JOIN SalesmanPlanFact s
  ON p.salesman_name = s.salesman_name
  AND s.plan_year = 2026
WHERE p.policy_date >= '2026-01-01'
```

#### 场景2：计算计划达成率
```sql
SELECT
  s.salesman_name,
  s.team_name,
  s.org_name,
  s.plan_total,
  SUM(p.premium) as actual_total,
  SUM(p.premium) / s.plan_total as achievement_rate
FROM SalesmanPlanFact s
LEFT JOIN PolicyFact p
  ON s.salesman_name = p.salesman_name
  AND p.policy_date >= '2026-01-01'
  AND p.policy_date < '2027-01-01'
WHERE s.plan_year = 2026
GROUP BY s.salesman_name, s.team_name, s.org_name, s.plan_total
ORDER BY achievement_rate DESC
```

#### 场景3：团队保费计划达成对比
```sql
SELECT
  s.team_name,
  s.org_name,
  SUM(s.plan_total) as team_plan_total,
  SUM(p.premium) as team_actual_total,
  SUM(p.premium) / SUM(s.plan_total) as team_achievement_rate
FROM SalesmanPlanFact s
LEFT JOIN PolicyFact p
  ON s.salesman_name = p.salesman_name
  AND YEAR(p.policy_date) = 2026
WHERE s.plan_year = 2026
GROUP BY s.team_name, s.org_name
ORDER BY team_achievement_rate DESC
```

## 4. 应用场景

### 4.1 填补机构信息缺口
**问题**：签单数据中的 `org_level_3` 字段可能缺失
**解决**：通过业务员姓名关联 `SalesmanPlanFact.org_name`

### 4.2 支持团队维度分析
**问题**：无法按团队分组统计保费
**解决**：新增 `team_name` 字段，支持团队维度的KPI计算

### 4.3 计划达成率分析
**问题**：无法评估业务员的保费计划达成情况
**解决**：
- 2025年：直接使用 `rate_total` 字段
- 2026年：实时计算 `SUM(premium) / plan_total`

### 4.4 业务员画像
**入司时间分析**：
```sql
SELECT
  salesman_name,
  entry_date,
  TIMESTAMPDIFF(YEAR, entry_date, CURRENT_DATE) as years_in_service,
  plan_total
FROM SalesmanPlanFact
WHERE plan_year = 2026
ORDER BY years_in_service DESC
```

**在岗时长分析**（2025年数据）：
```sql
SELECT
  salesman_name,
  months_in_service,
  actual_total / months_in_service as monthly_avg
FROM SalesmanPlanFact
WHERE plan_year = 2025
ORDER BY monthly_avg DESC
```

## 6. 注意事项

### 6.1 数据质量
- ✅ 所有业务员都有机构信息
- ⚠️ 约13.3%的业务员缺少团队信息
- ✅ 业务员姓名格式统一（"编号+姓名"）

### 6.2 数据时效性
- **2025年数据**：包含实际保费和达成率（历史数据）
- **2026年数据**：仅包含保费计划（预测数据）
- **建议**：每年更新一次计划数据

### 6.3 性能优化
- SalesmanPlanFact 视图已做去重处理（按 salesman_name + plan_year）
- 使用 `LEFT JOIN` 避免过滤掉没有计划的业务员
- 建议：为 salesman_name 创建索引（DuckDB会自动优化）

### 6.4 安全性
- Parquet 文件应放在数据管理目录，不提交到 Git
- 敏感数据（业务员联系方式）已脱敏
- 生产环境应考虑访问权限控制
