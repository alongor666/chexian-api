## 1. 背景与根因

执行山西诊断时，`/chexian-data-kpi` 命令的执行协议**靠 6 处人工纠偏**才得到正确结果：省份隔离、净额口径、字段名、枚举值、时间窗口、业务员粒度——命令一个都没给对。其中两处足以致命：

- **省份混查（静默错误）**：示例 SQL `FROM '…/current/*.parquet'` glob 全量，传"山西"会掺入四川数据，**不报错**。
- **示例 SQL 必然跑挂**：`WHERE endorsement_type IS NULL` 实证 `Binder Error`（该字段 fields.json 登记为源 Excel 列、但 ETL 未落 Parquet，只有 `endorsement_no`），且"剔除批改"口径与文件自身字段表的"净额+去重"自相矛盾。

**根因（一句话）**：技能自己**内联**了一套口径/字段/枚举/阈值 → 成了游离于 rules 体系之外的"影子事实源" → 必然与真 SSOT 漂移、且无闸拦截。命令第 79 行口称"唯一事实源 fields.json"，第 90 行却内联了一个注册表有、数据没有的字段，制造"已对齐"的假象。

**治理盲区**：CLAUDE.md §2 注册表 RED LINE（禁前端硬编码、禁 SQL 生成器硬编码）**只覆盖 `server/` 代码，没覆盖 `.claude/commands/**` 与 `.claude/skills/**`**——这就是技能能游离在外的根。

---

## 2. 双向审查结论（麦肯锡方法论 × 项目诊断家族约定）

审查列出 11 个输入缺陷。归位后的关键统计：

| 类别 | 数量 | 含义 |
|---|---|---|
| 技能"没挂靠"已有 SSOT | 9 / 11 | rules 体系早有权威规则（字段/时间/验证/枚举/粒度/赔付率/产物/worktree/边界），命令只是没引用 |
| SSOT "自身缺失" | 2 / 11 | ① 省份隔离无规则 ② "件均保费"未进注册表 |

**结论**：落实"所有技能挂靠 rules 体系"要两步——**先补缺的 SSOT（省份隔离、件均保费），再让技能全部挂靠**，并把原则固化为元规则 + governance 闸。

**用户修正（2026-06-27）**：满期赔付率（注册表 `earned_claim_ratio`）**是 KPI，保留并补全实现**（机构对比补赔款 JOIN + 满期口径），不删除、不外移到 cost-analysis。

## 6. Loop V2 推进协议（遵 `.claude/rules/loop-orchestration.md`）

每任务一条 evidence-loop 流水线：

1. **认领先于实现**：`bun scripts/backlog.mjs status <uid> IN_PROGRESS --actor <branch>` 并 push（event-log 认领锁，防跨会话重复劳动）。
2. **隔离 worktree**：新分支 off `origin/main`。
3. **🛡 对抗闸-1（计划）**：codex CLI（`codex exec --sandbox read-only - < prompt`，**不经 skill**，prompt 自包含喂 diff/片段）审设计 → 修 P0/P1。
4. **实现**（TDD 优先）。
5. **确定性闸**：`bun run verify:full` + `bun run governance`（correctness oracle，与 codex 闸正交）。
6. **🛡 对抗闸-2（完成 diff）**：codex CLI 单源审 diff → P0/P1 全修 + 复审通过。
7. **收尾**：`/chexian-commit-push-pr` → PR（**非 draft、非部署链 → 可 auto-merge**）+ backlog `status DONE --evidence` + 质量账本一行 + 复盘。
8. **合并探测** → `bun run loop:dispatch` 重算前沿 → 下一波。
