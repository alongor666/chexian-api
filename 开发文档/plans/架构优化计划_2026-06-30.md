## 优化方向一览

### P0 · 稳定性与可靠性（影响生产可用性）

#### 1. DuckDB 查询超时与熔断机制
- **现状**：慢查询仅日志告警（阈值 3000ms），无超时中断；综合分析 dashboard 可并发 7+ 查询
- **风险**：单个慢查询可阻塞连接池（10 连接），3-4 个用户同时打开 dashboard 即可能触发饱和
- **方案**：
  - 为 DuckDB 查询添加 `PRAGMA query_timeout=<ms>` 或 Node 层 AbortController 超时
  - 实现简易熔断：连续 N 次超时后短暂拒绝同类查询，返回降级响应
  - 连接池扩至 15-20 + 监控队列深度告警
- **涉及文件**：`server/src/services/duckdb.ts`、`duckdb-infra.ts`
- **验证**：benchmark-key-routes.mjs 压测 p99 < 3s

#### 2. 4 域 RLS 权限注入补全
- **现状**：customer-flow / quote-conversion / claims-detail / repair 的 SQL 生成器未消费 `req.permissionFilter`，临时用 `requireBranchAdmin()` 阻断非管理员访问
- **风险**：权限模型不完整，无法开放给普通用户
- **方案**：逐域注入 `permissionFilter` 到 WHERE 子句，移除 admin-only 降级
- **涉及文件**：`server/src/sql/claims-detail.ts`、`quote-conversion.ts`、`repair.ts`、`customer-flow.ts`
- **验证**：`bun run test --run` + `curl` 验证不同角色返回数据范围

### P1 · 性能优化（影响用户体验）

#### 3. 图表组件渲染优化
- **现状**：33+ 图表组件无 `React.memo`，ECharts 实例每次挂载重建；dashboard 单页 20+ 图表
- **影响**：筛选器变更触发全页面重渲染，浪费 CPU
- **方案**：
  - 高频图表组件（LineChart/BarChart/PieChart/RoseChart）包裹 `React.memo`
  - ECharts 实例复用：通过 ref 保持实例，数据变更时 `setOption` 而非重建
  - 对 dashboard 页面的非可视区域图表延迟渲染（IntersectionObserver）
- **涉及文件**：`src/widgets/charts/*.tsx`、各 feature panel 组件
- **验证**：React DevTools Profiler 对比渲染次数

#### 4. DuckDB 类型转换流式化
- **现状**：`convertBigIntToNumber()` 对全量结果递归遍历（Array → Object → 嵌套字段），10 万行结果耗时 50-200ms
- **方案**：
  - 大结果集（> 5 万行）改用流式转换，边转换边写 response
  - 小结果集保持现有逻辑（简单可靠）
- **涉及文件**：`server/src/services/duckdb-type-converter.ts`、`duckdb.ts`
- **验证**：导出 10 万行场景 p95 < 500ms

#### 5. 审计日志写入批量化
- **现状**：每请求独立 `fs.appendFile`，200 req/min 下可能产生 fs buffer 竞争
- **方案**：内存缓冲 100 条 / 1 秒刷盘，减少 I/O 系统调用
- **涉及文件**：`server/src/middleware/audit.ts`
- **验证**：soak 压测下日志完整性 + 无丢失

### P2 · 代码质量（影响可维护性）

#### 6. SQL 过滤器逻辑统一抽取
- **现状**：`buildWhere()` / `buildPolicyWhere()` 在 6+ SQL 生成器中重复实现，30+ 行逻辑近乎相同
- **风险**：筛选语义修改需同步 6 个文件，易遗漏导致数据不一致
- **方案**：
  - 新建 `server/src/sql/shared/policy-filters.ts`，统一 policy 层通用过滤条件构建
  - 各生成器保留自身特有条件，通用部分委托共享函数
  - 现有 `sql-builder.ts`（382 行）可作为底层，policy-filters 作为业务层
- **涉及文件**：
  - 新建：`server/src/sql/shared/policy-filters.ts`
  - 重构：`claims-detail.ts`(624 行)、`claims-heatmap.ts`(574 行)、`repair.ts`(499 行)、`quote-conversion.ts`、`cube/servability.ts`、`perspective-adapter.ts`
- **验证**：现有 SQL 单测全绿 + golden-baseline 零差异

#### 7. 大文件拆分（后端服务层）
- **现状**：`duckdb-domain-loaders.ts`(1,077 行) 包含 13 个 load* 函数；`cache-warmer.ts`(852 行)
- **方案**：
  - `duckdb-domain-loaders.ts` → 拆为 `loaders/` 子目录（dimension-loader / fact-loader / aggregate-loader / view-loader）
  - `cache-warmer.ts` → 提取配置（热路由列表）到独立文件，逻辑本体保留
- **验证**：`bun run verify:full` 全绿

#### 8. 大组件拆分（前端展示层）
- **现状**：6 个前端组件超 700 行（GeoRiskPanel 951 行、CrossSellAnalysisPanel 903 行等），混合数据加载/转换/渲染/业务逻辑
- **方案**：
  - 提取自定义 Hook（如 `useGeoInsights`、`useGeoTableData`）
  - 拆分子展示组件（GeoMapViewer / GeoKpiCards / GeoTableComparison）
  - 容器组件仅做组合编排，控制在 200 行内
- **优先拆**：GeoRiskPanel（951 行）→ 影响面小、结构清晰、可作模板
- **验证**：功能回归 + `bun run build` 零 TS 报错

### P3 · 架构增强（面向未来扩展）

#### 9. 图表配置工厂统一
- **现状**：15+ 图表组件内联 ECharts options，tooltip/grid/axis 配置重复；部分硬编码色值违反 DC-003 设计规范
- **方案**：
  - 新建 `src/shared/config/chartOptions/` 工厂函数（createTooltip / createGrid / createAxis）
  - 统一 SERIES_COLORS 调色板引用，消除硬编码 hex
  - tooltip 统一用 `escapeHtml()` 防 XSS
- **涉及文件**：`src/widgets/charts/*.tsx`、`src/features/*/components/*Chart.tsx`
- **验证**：视觉回归 + governance 图表色值检查

#### 10. 前端测试覆盖补强
- **现状**：前端测试覆盖率 ~18%（37 个测试文件 / 200+ 组件）；最大 6 个组件（均 700+ 行）零测试
- **方案**：
  - 按业务重要性排序，优先覆盖 dashboard bundle / cost analysis / claims detail
  - Hook 单测（useGrowthAnalysis 474 行无测试）+ 组件快照测试
  - 目标：核心 feature 覆盖率 > 50%
- **验证**：`bun run test --run --coverage`

#### 11. 性能监控体系
- **现状**：后端有 benchmark 脚本，前端无 LCP/INP/CLS 目标，无 bundle 预算
- **方案**：
  - 前端接入 `web-vitals` 库，上报核心指标
  - vite.config 设置 bundle 预算（单 chunk < 500KB）
  - CI 集成 Lighthouse 基线检查
- **验证**：CI 报告可观测

#### 12. 路由处理器模板化
- **现状**：18+ query 路由处理器遵循相同流程（Zod 校验 → 解析过滤器 → SQL 生成 → 执行 → 响应），但各自手写
- **方案**：
  - 提取高阶路由构建器 `createQueryRoute(sqlGenerator, schema, cacheKey)`
  - 保留自定义扩展点（后处理、额外校验）
  - 新路由只需声明式配置
- **验证**：现有 route 单测全绿 + 新路由可 5 分钟内接入

## 不建议做的事

| 方向 | 原因 |
|------|------|
| 引入 Redux/Zustand 替换 Context | 当前 Context + React Query 分层合理，6 Context 规模可控，迁移收益不抵成本 |
| 引入 DI 容器 | 服务层单例模式清晰，测试通过 mock 即可，DI 增加复杂度 |
| 指标注册表改数据库驱动 | 当前文件级注册表有类型安全 + 版本控制 + governance 校验，数据库丢失这些优势 |
| 前后端共享类型包 | 当前 REST API 边界清晰，镜像类型可控，共享包增加构建依赖 |
| 用 sparse-checkout 做主目录只读 | 已有事故记录（2026-06-03），1,416 文件被裁，禁止使用 |
