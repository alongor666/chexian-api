# 架构优化计划 — 2026-06-16

> **范围**：基于全量代码库审计（后端 252 文件 / 69K LOC，前端 411 文件 / 74K LOC，CI 8 条流水线）的架构级优化方向。
> **原则**：每项优化必须有可度量终止条件（遵守证据闭环协议）；优先级按"用户可感知收益 × 实施风险⁻¹"排序。

---

## 一、总览：六大优化方向

| # | 方向 | 核心问题 | 预期收益 | 优先级 |
|---|------|---------|---------|--------|
| 1 | **DuckDB 查询层优化** | SQL 编译无复用、CTE 级联深、维度笛卡尔膨胀 | 查询 p95 降 30-50% | P0 |
| 2 | **前端渲染性能** | 巨型组件(1401行)、memo 覆盖率 9%、Context 级联重渲染 | 首屏 TTI 降 40%、交互延迟降 60% | P0 |
| 3 | **SQL 生成器去重与抽象** | 维度映射 4 处重复、日期窗口逻辑 4 处重复、WHERE 构建不统一 | 维护成本降 50%、口径一致性 100% | P1 |
| 4 | **构建与包体积** | 无组件级代码分割、chunk 阈值偏大(1000KB)、ECharts 未 tree-shake | 首包体积降 30% | P1 |
| 5 | **后端架构解耦** | Route→SQL 直接耦合、God-File(data.ts 874行)、cache-warmer 848行 | 可测试性提升、部署风险降低 | P2 |
| 6 | **测试与 CI 加固** | 无覆盖率阈值、集成测试不在标准 CI、脚本层零测试 | 回归检出率提升 | P2 |

## 二、P0 — DuckDB 查询层优化

### 2.1 segmentCaseExpr 物化（B306-F01）

**现状**：`performance-analysis/shared.ts:123-155` 每行执行 8 层 `LIKE + CAST/TRIM` 字符串匹配。

**方案**：
1. ETL 阶段预计算 `segment_tag` 整型列写入 Parquet
2. 查询时 `WHERE segment_tag = N` 替代运行时 CASE WHEN
3. 字段注册表新增 `segment_tag`（`fields.json` → `generate.mjs`）

**度量**：`bench-universal-cube.mjs` 对比前后 p95，目标 CPU 降 20-40%。

### 2.2 YEAR 提取替换为 BETWEEN（B306-F02）

**现状**：`EXTRACT(YEAR FROM CAST(insurance_start_date AS DATE)) = N` 出现 10+ 次，破坏 DuckDB zonemap 索引。

**方案**：
```sql
-- 前
WHERE EXTRACT(YEAR FROM CAST(insurance_start_date AS DATE)) = 2026
-- 后
WHERE CAST(insurance_start_date AS DATE) BETWEEN '2026-01-01' AND '2026-12-31'
```

**度量**：单查询 scan 行数降 30-60%（`EXPLAIN ANALYZE` 对比）。

### 2.3 growth/yoy.ts 双 CTE 合并（B306-F03）

**现状**：`growth/yoy.ts:32-62` 两个独立 CTE 各做一次全量 PolicyFact 扫描。

**方案**：合并为单次扫描 + 条件聚合（`SUM(CASE WHEN year=... THEN ... END)`）。

**度量**：查询耗时降 50%（`benchmark-key-routes.mjs` 的 `/api/query/growth` 端点）。

### 2.4 SQL 预编译探索

**现状**：每次请求生成新 SQL 字符串 → DuckDB 重新解析。无 prepared statement 池。

**方案**：
1. 对高频路由（KPI、trend、dashboard bundle）的 SQL 模板参数化
2. DuckDB `connection.prepare(sql)` + `stmt.bind(params)` 复用
3. 连接池层面缓存 prepared statements（按 SQL hash）

**风险**：DuckDB Node API 的 prepare 支持需验证兼容性。先做 PoC 再推广。

**度量**：高频路由 p50 降 20%（编译开销占比约 15-25%）。

## 三、P0 — 前端渲染性能

### 3.1 巨型组件拆分

**现状**：5 个组件超 800 行，最大 `PerformanceAnalysisPanel.tsx` 1401 行。

| 组件 | 行数 | 拆分方案 |
|------|------|---------|
| PerformanceAnalysisPanel | 1401 | → Summary + Heatmap + Drilldown + TrendChart（各 200-350 行） |
| GeoRiskPanel | 951 | → GeoMap + RiskCluster + DrillAnalytics |
| CrossSellAnalysisPanel | 891 | → 按 Tab 拆分（OrgView + SalesmanView + TrendView） |
| NewEarnedPremiumTable | 891 | → TableCore + DualDateHeader + ExportAction |
| EnhancedKpiCard | 870 | → KpiValue + Sparkline + TrendBadge |

**度量**：React DevTools Profiler 对比拆分前后单次 filter 变更的渲染耗时。

### 3.2 React.memo 覆盖率提升

**现状**：144 个 feature 组件中仅 13 个（9%）使用 `React.memo()`。

**方案**：
1. 所有接收 `filters` prop 的组件包裹 `React.memo()`
2. 父组件的事件回调统一用 `useCallback`
3. 重点覆盖：dashboard 71 个文件、cost、claims-detail

**目标**：memo 覆盖率从 9% → 60%+。

### 3.3 Context 消费优化

**现状**：
- `useGlobalFilters()` 有 48 个消费者，任一 filter 字段变更全部重渲染
- `PermissionContext` + `DataContext` 在根级，auth 变更级联到所有路由

**方案**：
1. `useGlobalFilters()` 改为选择性订阅（`useFilterField('org')` 只监听 org 变更）
2. 评估 Zustand / Jotai 替换 FilterContext（原子化状态，按字段订阅）
3. 为 PermissionContext 添加 `useMemo` 缓存 value 对象

### 3.4 面板级 Suspense 边界

**现状**：仅路由级 `<Suspense>`，面板内 6 个并行 `useQuery` 全部 loading 时无骨架屏。

**方案**：
- 每个分析面板独立 `<Suspense fallback={<PanelSkeleton />}>`
- 配合 React Query 的 `suspense: true` 模式
- 下钻操作时局部 loading，不阻塞其他面板

## 四、P1 — SQL 生成器去重与抽象

### 4.1 统一维度字段映射

**现状**：维度→字段映射分散在 4 处（`cost/shared.ts`、`performance-analysis/shared.ts`、`salesman-ranking.ts`、`pivot.ts`），存在不一致风险。

**方案**：
1. 新建 `server/src/sql/shared/dimension-registry.ts` 作为唯一事实源
2. 所有 SQL 生成器从该注册表取维度定义
3. governance 新增检查：禁止 SQL 生成器中硬编码维度字段名

### 4.2 日期窗口逻辑统一

**现状**：`growth/{mom,yoy,ytd,custom}.ts` 各自实现日期窗口计算（custom.ts 单文件 344 行）。

**方案**：
1. 提取 `shared/date-window-builder.ts`：统一 `buildDateWindow(type, baseDate, params)` 接口
2. 四个变体调用统一接口 + 各自的聚合逻辑
3. 估计可减少 ~600 行重复代码

### 4.3 WHERE 构建统一

**现状**：`parseFiltersAndBuildWhere` vs `buildWhereFromFilterParams` 两套并存。

**方案**：收敛为单一 `buildWhereClause(filters, options)` 函数，旧 API 薄壳包装后逐步迁移。

### 4.4 硬编码指标公式清理（B305/B310）

**现状**：`earned_premium` 硬编码 9 处、`expense_ratio` 硬编码 5 处、`variable_cost_ratio` 硬编码 3 处，绕过指标注册表。

**方案**：
1. 逐个替换为 `getMetricSql('earned_premium')` 调用
2. governance 新增检查：扫描 SQL 生成器中的裸公式，拦截未注册指标
3. 分批实施，每批跑 `golden-baseline.mjs --compare` 确认零回归

## 五、P1 — 构建与包体积优化

### 5.1 组件级代码分割

**现状**：路由级 lazy loading 已覆盖 85%，但路由内所有面板/图表 eager 加载。

**方案**：
1. 重型分析面板（Cost、GeoRisk、ClaimsDetail）lazy import
2. ECharts 按需引入（仅 import 使用的 series type，不全量 re-export）
3. PDF 导出模块（ExportContext 513 行）从根级移到按需加载

**度量**：`bun run build` 后对比 `dist/assets/` 的 chunk 分布，目标首包降 30%。

### 5.2 chunk 策略优化

**现状**：`manualChunks` 分 5 组（vendor-react/echarts/data/export/ui），chunk 警告阈值 1000KB。

**方案**：
1. 阈值从 1000KB 降到 500KB，暴露过大 chunk
2. 新增 `vendor-table`（react-window + react-select 拆出）
3. 按 feature 分 chunk（dashboard、cost、claims-detail 各自独立）

### 5.3 Bundle 分析自动化

**现状**：有 rollup-plugin-visualizer 但无 CI 集成。

**方案**：
1. CI 流水线中生成 bundle size report
2. PR 评论自动对比 main 分支的 chunk 大小变化
3. 超过阈值的 PR 标记警告

## 六、P2 — 后端架构解耦

### 6.1 God-File 拆分

| 文件 | 行数 | 拆分方案 |
|------|------|---------|
| `routes/data.ts` | 874 | → `data/export.ts` + `data/import.ts` + `data/bulk-ops.ts` + `data/versioning.ts` |
| `services/cache-warmer.ts` | 848 | → `cache-warmer/tier1-hard-cache.ts` + `cache-warmer/org-iterator.ts` + `cache-warmer/task-batch.ts` |
| `config/query-routes-metadata.ts` | 832 | 评估从路由文件 AST 自动生成（消除手动同步） |
| `routes/auth.ts` | 532 | → `auth/jwt.ts` + `auth/pat.ts` + `auth/session.ts` |

### 6.2 Route→SQL 抽象层

**现状**：路由直接调用 `duckdbService.query(generateXxxQuery(config))`，SQL 输出格式变更级联到路由。

**方案**：
1. 引入 `QueryExecutor` 服务层：`executor.run(queryName, params) → TypedResult`
2. SQL 生成 + 执行 + 结果映射内聚在 executor 中
3. 路由只负责 HTTP 协议（参数校验、响应格式化）

**收益**：路由可独立测试（mock executor）、SQL 变更不影响路由代码。

### 6.3 指标注册表启动校验

**现状**：路由运行时调用 `getMetricSql('id')` 时才发现指标不存在。

**方案**：服务启动时遍历所有注册指标，用 DuckDB `EXPLAIN` 校验 SQL 表达式语法合法性。启动报错优于运行时 500。

## 七、P2 — 测试与 CI 加固

### 7.1 覆盖率阈值

**现状**：v8 覆盖率工具已配置但无最低阈值，CI 不因低覆盖率失败。

**方案**：
1. 后端覆盖率阈值 70%、前端 60%
2. `bun run test:coverage:check` 新增门禁任务
3. PR 评论展示覆盖率变化趋势

### 7.2 集成测试纳入 CI

**现状**：4 个集成测试文件（Parquet/DuckDB/SQLite）仅本地运行，CI 排除。

**方案**：
1. CI 矩阵增加 `test:integration` job（需安装 DuckDB 原生二进制）
2. 或在 deploy 流水线的 smoke test 阶段运行（已有 better-sqlite3 ABI 检查先例）

### 7.3 脚本层测试

**现状**：`scripts/` 37 个文件零测试覆盖（governance harness 只验权限不验正确性）。

**方案**：
- 优先覆盖：`sync-vps.mjs`（部署关键路径）、`cleanup-reports.mjs`（删除操作）、`backlog.mjs`（事件日志写入）
- 测试方式：快照测试（给定输入 → 校验输出结构），不需要完整 E2E

### 7.4 双重 rounding 修复（B277）

**现状**：SQL `ROUND(x, 2)` + 前端 `decimals=1` 导致精度丢失。

**方案**：SQL 层不 round，前端统一在展示层格式化。逐个路由替换 + golden-baseline 对比。

## 九、度量体系

每个优化项必须绑定以下至少一项度量：

| 类型 | 工具 | 阈值 |
|------|------|------|
| 查询性能 | `benchmark-key-routes.mjs` | p95 改善 ≥ 20% |
| 正确性 | `golden-baseline.mjs --compare` | 零差异 |
| 包体积 | `bun run build` + visualizer | 首包 ≤ 500KB gzip |
| 渲染性能 | React DevTools Profiler | filter 变更渲染 < 100ms |
| 覆盖率 | `bun run test:coverage` | 后端 ≥ 70%，前端 ≥ 60% |
| 治理 | `bun run governance` | 全部通过 |

## 十、风险与回退

| 风险 | 缓解措施 |
|------|---------|
| segmentCaseExpr 物化需 ETL 改动 | 影子模式先行（cube-shadow 模式验证） |
| React.memo 过度导致 stale props | 配合 useStableParams() + 充分测试 |
| SQL 抽象层引入间接层 | 先在 2 个低风险路由试点，再推广 |
| Zustand 迁移破坏现有 Context 消费者 | 渐进式：新组件用 Zustand，旧组件保留 Context 兼容层 |
| 包分割过细导致请求瀑布 | 配合 prefetch 策略 + HTTP/2 多路复用 |
