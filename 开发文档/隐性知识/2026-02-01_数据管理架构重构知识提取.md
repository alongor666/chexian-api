## 1. 架构设计决策

### 1.1 为什么选择 pipelines/warehouse/knowledge 三层结构？

**决策理由**:
- **分层清晰**: 参考 dbt + DataOps 最佳实践，将数据处理流程分为三个明确职责的层级
- **单一职责**: 每个目录只负责一件事，避免职责混乱
- **AI友好**: knowledge/ai/ 专门服务 NL2SQL，优化 AI 上下文加载
- **可追溯**: staging 保留中间数据，logs 记录处理过程，便于问题追踪

**三层职责**:

| 层级 | 职责 | 典型文件 |
|------|------|---------|
| **pipelines/** | 数据处理管道（ETL脚本） | transform.py, enrich.py |
| **warehouse/** | 数据仓库（结构化存储） | fact/policy/, dim/salesman_plan/ |
| **knowledge/** | 知识库（文档+规则） | ai/PARQUET_SCHEMA_KNOWLEDGE.md, rules/车险数据业务规则字典.md |

**数据流向**:
```
pipelines/    (加工)
    ↓
warehouse/    (存储)
    ↓
src/          (应用消费)
```

**关键洞察**:
> 重构前，数据文件和脚本混在一起，导致"找不到最新数据"和"不知道用哪个脚本处理"的问题。三层结构后，数据流向一目了然。

---

### 1.2 为什么采用星型模型（fact/dim）？

**决策理由**:
- **业务语义**: 符合数据仓库经典建模方法（Kimball 星型模型）
- **查询优化**: 事实表记录业务事件（保单），维度表提供上下文（业务员计划、组织映射）
- **扩展性**: 未来可轻松添加新维度表（如产品维度、地域维度）

**当前星型模型**:

```
fact/policy/                 (事实表 - 保单明细)
    └─ 车险保单综合明细表0127.parquet

dim/salesman_plan/           (维度表 - 业务员计划)
    └─ 业务员保费计划标准化数据.parquet

dim/业务员归属与规划/        (维度表 - 组织映射)
    └─ salesman_organization_mapping.json
```

**关键洞察**:
> 重构前，所有数据都平铺在 `保单明细/` 目录下。采用星型模型后，维度数据（如业务员计划）和事实数据（保单）分离，便于独立更新和维护。

---

### 1.3 目录命名的考虑因素

**命名原则**:
- **功能域**: 中文（数据管理/、开发文档/） - 便于业务人员理解
- **技术目录**: 英文小写（pipelines/, warehouse/, staging/) - 符合国际规范
- **子项目**: 中文动宾短语（原始数据加工/、保单明细/） - 描述性强

**对比**:

| 重构前 | 重构后 | 改进点 |
|--------|--------|--------|
| `保单明细/` | `warehouse/fact/policy/` | 明确数据类型（事实表） |
| `数据分析报告/` | `pipelines/` + `knowledge/` | 分离处理逻辑和知识文档 |
| `原始数据加工/scripts/` | `pipelines/` | 统一管道脚本位置 |

**关键洞察**:
> 中英文混用是为了平衡业务人员可读性和技术规范性。根目录用中文（数据管理/），技术目录用英文（pipelines/），避免路径中出现中文编码问题。

## 2. 技术约束

### 2.1 Python 脚本的位置和命名规范

**强制规则**:
- **位置**: 所有数据处理脚本必须在 `数据管理/pipelines/` 目录下
- **命名**: 动词_名词.py（如 `transform.py`, `enrich.py`）
- **入口**: 每个脚本支持命令行参数 (`-i`, `-o`)

**示例**:
```bash
# ✅ 正确
python3 pipelines/transform.py -i input.xlsx -o warehouse/fact/policy/output.parquet

# ❌ 错误：脚本放在子目录
python3 保单明细/scripts/convert.py ...
```

**关键约束**:
- 禁止在子项目内创建 `scripts/` 目录（违反分层原则）
- 所有脚本通过 `run.sh` 统一调用（避免路径混乱）

---

### 2.2 数据文件的存储位置规则

**强制规则**:

| 数据类型 | 存储位置 | 文件格式 | 保留期限 |
|---------|---------|---------|---------|
| **原始数据** | `staging/` | .xlsx | 7天（临时） |
| **事实表** | `warehouse/fact/policy/` | .parquet | 永久 |
| **维度表** | `warehouse/dim/{维度名}/` | .parquet/.csv/.json | 永久 |
| **中间结果** | `staging/` | .xlsx | 7天（临时） |
| **日志** | `logs/` | .log/.json | 30天 |

**命名规范**:
```

**关键约束**:
- ⚠️ **硬编码路径必须更新为 warehouse/**（重构过程中的坑）
- `staging/` 和 `logs/` 必须在 `.gitignore` 中（避免提交临时文件）

### 2.3 配置文件的组织方式

**配置分层**:
```

├── default.yaml         # 默认配置（通用）
└── tasks/               # 任务配置模板

**配置内容**:

**使用方式**:
```bash

**关键洞察**:
> 配置文件支持两种模式：默认配置（覆盖式）和任务配置（场景化）。对于重复性任务（如每周数据更新），建议创建任务配置文件。

### 3.1 数据处理流水线的标准流程

**完整流程**:
```
1. 原始数据放入 staging/
   ↓
2. 运行 enrich.py (续保类型匹配)
   ↓
3. 运行 transform.py (Excel → Parquet)
   ↓

   ↓

    -i staging/已匹配数据.xlsx \
    -o warehouse/fact/policy/保单明细_20260201.parquet

    --source staging/历史完整数据.xlsx \
    --target staging/新导出数据.xlsx \

**关键实践**:
- ✅ 使用 `run.sh` 统一入口（自动检查依赖）
- ✅ 中间文件命名加 `_已匹配` 后缀（标识处理阶段）

### 4.2 问题2: 续保单号字段处理混乱

**问题描述**:
原始 Excel 数据中，"是否续保"列实际存储的是续保单号（保单号格式），而非布尔值。

**症状**:
```
是否续保列内容: PDZA202600001234
预期: True/False
```

**根本原因**:
- Excel 列名和实际内容不匹配
- 缺少数据质量检查机制

**解决方案**:
```python
# transform.py 第169-190行
# 检查"是否续保"列的值是否看起来像保单号（长度>10的字符串）
sample_values = df['是否续保'].dropna().head(100).astype(str)
looks_like_policy_no = sample_values.str.len().mean() > 10

if looks_like_policy_no:
    # "是否续保"列实际存储的是续保单号
    df['续保单号'] = df['是否续保'].apply(
        lambda x: x if (pd.notna(x) and str(x).strip() != '') else None
    )
    # 重新生成布尔值的"是否续保"
    df['是否续保'] = df['续保单号'].notna()
```

**预防措施**:
1. 增加数据质量报告: `analyze_data_quality()` 函数
2. 在转换前检查字段内容类型
3. 输出转换质量报告到 `logs/转换质量报告.json`

## 5. 技术沉淀

### 5.1 企业级数据处理架构模式

**模式总结**:
```
pipelines/    (处理层) - ETL脚本，单一职责
    ↓
warehouse/    (存储层) - 星型模型，事实+维度
    ↓
knowledge/    (知识层) - 文档+规则，AI友好
```

**适用场景**:
- 车险业务数据处理（保单明细、业务员计划）
- 需要 AI 辅助的数据分析项目
- 多源数据整合（Excel、CSV、Parquet）

**可复用组件**:
- `run.sh` - 统一命令行入口
- `pipelines/transform.py` - Excel → Parquet 转换器
- `pipelines/enrich.py` - 数据匹配增强器
- `knowledge/ai/` - AI 知识库结构

## 7. 关键教训总结

### 7.1 重构三原则

1. **先理解，后重构**: 花时间画出当前架构图，理解数据流向
2. **全局搜索，更新引用**: 移动文件后，全局搜索路径引用并更新
3. **分步验证，逐层测试**: 重构 → 单元测试 → 集成测试 → 用户验收

### 7.2 数据处理三要素

1. **可追溯**: staging 保留中间数据，logs 记录处理过程
2. **可验证**: 每个步骤输出质量报告
3. **可重复**: 使用配置文件和命令行工具，避免手工操作

### 7.3 AI 协作三策略

1. **知识分层**: 快速参考 → 业务规则 → 完整字典（按需加载）
2. **路径标准化**: 使用相对路径，避免绝对路径硬编码
3. **上下文优化**: AI 知识库单独存放，优化 token 消耗
