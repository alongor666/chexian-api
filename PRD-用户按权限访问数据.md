## 1. 执行摘要

### 1.1 概述

本PRD描述了为**车险业绩分析看板系统**（纯前端内网应用）实现"用户按权限访问数据"功能的需求。该系统基于 React + TypeScript + DuckDB-WASM 技术架构，通过 Nginx 内网部署，用户通过浏览器访问。

**核心特点**：
- 纯前端架构，所有数据处理在浏览器端完成
- 内网部署，依赖 Nginx 安全控制
- 数据文件（Parquet）存储在服务器端，前端动态加载

### 1.2 业务背景

当前系统存在的问题：

| 问题 | 影响 | 严重程度 |
|------|------|----------|
| **无身份认证** | 任何访问内网的用户可直接查看所有数据 | P0-严重 |
| **数据无隔离** | 所有登录用户均可访问全量数据 | P0-严重 |
| **角色权限缺失** | 无法区分管理员、业务员、管理层等不同用户 | P0-严重 |
| **敏感信息泄露风险** | 业务员业绩排名、机构业绩对比等敏感信息暴露 | P0-严重 |
| **无访问控制** | 无法控制用户能看到哪些数据 | P0-严重 |

### 1.3 解决方案概述

针对纯前端内网应用的特点，采用**多层防护架构**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           安全防护层级                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Layer 1: 网络层（Nginx）                                                │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ • IP 白名单限制（仅内网IP可访问）                                  │   │
│  │ • HTTP Basic Auth（可选密码保护）                                 │   │
│  │ • HTTPS 加密传输（可选）                                          │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                              ↓                                          │
│  Layer 2: 认证层（前端）                                                 │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ • 用户登录界面                                                    │   │
│  │ • Token 认证机制                                                  │   │
│  │ • 会话过期管理                                                    │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                              ↓                                          │
│  Layer 3: 权限层（前端）                                                 │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ • 角色权限配置（用户配置文件）                                     │   │
│  │ • 数据过滤引擎（根据角色动态过滤）                                 │   │
│  │ • UI 权限控制（菜单/按钮/组件）                                    │   │
│  │ • 数据脱敏处理（敏感数据隐藏）                                     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 1.4 预期收益

| 收益指标 | 预期改善 |
|---------|---------|
| 数据安全性 | ✅ 内网IP白名单 + ✅ 前端权限控制 |
| 合规性 | 满足内网数据安全访问要求 |
| 用户体验 | 用户仅看到授权数据，减少信息干扰 |
| 管理效率 | 管理员可通过配置文件管理用户权限 |

```
│                                                                         │
```

### 2.2 问题详细分析

| 问题 | 详细描述 | 影响 |
|------|----------|------|
| **无身份验证** | 任何知道内网地址的人都能访问系统 | 数据完全暴露 |
| **数据无隔离** | 所有用户加载相同的数据文件 | 无法区分查看权限 |
| **敏感数据泄露** | 业务员业绩排名、机构对比数据完全公开 | 商业机密泄露风险 |
| **管理功能缺失** | 无法管理用户、分配角色 | 无法精细化管控 |

### 2.3 用户痛点

#### 2.3.1 机构管理员视角

> "作为省级分公司管理员，我不希望下属地市公司的人员看到其他地市的业绩数据，但目前系统没有数据隔离功能，所有数据都是公开的。"

**具体痛点**：
- 无法限制下属机构只能查看本机构数据
- 无法隐藏其他机构的敏感业绩信息
- 无法为不同下属机构设置不同的数据访问权限

#### 2.3.2 总公司管理层视角

> "作为总公司管理层，我希望能看到全公司数据，但不希望所有员工都能看到全公司数据。目前系统无法实现这种分级管控。"

**具体痛点**：
- 无法实现"总公司看全部、分公司看辖属"的分级权限
- 无法为高管、部门经理、业务员设置不同的数据可见范围
- 无法控制敏感数据的知悉范围

#### 2.3.3 系统管理员视角

> "作为系统管理员，目前没有用户管理功能，无法新增用户、分配角色、回收权限，所有人共用一个访问入口。"

**具体痛点**：
- 缺乏用户账号管理能力
- 无法按需分配和回收权限
- 无法知道谁在什么时候访问了系统

### 2.4 问题根因分析

| 根因 | 说明 |
|------|------|
| 架构设计缺失 | 初始设计未考虑权限控制需求，缺少认证授权模块 |
| 数据层无隔离 | 所有用户加载相同的 Parquet 文件 |
| 前端无权限控制 | 页面组件未根据用户权限动态渲染 |
| 缺少用户系统 | 无用户账号、角色、权限的数据模型 |

### 2.5 业务价值

实现本功能将带来以下业务价值：

1. **数据安全保障**：防止内网中未授权用户访问敏感数据
2. **分级权限管理**：实现"总公司看全部、分公司看辖属"的分级管控
3. **责任可追溯**：通过登录记录实现访问行为追溯
4. **用户体验优化**：用户仅看到与自身相关的数据，减少信息干扰

## 3. 目标与成功指标

### 3.1 总体目标

在纯前端内网应用架构下，建立多层防护的用户权限管理体系：

1. **网络层安全**：通过 Nginx IP 白名单限制访问来源
2. **认证层安全**：实现用户登录认证机制
3. **权限层安全**：基于角色的数据访问控制
4. **展示层安全**：UI 权限动态控制

### 3.2 具体目标

| 目标ID | 目标描述 | 成功指标 |
|--------|----------|----------|
| OBJ-01 | 实现用户登录认证 | 用户可通过用户名密码登录，登录成功率≥99% |
| OBJ-02 | 建立角色权限体系 | 定义不少于5种角色，每种角色权限清晰 |
| OBJ-03 | 实现数据级权限控制 | 用户只能查看其权限范围内的数据 |
| OBJ-04 | 提供权限配置方式 | 支持通过 JSON 配置文件管理用户和角色 |
| OBJ-05 | 实现访问日志记录 | 记录用户登录和访问行为（本地存储） |

### 3.3 关键结果 (Key Results)

#### KR-01：认证系统
- [ ] 登录页面正常显示和验证
- [ ] 登录成功率 ≥ 99%
- [ ] 会话超时自动退出（默认30分钟，可配置）
- [ ] 支持"记住我"功能

#### KR-02：角色体系
- [ ] 定义角色：超级管理员、机构管理员、业务员、访客、只读用户
- [ ] 角色权限配置通过 JSON 文件管理
- [ ] 角色切换响应时间 ≤ 500ms

#### KR-03：数据权限
- [ ] 数据过滤准确率 100%
- [ ] 权限检查对查询性能影响 ≤ 10%
- [ ] 支持按机构代码、业务员代码等维度过滤

#### KR-04：UI权限
- [ ] 无权限用户无法访问受限页面
- [ ] 无权限菜单/按钮隐藏或禁用
- [ ] 权限变更后界面实时更新

#### KR-05：日志记录
- [ ] 记录用户登录行为
- [ ] 记录数据访问行为
- [ ] 日志存储在本地（localStorage）

### 5.2 详细旅程

| 5 | 查看KPI指标卡 | 显示"我的业绩"数据（仅本人） | ✅ 数据权限 |

**预期结果**：
- 王芳只能看到自己的业绩数据
- 无法选择查看其他业务员的业绩
- 无法导出数据（业务员角色无导出权限）

| 4 | 下钻到地市 | 显示所辖地市数据 | ✅ 数据权限 |
| 5 | 查看用户管理（管理员功能） | 显示用户列表 | ✅ 页面权限 |
| 6 | 尝试查看其他省份数据 | 数据为空或提示无权访问 | ✅ 数据权限 |

**预期结果**：
- 李华可以查看本省及下级地市的数据
- 李华可以访问用户管理页面
- 李华无法查看其他省份的数据

### 6.2 功能需求详细说明

#### FR-01: 网络安全层配置（Nginx）

**需求ID**: FR-01

| FR-04.4 | 权限验证 | 返回数据前验证是否越权 |

**数据权限规则**：
```typescript
// 根据用户角色和数据范围构建过滤条件
function buildDataPermissionFilter(user: User): { orgFilter: string; agentFilter: string } {
  switch (user.dataScope) {
    case 'all':
      return { orgFilter: '1=1', agentFilter: '1=1' };
    
    case 'org_hierarchy':
      // 机构管理员：查看本机构及下级机构
      return {
        orgFilter: `org_code LIKE '${user.orgCode}%'`,
        agentFilter: '1=1'
      };
    
    case 'org':
      // 机构只读：只查看本机构
      return {
        orgFilter: `org_code = '${user.orgCode}'`,
        agentFilter: '1=1'
      };
    
    case 'personal':
      // 业务员：查看本人及团队
      return {
        orgFilter: `org_code = '${user.orgCode}'`,
        agentFilter: `agent_code = '${user.agentCode}' OR team_code = '${user.agentCode}'`
      };
    
    case 'limited':
      // 访客：限制访问特定机构
      const allowedOrgs = user.dataLimit?.map(o => `'${o}'`).join(',') || "''";
      return {
        orgFilter: `org_code IN (${allowedOrgs})`,
        agentFilter: '1=1'
      };
    
    default:
      return { orgFilter: '1=1', agentFilter: '1=1' };
  }
}

// 修改 SQL 查询生成器，注入权限过滤
function generateKPIQuery(filters: FilterState, user: User): string {
  const permission = buildDataPermissionFilter(user);
  
  return `
    SELECT 
      SUM(premium) as total_premium,
      COUNT(*) as policy_count
    FROM policy_fact
    WHERE 1=1
      AND ${permission.orgFilter}
      AND ${permission.agentFilter}
      -- 其他筛选条件...
  `;
}
```

**用户故事**：
> 作为业务员，我希望只能看到自己和团队的业绩数据，而看不到其他人的数据。

**验收标准**：
- [ ] 业务员登录后，筛选器仅显示本机构选项
- [ ] 查询结果仅包含权限范围内的数据

```typescript
// 1. 路由守卫组件
function ProtectedRoute({ children, requiredPermission }: Props) {
  const { user, hasPermission } = useAuth();
  
  if (!user) {
    return <Navigate to="/login" replace />;
  }
  
  if (requiredPermission && !hasPermission(requiredPermission)) {
    return <Navigate to="/unauthorized" replace />;
  }
  
  return <>{children}</>;
}

// 2. 权限 Hook
function usePermission(permission: string): boolean {
  const { user } = useAuth();
  if (!user) return false;
  if (user.roleId === 'admin') return true;
  return user.permissions.includes(permission);
}

// 3. 权限组件
function Can({ permission, children, fallback = null }) {
  const hasPermission = usePermission(permission);
  return hasPermission ? <>{children}</> : <>{fallback}</>;
}
```

```typescript
    WHERE 1=1
  `;
  
  // 添加用户权限过滤
  const permissionConditions = [];
  
  if (user) {
    // 机构过滤
    if (permissionFilter.orgCodes.length > 0) {
      const orgList = permissionFilter.orgCodes.map(c => `'${c}'`).join(',');
      permissionConditions.push(`org_code IN (${orgList})`);
    }
    
    // 业务员过滤
    if (permissionFilter.agentCodes.length > 0) {
      const agentList = permissionFilter.agentCodes.map(c => `'${c}'`).join(',');
      permissionConditions.push(`agent_code IN (${agentList})`);
    }
  }
  
  const permissionWhere = permissionConditions.length > 0
    ? `AND ${permissionConditions.join(' AND ')}`
    : '';
  
  // 其他筛选条件...
  const filterWhere = buildFilterConditions(filters);
  
  return `
    ${baseQuery}
    ${permissionWhere}
    ${filterWhere}
  `;
}
```

### 11.3 重要说明：纯前端安全的局限性

⚠️ **重要提示**：纯前端权限控制存在固有限制，需要明确告知用户：

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| **浏览器可调试** | 用户可通过浏览器开发者工具修改前端代码或绕过检查 | 网络层 Nginx IP 白名单是第一道防线 |
| **数据文件可访问** | 如果用户知道数据文件URL，可能直接下载 | 网络层限制数据目录访问权限 |
| **配置可查看** | 配置文件（users.json）可能被查看 | 密码使用 bcrypt 加密，敏感配置可考虑编码 |
| **无后端验证** | 所有权限验证都在前端，无法防止专业攻击 | 依赖内网安全性，作为内网工具使用 |

**适用场景**：
- ✅ 内网环境，信任的用户群体
- ✅ 防止误操作和常规越权访问
- ✅ 满足内网数据访问管理需求

**不适用场景**：
- ❌ 对抗性安全环境
- ❌ 高度敏感数据保护
- ❌ 需要防篡改的场景

### 14.3 技术实现最佳实践

#### 前端权限组件模式

```typescript
// 权限检查 Hook
import { useMemo } from 'react';
import { useAuth } from './useAuth';

export function usePermission(permission: string): boolean {
  const { user } = useAuth();
  
  if (!user) return false;
  if (user.roleId === 'admin') return true;
  
  return user.permissions.includes(permission);
}

// 权限检查组件
interface CanProps {
  permission: string;
  children: React.ReactNode;
  fallback?: React.ReactNode;
}

export function Can({ permission, children, fallback = null }: CanProps) {
  const hasPermission = usePermission(permission);
  return hasPermission ? <>{children}</> : <>{fallback}</>;
}

// 路由守卫
function ProtectedRoute({ children, requiredPermission }: Props) {
  const { isAuthenticated, hasPermission } = useAuth();
  
  if (!isAuthenticated) {
    return <Navigate to="/login" replace />;
  }
  
  if (requiredPermission && !hasPermission(requiredPermission)) {
    return <Navigate to="/unauthorized" replace />;
  }
  
  return <>{children}</>;
}
```

#### 数据权限过滤

```typescript
// 权限过滤服务
export class PermissionFilterService {
  static buildOrgFilter(user: User): string {
    switch (user.dataScope) {
      case 'all':
        return '1=1';
      case 'org_hierarchy':
        return `org_code LIKE '${user.orgCode}%'`;
      case 'org':
        return `org_code = '${user.orgCode}'`;
      case 'personal':
        return `org_code = '${user.orgCode}'`;
      case 'limited':
        const allowedOrgs = user.dataLimit?.map(o => `'${o}'`).join(',') || "''";
        return `org_code IN (${allowedOrgs})`;
      default:
        return '1=1';
    }
  }
  
  static buildAgentFilter(user: User): string {
    switch (user.dataScope) {
      case 'personal':
        return `agent_code = '${user.agentCode}' OR team_code = '${user.agentCode}'`;
      default:
        return '1=1';
    }
  }
  
  static buildFullFilter(user: User): string {
    const orgFilter = this.buildOrgFilter(user);
    const agentFilter = this.buildAgentFilter(user);
    return `(${orgFilter}) AND (${agentFilter})`;
  }
}
```

### 14.4 安全注意事项

⚠️ **重要安全提示**

由于本系统是纯前端应用，存在以下固有限制：

| 风险 | 说明 | 建议 |
|------|------|------|
| **配置文件可见** | users.json 和 roles.json 可被下载查看 | 使用 bcrypt 加密密码，不要存储明文密码 |
| **前端代码可修改** | 用户可修改浏览器中的 JavaScript 代码 | 依赖 Nginx 网络层安全作为主要防线 |
| **数据文件可访问** | 如果知道 URL，可直接下载数据文件 | 限制 /data/ 目录的访问权限 |
| **无防篡改保护** | 无法防止专业用户绕过前端检查 | 仅在内网环境使用，信任用户群体 |
