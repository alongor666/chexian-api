# ADR + 复杂度分级响应（ADR Tiered Response）

> 精髓借鉴：super-powers-team-suite architect 的"三档响应 + ADR ≥2 备选方案"。本 skill 复用既有 `开发文档/TECHNICAL_DECISIONS.md` 作为 ADR 之家（不另起目录，守单一事实源），补上"结构化模板 + 按复杂度分档"两件它缺的事。
>
> 核心思想：**响应深度匹配改动复杂度**——bugfix 别写长篇大论，新模块别只甩一句话。

## 1. 三档响应（按改动复杂度选档）

| 档 | 适用 | 产物 | 硬约束 |
|----|------|------|--------|
| **L1 架构签收** | bugfix、小改、配置更新 | 一句话确认"不破坏既有架构/契约/不引新依赖" | **严格 ≤3 行，超了算不合格** |
| **L2 影响评估** | 新特性、跨模块改动、重构 | 直接/间接受影响模块 + 接口契约变更 + 必要时写 ADR | 列出受影响模块的依赖方向 |
| **L3 完整 ADR** | 新项目、新模块、技术栈变更 | 完整决策记录（见 §2 模板）+ 数据流/模块划分 | 必含 ≥2 个备选方案 |

**L1 签收模板**（≤3 行）：
```
✅ 架构签收 — 影响模块：<模块> — 接口不变 / 符合既有 ADR / 无新依赖
```

## 2. ADR 模板（记入 `开发文档/TECHNICAL_DECISIONS.md`）

新增决策时追加一节，**必含至少 2 个备选方案**（只有一个方案 = 没真正权衡，不合格）：

```markdown
## N. <决策标题>

- **状态**: 采纳 | 已弃用 | 被替代（被 ADR-M 取代）
- **背景**: 为什么需要这个决策？解决什么问题？
- **决策**: 决定了什么？（具体、可执行）
- **备选方案**（≥2）:
  - 方案 A: <描述> — 优点 … / 缺点 …
  - 方案 B: <描述> — 优点 … / 缺点 …
- **结论**: 为什么选这个而非其他？
- **影响**: 波及哪些模块？带来什么约束？
- **代码位置**: `path/to/file.ts:line`
```

> 与现有 `TECHNICAL_DECISIONS.md` 的 freeform 条目兼容：旧条目不强制回填，**新决策按此结构化模板写**，逐步收敛。

## 3. 子域护栏（写 ADR/架构说明时）

| 子域 | 护栏 |
|------|------|
| 样式/UI | 每个样式决策须引用 `src/shared/styles/index.ts` 或既有模式作证据，不凭空造 |
| 模块 | 标注依赖方向（依赖谁 / 被谁依赖），防循环依赖 |
| 接口契约 | 标注向后兼容状态与弃用策略；与 `api-routes.ts` / `routes.ts` 注册表一致 |
| 决策(ADR) | 至少 2 个备选方案 |
| 签收 | ≤3 行 |

## 4. 与现有体系的关系

- **指标/字段决策** → 仍走 CLAUDE.md §2 注册表流程（registry 是代码层 SSOT），ADR 只记"为什么这么设计"的架构层理由。
- **部署链决策** → 走 `.claude/rules/deploy-chain-sop.md`，不重复进 ADR。
- **本 skill 专管**：架构/技术选型类改动的"响应该写多深 + 决策怎么结构化留痕"。
- **复杂度↔档位不匹配**（如新模块只给了 L1 签收）= 红灯，补到对应档位再继续。

> 母原理：本 skill 是 `agent-system-design-principles` 原理⑤（投入配风险）的落地件。
