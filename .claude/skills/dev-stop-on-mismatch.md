# 开发执行期停机纪律（stop-on-mismatch）

> 精髓借鉴：super-powers-team-suite 的 developer "stop-on-mismatch"。本 skill 用 chexian 自己的护栏体系重述，落地到本项目的「缺口清单 / BACKLOG / 注册表」三条升级通道。
>
> 定位：CLAUDE.md §0 红线"修补不拆除"+ §7"信息缺口→BLOCKED"覆盖的是「任务开始前」和「业务口径」；本 skill 补的是**任务执行中途**——代码写到一半发现现实与计划不符时的处置纪律。

## 1. 三类必停触发器

| 触发器 | 识别信号 | 立即动作 | 升级通道 |
|--------|---------|---------|---------|
| **架构/契约冲突** | 实现与 `ARCHITECTURE.md`、API 路由契约（`api-routes.ts` / `routes.ts`）、指标/字段注册表的既有定义矛盾 | 停 → 在回复中标 `⚠️ ARCH_CONFLICT` | 注册表改动走 CLAUDE.md §2 流程；路由/架构改动先问用户 |
| **范围扩张** | 任务需要改"当前任务清单之外"的文件，或引入未声明的新依赖/新字段/新指标 | 停 → 标 `⚠️ SCOPE_EXPANSION` | 新需求登记 BACKLOG（ID 取全局最大 +1，追加到末尾）由用户排期 |
| **设计缺陷/口径矛盾** | 任务描述技术上不可行，或与业务规则字典/指标字典口径相互矛盾 | 停 → 标 `⚠️ BLOCKER` | 业务口径错误登记缺口清单 + BACKLOG，**禁止直接改口径**（§7） |

## 2. 禁用话术（出现即违规）

这些话术等于"未登记的偷偷扩范围 / 静默变通"，**一律禁止**：

- ❌ "我绕一下 / 变通处理一下"
- ❌ "顺手把这个也改了 / 就多改这一个文件"
- ❌ "先这样，以后再修 / 留个 TODO 回头补"
- ❌ 静默地重新解释任务、单方面重新设计
- ❌ 同一处连续修 3 次还没好，继续打补丁（第 3 次失败必须停 + 升级）

## 3. 停机后的升级模板

在对话中给出（并按通道登记），等用户/上游裁决，**不要自行绕过**：

```
## ⚠️ 停机升级
**类型**：ARCH_CONFLICT / SCOPE_EXPANSION / BLOCKER / 3次修复失败
**任务**：<当前在做什么>
**冲突点**：<现实 与 计划/契约 的具体矛盾，附 file:line>
**影响范围**：<会波及哪些模块/口径/下游>
**建议**：<至少给一个可选方案，不替用户拍板>
**状态**：⏸️ 已停机，等待裁决
```

## 4. 与现有护栏的关系（不重复造轮子）

- **任务开始前**的"先搜再写""信息缺口→BLOCKED" → 已由 CLAUDE.md §0/§7 覆盖，本 skill 不重复。
- **业务口径不确定** → 走 §10 领域知识铁律 + 业务规则字典，先问用户。
- **本 skill 专管**：已经进入编码/改 SQL 后，中途撞墙时的"停-记-升"动作纪律。
- **3 次失败规则** 与系统性 debug 配合：连续失败先怀疑根因/架构，而非继续试补丁。

> 母原理：本 skill 是 `agent-system-design-principles` 原理④（有界授权+断路器）的落地件。
