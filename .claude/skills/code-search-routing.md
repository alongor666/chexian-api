# 代码检索工具路由（Code Search Routing）

> 来源：评估 super-powers-team-suite 的 "codegraph 优先 → grep fallback" 规则时得出的结论。**固定优先级是教条，不是科学**——最优做法是按查询意图路由到「能以最小不确定性给出正确答案」的工具。"更高抽象的工具" ≠ "永远更好"。
>
> 语义独立性：本条原则抹掉 chexian/DuckDB 仍成立（通用工程原则，过 rule-promotion-gate 的语义独立测试 ✅），§3 是本项目的落地路由。

## 1. 路由表（按查询意图选刀）

| 查询意图 | 最优工具 | 理由 |
|---------|---------|------|
| 谁调用 X / X 调用谁 / 改 X 的影响面 / 类型层级 | **LSP（find-references / call-hierarchy）** | 来自类型检查器，**精确 + 实时不陈旧**；grep 按名字搜会被注释/字符串/同名方法/别名导入污染（有损代理）|
| 某字符串/id/列名/路由串/环境变量/中文业务词 在哪出现 | **grep** | 这些多半不是符号，图谱可能没索引；grep 精确且覆盖一切文件类型 |
| 不知道名字、按"概念/做什么"找代码 | **embedding 语义检索** | grep 要求已知确切 token |
| 刚改过的文件里做精确存在性匹配 | **grep** | grep 永远对应磁盘当前字节；图谱/embedding 索引有陈旧期 |
| config / SQL / markdown / JSON / Parquet 邻接文本 | **grep** | 图谱面向 AST/符号，覆盖不到 |

**精确性排序**：LSP > grep(对字面量) > embedding 图谱。
**覆盖面/抗陈旧排序**：grep > LSP > embedding 图谱。
→ 把 embedding 图谱无条件置顶（"图谱优先"）在科学性上站不住；"grep 唯一"则放掉了符号级引用/影响的真正胜势。

## 2. 反教条原则

- ❌ 固定"图谱优先 → grep fallback"：对字面量/跨格式查询是错配，且有索引陈旧风险（图谱会"撒谎"）。
- ❌ 固定"grep 唯一"：对"改这个接口波及谁"这类查询是有损代理。
- ✅ **按问题类型选刀**：先判"我在找符号引用，还是找文本/字面量，还是按概念找"，再选工具。

## 3. chexian 落地路由（本项目证据）

本项目是数据分析库，**绝大多数检索目标是字符串而非符号**——指标 id、字段名、路由串、Parquet 列名、SQL 模板片段、中文业务关键词。这些在代码图谱里多半不是一等公民；且本项目频繁改 SQL 生成器/注册表，陈旧图谱索引会主动误导。

| 在 chexian 搜什么 | 用什么 |
|------------------|--------|
| 指标/字段是否已存在（CLAUDE.md §2/§字段注册表流程）、SQL 片段、路由串、中文口径词 | **grep**（默认，与 CLAUDE.md §0 红线"先搜再写"一致）|
| `server/src` 里"改这个函数/接口/类型波及哪些调用方" | **LSP find-references / call-hierarchy**（grep 干不好的那类）|
| 跨域"哪里还有类似实现"（防重复造轮子）| grep 起步，符号确认用 LSP |

**结论**：grep 作默认在本项目是经验正确（非落后）；真正该补的升级是给 TS 层用 **LSP 精确引用/影响分析**，而非照搬 embedding 图谱优先。

## 4. 与现有护栏的关系

- CLAUDE.md §0 红线"先搜再写 `grep -r`" → 本 skill 不推翻它，而是**精化**：grep 仍是默认，符号级引用/影响查询补 LSP。
- 不引入 CodeGraph MCP 强依赖（避陈旧性陷阱 + 避 AGPL 套件的工具绑定）。

> 母原理：本 skill 是 `agent-system-design-principles` 原理⑤（按问题选刀 = 投入配风险）的落地件。
