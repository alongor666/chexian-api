# 对抗性评审：重构/迁移审计计划（Refactor-Audit Plan Review）

> 来源：2026-06 ApiClient 神类拆分（1250 行 → core + 10 子客户端）Phase 2 适配审计计划的对抗性评审。沉淀"如何严苛评审一份'证明重构无遗漏'的计划"。
>
> 语义独立性：纯通用工程评审方法，抹掉 chexian/车险仍成立（应过 `rule-promotion-gate`），可上提共享仓；§8 是本项目落地锚点。
>
> 配套执行 skill：[chexian-refactor-audit-execute](./chexian-refactor-audit-execute.md)（被评审计划照此标准来打分）。

## 0. 立场与铁律

评审的对象是**计划**，不是（只是）代码。一份"看起来很 MECE、有红队章节、分了 A–F 维度"的计划，恰恰最危险——它的完整性是**表演性**的，盲区藏在"检查项的措辞"和"严重度的分档"里。

| 铁律 | 含义 |
|------|------|
| **验证不声称（对计划同样适用）** | 计划里每条事实声明（"client.ts 1250 行"/"82 个契约"/"transport 只读"/"10 个子客户端"）都先用 `grep`/`git show`/`Read` 核对。**计划写错的事实本身就是一条发现**，且动摇其结论可信度。 |
| **工具≠目标** | 每个"检查项"问一句：*这个命令/方法真能测到它声称要测的东西吗？* 大量计划用错刀（tsc 测不到字符串路径改动；数一个文件的用例数测不到分散在多文件的契约）。 |
| **重判严重度** | 计划自己的分档（阻塞/非阻塞、P0/P1）必须挑战。最常见病：一个**近乎必 FAIL** 的项被写成"中优先级 checklist 项"。 |
| **MECE 由你补全** | 计划声称 MECE 不代表 MECE。用固定维度清单（§3）扫一遍，缺哪维直接点名。 |
| **证过去 vs 防未来** | 见 §4。这是最深的方法论刀。 |

## 1. Step 0 — 地基核验（先把计划的事实打穿）

不读完计划就开评是外行。但读完就信是更大的外行。**对计划引用的每个数字/结构/不变量，跑命令核对**：

- 文件规模："X 行神类" → `git show <split前commit>~1:<file> | wc -l` 取拆分前真值。
- 覆盖数量："N 个契约用例" → 别数 `expect*` 助手（可能用 `it.each`/`.toContain`），数 `it(`/`test(` + `it.each` 数组条目。注意契约可能**集中在一个文件**也可能**按域分散**——数错地方=评审失效。
- 接口形状："transport 只读" → `Read` 那个 interface；**但别停在 interface**：再 `grep` 具体类是否仍 `public` 暴露写方法、且**被外部调用**（这是 §2 的经典反例）。
- 残留调用："tsc 绿即无残留" → 自己 `grep` 旧形态调用点 + 测试 mock + bracket 访问，确认 tsc 盲区里没有幸存者。

> 产出：一张"计划声明 vs 实测"对照表。任何不符 = 发现（标注它如何影响计划结论）。

## 2. Step 1 — 工具有效性质询（盲区目录）

对计划每个检查项，套这张"刀—盲区"表。**计划没声明盲区的，替它声明；声明了但当 checklist 项轻描淡写的，升级严重度。**

| 计划常用的"刀" | 它**测不到**什么（盲区） | 评审动作 |
|---------------|------------------------|---------|
| `tsc`/`bun run build` | 字符串字面量改动（路径、query key、HTTP 动词）；bracket/反射访问 `obj['x']`；测试里的 mock 形状；被同 PR 一起删掉调用点的"丢失方法" | 要求补 grep + 基线 diff |
| 契约测试断言 path | **鉴权头**（Authorization/Bearer）、**缓存行为**（dedupe key / SW / react-query key）、请求体、默认值、编码 | 点名"契约只锁 URL，没锁 X" |
| "数某文件的用例数" | 契约分散在多文件时漏数；用例存在≠断言有效（可能 assert 了错的期望值） | 核对契约**布局**与**断言内容** |
| 读 interface 确认"只读" | 具体类/单例仍 public 暴露写方法且被外部调用 | grep 写方法的**外部调用点** |
| "CI 绿/生产健康绿" | CI 常**无数据**跑（集成/数据路径未覆盖）；绿只证类型+单测 | 区分"绿证明了什么 vs 没证明什么" |
| 人工列举"为何保留" | 定性判断无**守恒**支撑——漏迁一个方法（连调用点一起删）无人发现 | 要求守恒恒等式（§3-B） |
| E2E 点页面 | 只覆盖被点到的页；非交互路径/错误路径/并发路径漏 | 要求列出**未覆盖**页与路径 |

## 3. Step 2+3 — 缺失维度扫描（MECE 兜底清单）

一份神类拆分/大迁移审计计划**必须**覆盖以下维度。逐条对计划打勾，缺的直接列为"新发现漏项"：

| 维度 | 必答问题 | 缺失的典型后果 |
|------|---------|--------------|
| **行为等价（金 master）** | 有没有"对拆分前基线"捕获真实发出的请求（URL+方法+体+头+dedupe key）做 diff？ | 逐方法搬运的转写错误（路径 typo/默认值丢/GET↔POST）静默漏网 |
| **守恒恒等式** | `count(原公开方法) − count(保留) == count(分散后)` 有没有自动校验？ | 漏迁/丢方法无信号 |
| **横切传输行为** | 401 静默刷新 / GET 合并 / 超时 / 取消 这些**所有方法共享**的核心逻辑，有没有独立测试？是否纳入门禁？ | 风险最高的代码常恰恰**零测试、不设门禁** |
| **持久护栏（防回归）** | 拆完用什么**机制**防后人把它拆回去（子模块注入写权限 / new 第二个 core / 方法回流神类）？是 checklist 还是**验收闸**？ | 拆分价值随时间蒸发；这才是拆分的**目的** |
| **门禁迁移连续性** | 凡是**按路径**锚定旧文件的门禁/脚本，是否已重指到**新文件集**？ | "掏空陷阱"：业务方法搬走后，原门禁只剩守着空壳（见 §8） |
| **文档/索引一致性** | 索引、必读文档、规则、agent/skill 的影响半径是否含新文件？同名异物是否区分？ | 后人按陈旧地图行动 |
| **回滚/还原安全** | 跨 N 个已合并 PR 出事了，回退路径是什么？ | 事故时无预案 |
| **性能/模块图** | 拆分对 bundle/tree-shaking/初始化序有无影响？ | 隐性体积或初始化 bug |
| **运行时实证** | build/全量单测/E2E/健康检查/端点 diff 是否**本轮真跑**（非"上次绿"）？ | 回归被"契约绿"掩盖 |

## 4. Step 4 — "证过去 vs 防未来"判别（最深的刀）

重构审计有**两个互不替代**的目标，弱计划把它们坍缩成同一套工具：

- **迁移正确性（证过去）**：搬运有没有出错？唯一可靠工具 = **对拆分前基线的特征化/金 master diff**。契约测试**不胜任**——若你照新代码写契约、新代码又搬错了，两者一起错、测试照绿（自洽陷阱）。
- **防漂移（防未来）**：以后改动会不会偏离？工具 = **契约测试 + 门禁 + 护栏**。

> **评审判据**：若计划用"契约 82 绿"同时充当"证明搬运无误"+"防未来漂移"，判**方法论缺陷**——它有"防未来"的刀，缺"证过去"的刀。点名要求补金 master 基线 diff。

## 5. Step 5 — 可证伪的总闸

计划是否定义了**单一聚合判据**（"审计通过 ⟺ 阻塞项零 FAIL ∧ 守恒恒等式成立 ∧ ≥1 条持久护栏已合并 ∧ 横切核心有测试"）？还是只有"逐项 PASS/FAIL"的清单？

无总闸的计划永远"差不多完成"，没有一个布尔真值。要求补一个**可证伪**的 Definition of Done。

## 6. 反模式（弱评审长这样）

- ❌ 复述计划结构、夸它"很全面" → 没有任何新发现。
- ❌ 只挑文档陈旧（最易找）→ 漏掉门禁掏空、横切零测试这类高severity结构病。
- ❌ 接受计划的严重度分档不挑战。
- ❌ 不跑命令，凭计划自述下结论（违反"验证不声称"）。
- ❌ 把范围外（OUT）的项当发现刷数量。

## 7. 产出格式

1. **地基核验对照表**：计划声明 vs 实测（含证据命令）。
2. **逐 ID 裁决**：对计划每个检查项 `PASS/FAIL/N-A + 证据 + 工具盲区批注`。
3. **严重度重判**：哪些项被低估/高估，给出新分档。
4. **新发现漏项**：§3 维度扫描缺的，逐条 + 后果 + 阻塞/非阻塞。
5. **总闸建议**：一句可证伪的 DoD。
6. **公允**：明确点出计划**做对**的地方（避免为批而批失真）。

## 8. 本项目落地锚点

- 门禁掏空陷阱实例：`scripts/check-hotfile-contracts.mjs` 仅锚 `src/shared/api/client.ts`；业务方法搬到 `*-api.ts` 后，该门禁只守 ~15 个保留方法，80% API 面失保护。评审此类拆分必查"按路径锚定的门禁是否重指新文件集"。
- 契约现状：`tests/api/client-contracts.test.ts`（83 例，`it.each` + `.toContain(path)`，断写操作动词，**不断鉴权头**）。
- 横切核心：`src/shared/api/client-core.ts`（401 刷新 / GET 合并 / 超时）—— 当前零专测、不在门禁。
- 验证命令：`bun run build` · `bun run test` · `bun run governance` · `git show <commit>~1:<file>`。

## 9. Definition of Done（本评审）

- [ ] 计划每条事实声明已实测核对（含至少一处用 `git show` 取拆分前真值）。
- [ ] 每个检查项标注了所用刀的盲区。
- [ ] §3 九维全部扫过，缺失项已列。
- [ ] 给出了"证过去 vs 防未来"判别结论。
- [ ] 给出可证伪的总闸 + 严重度重判 + 公允的优点。
