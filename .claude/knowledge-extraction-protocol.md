# 隐性知识提取与结构化协议

**目标**: 系统化提取对话中的隐性知识,转化为可复用的结构化文档

**核心原则**:
1. ✅ **零遗漏**: 不遗漏任何您在对话中提到的业务规则、决策逻辑、约束条件
2. ✅ **确认优先**: 所有提取的知识必须经过您的确认
3. ✅ **结构化存储**: 转化为易于检索的文档格式
4. ✅ **双向追溯**: 从代码反向追溯到知识来源,从知识正向追溯到代码实现

---

## 🎯 工作流程

### Phase 1: 全面扫描 (Knowledge Scan)

**触发时机**: 每次重要对话结束后

**扫描范围**:
```
对话历史
  ├── 您明确陈述的业务规则 ("X字段表示...", "Y规则是...")
  ├── 您纠正的错误理解 ("不对,应该是...", "我之前说的不对...")
  ├── 您提供的背景信息 ("这个系统是为了...", "历史原因是...")
  ├── 您确认的假设 ("可以这样做", "同意这个方案")
  ├── 您提出的约束 ("必须...", "不能...", "禁止...")
  ├── 您解释的例外情况 ("除非...", "特殊情况下...")
  └── 您指定的优先级 ("优先级: X > Y > Z")
```

**扫描方法**:
1. 关键词搜索: "规则"、"应该"、"必须"、"不能"、"注意"、"重要"
2. 纠正标记: "不对"、"不是"、"错误"、"重新理解"
3. 决策点: "决定"、"选择"、"确定"、"采用"
4. 约束条件: "要求"、"限制"、"条件"、"前提"
5. 业务术语: "口径"、"含义"、"表示"、"对应"

### Phase 2: 知识分类 (Knowledge Classification)

**分类维度**:

#### A类: 业务规则 (Business Rules)
```
定义: 描述业务逻辑、数据含义、计算规则

示例:
- "续保单号为空表示非续保保单,不是数据缺失"
- "负保费表示批改退费,必须伴随批改记录"
- "实收保费 = SUM(保费) (正负抵消)"

存储位置:
  - 数据管理/knowledge/rules/车险数据业务规则字典.md (业务规则章节)
  - 开发文档/00_index/DATA_INDEX.md (快速查找)
```

#### B类: 技术约束 (Technical Constraints)
```
定义: 描述技术实现限制、架构要求

示例:
- "日期字段存储为字符串,必须CAST转换后才能计算"
- "Worker与主线程通信必须使用Arrow IPC,禁止JSON序列化"
- "DuckDB强制要求COOP/COEP头"

存储位置:
  - CLAUDE.md § 2 护栏 (架构协议)
  - 开发文档/TECH_STACK.md (技术栈说明)
```

#### C类: 开发规范 (Development Standards)
```
定义: 描述代码风格、工作流程、验证协议

示例:
- "所有修改必须先写单元测试,再写实现代码"
- "禁止自我安慰式开发,必须在浏览器实测验证"
- "提交前必须运行治理校验"

存储位置:
  - CLAUDE.md § 6 验证协议
  - 开发文档/00_index/CODE_INDEX.md (开发规范)
```

#### D类: 历史决策 (Historical Decisions)
```
定义: 描述为什么采用某种方案、拒绝其他方案的原因

示例:
- "为什么用DuckDB而不是SQLite: 支持大规模数据分析"
- "为什么用Arrow IPC而不是JSON: 性能提升10倍"
- "为什么采用分层加载策略: Token消耗降低80%"

存储位置:
  - PROGRESS.md (决策记录)
  - BACKLOG.md (任务背景)
```

#### E类: 例外情况 (Exception Cases)
```
定义: 描述特殊场景、边界条件、错误处理

示例:
- "除非是营业货车,否则吨位分段字段没有意义"
- "当续保单号非空但是否续保=False时,表示数据异常"
- "批改类型为'16退保'时,保费必须为负值"

存储位置:
  - 数据管理/knowledge/ai/DATA_FLOW_KNOWLEDGE.md (异常模式)
  - 开发文档/00_index/DATA_INDEX.md (常见陷阱)
```

#### F类: 待确认问题 (Pending Questions)
```
定义: 对话中提出但未确认的问题、歧义、假设

示例:
- "是否需要考虑跨年续保的情况?" (待确认)
- "批改记录是否可能有多条?" (待确认)
- "过户车是否一定有批改记录?" (待确认)

存储位置:
  - BACKLOG.md (新增任务,状态=PROPOSED)
  - 开发文档/00_index/PROGRESS_INDEX.md (待解决问题)
```

```markdown

**AI提取理解**:
[规范化表述]

- 🔧 需要修正: [指出哪里不准确]
```

**您的反馈方式**:
1. ✅ 确认正确 → AI归档到知识库
2. 🔧 修正理解 → AI重新提取并再次确认
3. ➕ 补充信息 → AI更新规则并再次确认
4. ❌ 丢弃 → AI标记为"非规则"并说明原因

## 🔄 持续改进机制

### 1. 知识审计 (每季度)
```
目标: 发现并修正过时、不一致、冲突的知识

步骤:
1. 代码反向扫描:
   - 检查代码实现是否与知识库规则一致
   - 发现"只有代码、没有文档"的隐性知识

2. 文档交叉验证:
   - 检查不同文档对同一规则描述是否一致
   - 发现冲突定义

3. 用户体验反馈:
   - 收集AI协作中的常见问题
   - 识别知识缺口

4. 清理与更新:
   - 归档过时知识
   - 修正冲突定义
   - 补充缺失规则
```

### 2. 知识版本管理
```
主知识库: [文件路径].md (当前版本)
历史版本: [文件路径]_v[YYYYMMDD].md (归档版本)

变更日志:
- [YYYY-MM-DD] 新增规则X
- [YYYY-MM-DD] 修正规则Y
- [YYYY-MM-DD] 归档规则Z
```

### 3. 知识质量指标
```
完整性指标:
- 字段规则覆盖率 = 有完整规则的字段数 / 总字段数
- 代码实现覆盖率 = 有代码实现的规则数 / 总规则数
- 测试验证覆盖率 = 有测试的规则数 / 总规则数

准确性指标:
- 规则一致性评分 = 1 - (冲突规则数 / 总规则数)
- 代码规则一致性 = 代码与规则一致的实现数 / 总实现数

可用性指标:
- 知识检索成功率 = 找到相关知识的查询次数 / 总查询次数
- 平均检索时间 = 从查询到定位知识的平均时间
- 用户满意度 = 用户反馈评分
```
