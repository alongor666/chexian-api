## 4. 数据传输层改造

### 4.1 当前架构（前端）

```
主线程                 Worker线程
  ↓                      ↓
DataContext          DuckDBWorker
  ↓ postMessage         ↓
  |------------------→ query(sql)
  |                      ↓
  |                   执行DuckDB查询
  |                      ↓
  |                   返回Arrow IPC Buffer
  |←------------------|
  ↓ tableFromIPC
使用数据
```

### 4.2 目标架构（后端）

```
前端                    后端
  ↓                      ↓
useKpiData Hook      Express /api/query/kpi
  ↓ HTTP Request         ↓
  |------------------→ Auth Middleware (JWT)
  |                      ↓
  |                   Permission Middleware (行级过滤)
  |                      ↓
  |                   DuckDB Service (查询)
  |                      ↓
  |                   返回 JSON 或 Arrow IPC
  |←------------------|
  ↓
使用数据
```

### 4.3 改造重点

| 层级 | 前端 | 后端 | 改造难度 |
|------|------|------|----------|
| **请求发起** | `worker.postMessage()` | `fetch('/api/...')` | 🟡 中等 |
| **认证** | localStorage (不安全) | JWT Token验证 | 🟢 简单 |
| **权限过滤** | 前端PermissionContext | 后端WHERE注入 | 🟡 中等 |
| **查询执行** | DuckDB-WASM | DuckDB Node | 🟢 简单 |
| **响应格式** | Arrow IPC Buffer | JSON（或Arrow IPC） | 🟢 简单 |

**推荐方案**:
- Phase 1: 使用JSON响应（开发快速，调试方便）
- Phase 2: 切换到Arrow IPC（性能优化，数据量大时）

## 5. 权限系统改造

### 5.1 当前权限机制（前端）

```typescript
// ❌ 前端权限验证（不安全，可被绕过）
const PermissionContext = createContext<UserPermission>({
  username: 'leshan',
  role: UserRole.ORG_USER,
  organization: '乐山',
});

// ❌ 前端WHERE子句生成（用户可修改代码绕过）
const whereClause = permission.role === UserRole.ORG_USER
  ? `org_level_3 LIKE '%${permission.organization}%'`
  : '1=1';
```

**安全问题**:
1. 用户可通过浏览器DevTools修改权限配置
2. 用户可直接访问DuckDB-WASM，查询任意数据
3. 用户可绕过筛选器，看到其他机构数据

### 5.2 目标权限机制（后端）

```typescript
// ✅ 后端JWT验证（强制执行）
app.use('/api/*', authMiddleware);

function authMiddleware(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  const user = jwt.verify(token, JWT_SECRET); // 验证签名
  req.user = user; // 注入用户信息
  next();
}

// ✅ 后端行级安全过滤（SQL注入）
function permissionMiddleware(req, res, next) {
  const { user } = req;

  // 生成权限WHERE子句
  if (user.role === 'org_user') {
    req.permissionFilter = `org_level_3 LIKE '%${user.organization}%'`;
  } else {
    req.permissionFilter = '1=1';
  }

  next();
}

// ✅ 查询执行（自动注入权限过滤）
const sql = generateKpiQuery(
  combineWhereClause(
    req.body.filters,        // 用户筛选条件
    req.permissionFilter     // 权限过滤（后端强制）
  )
);
```

**安全保障**:
1. ✅ JWT签名验证，无法伪造Token
2. ✅ 权限过滤在后端执行，前端无法绕过
3. ✅ 数据库连接在后端，前端无法直接访问

## 6. 云服务器方案评估

### 6.1 配置需求分析

**数据量估算**:
- Parquet文件大小: 约 50-100MB（基于现有测试数据）
- DuckDB内存需求: 文件大小 × 2-3倍（约 150-300MB）
- Node.js进程: 约 200MB
- Nginx: 约 50MB
- **总计**: 约 500-700MB 内存

**并发查询**:
- 预计同时在线用户: 5-10人
- 每用户平均查询: 2-3个并发请求
- 峰值并发查询: 约 15-30个

**推荐配置**:
- CPU: 2核（足够处理DuckDB查询）
- 内存: 4GB（留有2-3倍余量）
- 磁盘: 40GB SSD（系统 + 数据 + 日志）
- 带宽: 3-5 Mbps（内网使用，带宽需求不高）

## 11. 备选方案

### 11.1 方案B: 混合模式

**描述**: 保留前端DuckDB-WASM，增加后端API用于权限验证

```
前端数据加载:
  ↓
后端验证权限 → 返回数据访问Token
  ↓
前端DuckDB-WASM加载 → 使用Token解密数据
```

**优点**:
- ✅ 开发量更小（前端改动少）
- ✅ 离线使用（数据缓存在浏览器）

**缺点**:
- ❌ 安全性较弱（数据仍在前端）
- ❌ 无法阻止技术用户破解Token

**评估**: ❌ **不推荐**，无法满足"数据权限隔离"核心目标

### 11.2 方案C: 桌面应用（Electron）

**描述**: 使用Electron打包为桌面应用，数据存储在本地

**优点**:
- ✅ 无需云服务器
- ✅ 离线使用

**缺点**:
