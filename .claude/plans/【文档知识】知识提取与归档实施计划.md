## 🎯 Phase 1: 代码探索 (已完成 ✅)

### Agent 1: 核心代码模块分析

**关键发现**:

#### Arrow IPC vs JSON的技术决策
**位置**: `src/shared/duckdb/worker.ts:76-77`
- 性能: DuckDB直接返回Arrow Table,JSON序列化会造成巨大开销
- 保真度: Arrow保持原始类型,避免DECIMAL→Number精度损失
- 传输: 二进制格式压缩率比JSON高70%+
- 内存: 流式IPC支持大结果集分块传输

#### Worker架构的必要性
**位置**: `src/shared/duckdb/client.ts:30`
- 浏览器兼容: DuckDB-WASM需在独立线程运行,避免阻塞UI
- 沙箱安全: Worker运行在独立沙箱,限制DOM访问
- 并发控制: 支持多查询并行(`batchId`机制)
- 请求取消: `requestId`实现UI级别查询取消

#### PolicyFact视图去重逻辑
**位置**: `src/shared/duckdb/client.ts:136-156`
- 业务需求: 同一保单多条记录(批改/续保),需按保单号取最大保费
- 数据完整性: `MAX(premium)` + `FIRST()`确保保费准确且关联字段不丢失
- 可选字段: 动态处理`tonnage_segment`等可选字段存在性
- 视图分离: `PolicyFact`和`PolicyFactRenewal`两个视图应对不同分析维度

### Agent 2: Git历史分析

**关键发现**:

#### B041任务: 日期字段类型转换
- Parquet日期字段默认VARCHAR
- PolicyFact视图保留VARCHAR导致所有查询失败
- DuckDB日期函数必须接收DATE类型
- **教训**: 数据仓库视图必须确保字段类型正确,类型转换应该在数据加载时完成

#### B038任务: 代码质量审查
- 测试覆盖率<10%
- 122处`any`类型,32个文件涉及
- 203处`console.log`
- Bundle~2MB
- **教训**: 代码质量是可持续发展的基础,测试覆盖率必须达到70%+

#### B022任务: 营业货车图表合并
- 信息密度: 合并后展示更多维度
- 交互流畅: 点击直接下钻,无需切换
- 视觉统一: 颜色一致性更好
- 空间效率: 减少图表占用空间

#### B013任务: 技术栈感知协作体系
- 知识断层: 技术栈特性未文档化
- 重复错误: DuckDB日期函数、ISO周vs自然周反复出现
- 协作效率: 新协作者上手成本高

### Agent 3: 测试文件分析

**关键发现**:

#### 业务规则边界
- mapping.test.ts: 多别名支持边界(精确匹配优先、中英文混合、自定义配置)
- validator.test.ts: 类型容错(布尔字段支持BOOLEAN/INTEGER/VARCHAR、日期VARCHAR可解析)

#### 测试用例背后的业务规则
- 续保单号条件非空: 空字符串和NULL都视为无效续保
- 负保费处理: 正保费(签单收入)、零保费(未生效/退保)、负保费(退费)
- 商车自主定价系数NULL: NULL表示未使用,计算时排除或单独统计
- 批改类型枚举: 保单生命周期重要标记,影响保费计算

## 📋 Phase 2: 知识分类 (已完成 ✅)

### 6大知识体系分类

#### 1️⃣ 业务规则类 (15条)
**数据口径规则**:
- 空值有业务含义(续保单号空=非续保,不是缺失)
- 负保费=批改退费,必须伴随批改记录
- 实收保费=SUM(保费)(正负抵消)
- 毛保费=SUM(ABS(保费))(不考虑退费)

**字段关联规则**:
- 是否续保 ↔ 续保单号非空 (100%一致)
- 险类=商业保险 ↔ 商车自主定价系数非空
- 批单号非空 ↔ 批改类型非空
- 终端来源=0110融合销售 ↔ 是否电销=True

#### 2️⃣ 技术约束类 (18条)
**DuckDB-WASM强制要求**:
- COOP/COEP头必须配置
- 日期字段必须CAST AS DATE
- 日期函数只接收DATE类型参数

**Arrow IPC协议**:
- 禁止JSON序列化(性能/保真度/传输效率)
- 使用'stream'格式支持大结果集
- QueryResult类型为Uint8Array

**SQL安全约束**:
- 只读强制、聚合强制、视图边界、隐私保护

#### 3️⃣ 开发规范类 (10条)
**三层验证协议**:
1. bun test (单元测试)
2. Chrome DevTools (浏览器实测)
3. 用户验收 (功能验证)

**护栏机制**:
- 业务口径护栏: 不得删除已有别名
- 架构协议护栏: 禁止JSON、禁止删除CORS头、禁止npm/yarn
- 代码质量护栏: 治理校验必须通过

#### 4️⃣ 历史决策类 (8条)
**技术选型决策**:
- DuckDB-WASM: 浏览器内执行、Parquet原生支持、SQL分析能力、性能
- Monaco编辑器: VS Code同源、专业SQL支持、安全控制、可扩展性
- 参数化查询: 安全性、灵活性、易用性、可维护性

#### 5️⃣ 例外情况类 (6条)
**边界条件**:
- 续保单号: 空字符串和NULL都视为无效
- 保费异常: 极大值/极小值处理
- 日期未来值: 大于当前日期的处理

#### 6️⃣ 最佳实践类 (12条)
**数据准备**:
- 类型转换应该在数据加载时完成
- 视图必须确保字段类型正确

**代码质量**:
- 测试覆盖率必须达到70%+
- 类型安全不能妥协(any类型最小化)

**协作规范**:
- 技术栈特性必须文档化
- 三大索引是快速接力的关键
- 实际验证比理论验证更重要
