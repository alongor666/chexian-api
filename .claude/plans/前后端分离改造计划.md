# 前后端分离改造计划

## 概述

将车险业绩分析系统从纯前端架构（DuckDB-WASM）改造为前后端分离架构，实现真正的数据权限隔离。

**改造目标**：
- 后端控制数据访问，按用户权限返回数据（行级安全）
- 部署到云服务器，同事随时可用
- 保持现有 UI 和功能不变

## 技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| 后端框架 | Node.js + Express + TypeScript | 复用现有 12 个 SQL 生成器 |
| 数据库 | DuckDB (服务端版本) | 100% SQL 兼容，零重写 |
| 部署环境 | 云服务器（阿里云/腾讯云） | 2核4G，约 50-80元/月 |
| 安全 | IP 白名单 + HTTPS + JWT | 仅公司 IP 可访问 |

## 权限隔离机制

```
用户登录（乐山机构）
    ↓
后端验证 → 返回 JWT Token
    ↓
用户请求数据
    ↓
后端自动注入 WHERE org_level_3 LIKE '%乐山%'
    ↓
只返回该机构数据 ✅ 其他机构数据不可见
```

## 架构简化路线图（双模式策略）

> **问题背景**：当前系统采用 API/Local 双模式架构，2026-02-04 排查"仪表盘无数据"问题时，发现三层状态不同步是复杂性根源。

### 双模式定位

| 模式 | 用途 | 长期策略 |
|------|------|----------|
| **API 模式** | 生产环境：多用户协作、权限过滤、数据安全 | ✅ 主模式 |
| **Local 模式** | 开发调试、离线演示、无后端环境降级 | 保留为降级方案 |

### 当前状态（2026-02-04）

- **API 模式** (`isApiMode = true`)：后端 DuckDB 执行查询，JWT 认证，行级安全
- **Local 模式** (`isApiMode = false`)：前端 DuckDB-WASM，匿名访问

### 状态管理关键点

⚠️ **三层状态必须理解**（详见 CLAUDE.md §1.5）：
1. `DataContext.isDataLoaded` - 后端有"当前"文件
2. `DataContext.isApiMode` - 是否使用 API 模式
3. `duckdbClient.isDataLoaded()` - 前端 DuckDB 已加载数据

**正确判断公式**：`isDataEnabled = isApiMode || isLocalInitialized`

### 未来简化方向（Phase 5 候选）

如需简化架构，可考虑：
1. **移除 Local 模式**：所有数据操作必须通过后端 API
2. **统一状态管理**：合并三层状态为单一 `dataReady` 标志
3. **代价**：失去离线能力、开发调试需启动后端

**当前决策**：保留双模式，待 Phase 4 部署稳定后再评估简化。

## 已确认问题（2026-02-04）

| 问题 | 答案 | 影响 |
|------|------|------|
| **远程办公支持** | 不用 VPN，仅公司内网访问 | IP 白名单配置简化 |
| **数据更新方式** | 用户手动上传到服务器 | 需保留上传 API |
| **并发用户数** | 最多 100 人 | 2核4G 服务器需评估是否足够 |

### 服务器配置建议

100 并发用户对 2核4G 服务器的压力评估：
- DuckDB 单查询内存：约 50-200MB
- 并发查询峰值：预估 10-20 个同时查询
- **建议**：2核4G 基本满足，但建议监控内存使用，必要时升级到 4核8G
