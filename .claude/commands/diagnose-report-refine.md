# 诊断报告结论精炼器（/diagnose-report-refine）

> 输入一份**已生成的诊断报告草案**，替代人眼找出问题/规律/异常点，用精准且符合数据事实、符合行业通用表达的语言，产出「全局概览 + 各表隐性结论」的**正式版**。草案只是输出草稿，正式版经过「替代人眼」的提炼。
>
> **分工铁律**：数字由 `refine_verify.py` 回查保真，判断由本文档方法论保质。**禁止肉眼读 md 表格数字**——一律引用事实包，杜绝看错。

## 适合场景

- 「把这份续保报告的结论**精炼**一下 / 做成**正式版**」
- 「**替代人眼**看这份报告，找出背后的问题、规律、异常点」
- 跑完 `diagnose-renewal --org-report/--branch-report` 后，把规则化草稿（「问题一/问题二」）提炼为管理层可直接读的判断

**不适合**：生成报告本身（用 `diagnose-renewal` 等生成器）；出险率/费用率 5Why 解读（用 `rewrite-conclusion`，本技能续保 adapter 之外的域尚未接入）。

## 调用方式

```bash
# 用户视角
/diagnose-report-refine --report <报告.md路径>
/diagnose-report-refine --report <报告.md> --context <背景.md>   # 可选：注入人/机构/事件背景
/diagnose-report-refine --report <报告.md> --overwrite          # 可选：覆盖草案（默认另存正式版）
```

## 执行流程（AI 必须按序执行）

### Step 1 — 取事实包（确定性，禁止跳过）

```bash
python3 数据管理/pipelines/refine_verify.py --report <报告.md>
```

输出 JSON 事实包：`meta`（域/视角/机构/年月/客户类别/截止日）+ `tables[]`（每表 total/funnel/rows/top_anomaly/gap/contrast/maturity）+ `verify`（回查结果）。

### Step 2 — 校验闸（硬边界）

- `verify.ok == false` → **立即中止**，向用户列出 `verify.details` 中 `match=false` 的差异，**不产出正式版**（报告数字与 Parquet 不符，不可据此下判断）。
- `verify.skipped` 存在（无 adapter 域/无数据）→ 继续，但正式版**显式标注「未回查」**。

### Step 3 — 按方法论写（见下方「方法论」「写法规范」）

- 概览（置顶）+ 逐表隐性结论
- **所有数字只引用事实包**（`total`/`rows`/`top_anomaly`/`gap`/`contrast`），不另算、不肉眼读表

### Step 4 — 过自检清单（见下方），不合格则重写

### Step 5 — Write 正式版

- 路径：`<原名>_正式版.md`（同目录；`--overwrite` 时覆盖原文件）
- 结构：**概览置顶** → 各表【原数据表逐字保留】+ 草案结论段落替换为「现象提问式」隐性结论
- frontmatter 标注 `source`（草案文件名）、`generated`、`method: diagnose-report-refine`

## 方法论（两层）

### 通用写作层（所有诊断报告通用）

1. **金字塔原理**：概览总判断先行，各表隐性结论承接
2. **定量后定性**：每条判断由数据导出
3. **数字优先 · 节约认知资源为最高准则**：精确数字优先（`续保率 46.2%`、`未报价 15 件`），**禁数字文字化**（「五成」「过半」「回到五成以上」）。模糊词（可能/或许/值得关注/一定程度上/反映出）与数字文字化都是认知噪声，一并禁。仅当文字确实比数字更省读者认知时才用文字。
4. **禁管理动作**：只呈现现象/规律/杠杆，不写「该派单给谁 / 怎么做 / 应当」。改进方向靠「对比落差」让其自现，不写建议句。
5. **不堆砌**：每条判断保留最有支撑力的 2-3 个数字。

### 续保领域口径层（adapter 红线，违反即低质量）

- **续保影响度是负面指标**（越高越坏）：措辞「拉大整体续保缺口」，**禁用「贡献」**（中文语境贡献为褒义）
- **已到期 = 最终留存**（`maturity=matured`，表一/表五）；**未到期 = 进度**（`maturity=progress`，临期/当月未到期/当月/首日/首周）——**绝不把进度口径当塌方**，措辞用「进度/尚未续回」而非「已流失」
- **临期 7 天 ≠ 已流失**（仍在续保动作窗口内）
- **已续回 = 已签单续保**（is_renewed），含起保日在未来的提前续保
- **可续期锚点 = 到期前 30 天**（四川规则）
- **对比落差口径**：续回**不是**报价的严格子集（实测约 0.16% 续回未经报价，`is_renewed ⊄ is_quoted`）。事实包 `contrast`：matured 表（`source=parquet_exact`）的 `quoted_renew_rate`/`unquoted_renew_rate` 是回查精确值，可直接引用；progress 表（`approx=true`、`unquoted_renew_rate=null`）的 `quoted_renew_rate` 仅为上界。**禁止把未报价路径续保率写成「0%」**，除非该表 `unquoted_renew_rate` 精确值确为 0；否则写「未报价路径几乎不成交」。
- **续保缺口必拆两段（应知应会·必写）**：任何续保缺口都由两段构成 —— ① **未报价 ＝ 没接触客户**（应续 − 已报价，续保率结构性为 0，零接触成本）；② **报价未成交 ＝ 接触了没转化**（已报价 − 已续保）。两者之和恒 ＝ 流失件数（口径见 `renewal_common.MATURED_GLOSSARY`，事实包 funnel 已给三个件数，相减即得，**非肉眼读表**）。每张 matured 表与概览**必须把缺口拆成这两段各报一次**，**两段并重、不分主次**呈现——**禁以件数多少判「主导段」**（曾误用「报价未成交件数 > 未报价件数」就判「短板在成交端、不在报价覆盖」、把未报价淡化为次要，是错的）。只报一个笼统续保率＝信息不全。
- **未报价是「最不该发生」的基础失守（性质铁律，2026-06-08 用户红线）**：报价 ＝ 接触客户，是续保漏斗的**必要前置动作**；每一单都应报价接触，**未报价 ＝ 零接触 ＝ 必然 100% 流失**（续保率结构性 0）。故无论件数多少，未报价段都是**性质最严重、最该正视**的一段，与「报价后未成交」**两段并重、不分主次**；**严禁**写「短板在成交转化、不在报价覆盖」一类淡化未报价的定调（违背「每一单都应报价接触」铁律）。
- **报价未成交 ＝ 候选原因清单，永远「待归因」（绝不武断）**：报价了却没成交，可能源于「理赔不满意 / 服务不满意 / 价格高于同业 / 跟进不到位 / 公司主动剔除」等。**当前 Parquet 只能拆出「未报价 vs 报价未成交」两个件数，无法区分具体是哪个原因**——故写报价未成交段时，只陈列**件数 + 把上述原因列为「待进一步归因（需理赔满意度 / 同业报价 / 跟进记录 / 核保剔除标记等数据，或 `--context` 注入）」**，禁止断言是某一个原因。这是「绝不武断归因」红线在续保域的具体化。

## 写法规范

### 现象提问式（每张表，核心）

陈列 **漏斗递进 → 缺口归因到异常点 → 对比落差锚点**，让事实自己说话。不出现「结论/问题」字样、不下显性判断动词。1-3 句，克制。落差锚点让改进杠杆自现，不加动作句。

> 示例（乐山表一·已到期 matured）：
> `78 件应续 → 63 件报价 → 36 件续回。15 件缺口来自未报价，刘洋新占 10 件。已报价续保率 57.1%，未报价 0%。`
> （此处「未报价 0%」用的是 matured 表 `contrast.unquoted_renew_rate` 的 `parquet_exact` 精确值；progress 表该值为 `null`，不可写具体百分比。）

> 进度表（progress）措辞区分（乐山表二·临期）：
> `未来 7 天 50 件到期，已续回 15 件（进度，仍在续保窗口内）。13 件尚未报价、距到期不足 7 天。`

### 概览（置顶）

- 一句**全局总判断**（跨表收敛的本质问题）——**两段并重、不分主次**地点出两道关口（未报价 ＝ 零接触必然流失、报价未成交 ＝ 体量更大待归因），**禁以件数判主次、禁写「不在报价覆盖」淡化未报价**
- 3-5 条**要点**（每条现象式，跨表规律/异常）——其中**一条专列缺口两段拆分**：未报价 X 件（没接触）＋ 报价未成交 Y 件（接触没转化，原因待归因）
- 一句**口径提醒**：已到期=最终留存 vs 未到期=进度

## 自检清单（产出前逐项过）

- [ ] 核心判断由数据决定、无凑数
- [ ] 无模糊词、无数字文字化（「五成」等）
- [ ] 无管理动作（不写「该怎么做」）
- [ ] 区分现象与根因（交叉去重：同一批对象的多个红项收敛为一个根因）
- [ ] 不把恒等式当发现（占比 >80% 的维度值不算发现）
- [ ] 续保口径红线全守（影响度=负面、已到期 vs 未到期、临期≠已流失）
- [ ] **续保缺口已拆「未报价（没接触）＋ 报价未成交（接触没转化）」两段，总判断两段并重、不分主次（未以件数判主次、未写「不在报价覆盖」淡化未报价）**
- [ ] **报价未成交的原因以「待归因候选」列出（理赔/服务/价格/跟进/主动剔除），未武断断言单一原因**
- [ ] **过 `python3 数据管理/pipelines/lint_renewal_judgment.py <正式版.md>` 判断红线闸（0 违反）**

> **自迭代积累回路（判断偏治理的核心机制）**：判断质量无机器真值（真值=用户认可/否决）。用户每否决一条判断 → 立即抽成 `数据管理/pipelines/judgment_redlines.json` 的一条 `forbid`（禁词）或 `must_contain`（必含），永不删历史规则 → 下次 lint 自动挡同型错。正反例三来源：①对话中用户否决（分析提炼）②低置信时主动询问用户后入库 ③用户直接输入。北极星指标 = **一次过率**（`verify.ok` + driver 17/17 + lint 0 违反 + 用户零否决）。阈值/58% 目标沿用项目既有规则、不改；数据有 `refine_verify` 即唯一事实源。
- [ ] 概览总判断一句话说清本质
- [ ] **所有数字来自事实包（经 Step 2 回查）**

## 预留输入接口（内容逐渐积累）

| 接口 | 作用 | 缺省行为 |
|---|---|---|
| `--context <md>` | 注入关键人/机构/事件背景（AI 编不出的数字外上下文，如「刘洋新责任田 6 月才划入」） | 缺省=仅基于数据；不确定处**标注「需人工确认背景」、不武断下判断** |
| `--style <md>` | 表达范式样例，学行业遣词造句 | 缺省=用本文档写法规范 |
| `--audience <角色>` | 读者视角（分公司总/机构负责人/团队主管）决定详略口吻 | 缺省=机构负责人 |

> 有 `--context` 就把背景纳入判断（如「报价缺位属交接期、非执行问题」）；没有就只基于数据、不确定处标注待确认，**绝不武断归因**。
