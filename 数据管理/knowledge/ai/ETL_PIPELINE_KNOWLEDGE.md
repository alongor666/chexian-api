
## 1. 3层分片架构

### 分片类型

| 层 | 边界 | 更新频率 | 说明 |
|----|------|----------|------|
| static | 签单日期 ≤ `static_cutoff`（2023-12-31） | 永不更新 | 已满期1年+的历史数据，一次性转换后不再重生成 |
| weekly | 签单日期 ≥ `weekly_start`（2024-01-01） | 每周重新转换 | 活跃期数据，含最新保单 |
| daily | 日增量 xlsx | 每日 | 暂存 staging/，周更时并入 weekly |

### 分片配置

文件：`数据管理/shard-config.json`

```json
{
  "static_cutoff": "2023-12-31",
  "weekly_start": "2024-01-01",
  "renewal_source_pattern": "续保业务类型匹配*.xlsx",
  "renewal_apply_to": "weekly"
}
```

### 分片保护机制

- static 分片：**parquet 文件存在则跳过**，不重新转换
- 需要重新转换 static 分片时：**必须手动删除旧 parquet 文件**
- weekly 分片：每次运行都重新转换（覆盖旧文件）

## 2. 字段变换规则（CRITICAL）

### 2.1 日期字段的"双重身份"

**这是最容易混淆的变换规则**：

| xlsx 源字段名 | → Parquet 列名 | → mapping.ts 域字段 | 业务含义 |
|---------------|----------------|---------------------|----------|
| **缴费日期** | **签单日期** | `policy_date` | 业绩统计日期（销售指标、月报用这个） |
| **签单日期**（原） | **提核日期** | `underwriting_date` | 审批/提核日期（风险管理用这个） |
| 保险起期 | 保险起期 | `insurance_start_date` | 保险责任开始日期 |

**为什么这样做**：业务上"缴费日期"才是实际出单日期（钱交了才算），而原始"签单日期"更接近提核/审批日期。transform.py 进行了重命名对齐，使前后端统一使用"签单日期"表示业绩日期。

### 2.2 字段重命名映射

| xlsx 源字段 | → Parquet 列名 | 变换逻辑 |
|-------------|----------------|----------|
| 签单/批改保费（含税） | 保费 | 直接重命名，保留正负值 |
| 险别 | 险别组合 | 直接重命名 |
| 案件数 | 赔案件数 | 转 int64，空值→0 |
| 赔款合计 | 已报告赔款 | 转 float64，空值→0 |
| 总费用金额 | 费用金额 | 转 float64，空值→0 |
| 代理人/经纪人 | 经代名 | 字符串保留原值 |
| 交叉销售标识-驾意 | 交叉销售标识 | 布尔转换（是→True） |
| 交叉销售保费-驾意 | 交叉销售保费_驾意 | 连字符→下划线（Parquet 列名不允许连字符） |

### 2.3 生成字段（源 xlsx 中不存在）

| 生成字段 | 生成逻辑 | 依赖源字段 |
|----------|----------|------------|
| 是否可续 | `NOT(批改类型 LIKE '%解除合同%' OR '%退保%')` | 批改类型 |
| 是否电销 | `终端来源 LIKE '0110%' OR '融合销售'` | 终端来源 |
| 是否续保（布尔） | `续保单号 IS NOT NULL` | 是否续保（实际存储续保单号） |
| 续保单号 | 从"是否续保"列提取（该列实际存保单号格式数据） | 是否续保（原始列） |
| 车险风险等级 | `COALESCE(车险分等级, 小货车评分, 大货车评分)` | 3个互斥评分字段 |
| 续保模式 | 从续保业务类型复制 | 续保业务类型 |

### 2.4 特殊处理

| 字段 | 特殊处理 | 原因 |
|------|----------|------|
| 车牌号码 | 只保留前2位（如"川A"） | 隐私保护 + 省份分析足够 |
| 车架号 | 全文保留（17位 VIN） | 车辆唯一标识，去重需要 |
| adminadmin | 按三级机构拆分为虚拟业务员 | 系统账号需分摊到各机构 |

### 3.3 DuckDB union_by_name 行为

三个分片 Parquet 通过 `read_parquet([...], union_by_name=true)` 自动合并：
- 01c（限摩）缺少的 14 个字段 → 自动填 `NULL`
- 01a（旧格式）缺少的 9 个新字段 → 自动填 `NULL`
- 不需要手动处理 schema 差异

## 4. 域分片策略

transform.py 的 `--domain` 参数控制输出哪些字段：

| 域 | 保留字段 | 后处理 | 输出路径 |
|----|----------|--------|----------|
| `policy` | core + optional，排除赔付/费用/报价 | 无 | `current/*.parquet` |
| `claims` | 保单号+车架号+赔案件数+已报告赔款+费用金额 | 按保单号聚合，只保留有赔付数据的行 | `claims/latest.parquet` |
| `quotes` | 续保单号+签单日期 | 只保留已报价+续保单号非空，按续保单号去重 | `quotes/latest.parquet` |
| `all` | 全部（core + optional） | 无 | 默认模式 |

## 5. "是否续保"字段的陷阱

**这是 xlsx 数据最大的坑**：

xlsx 中的"是否续保"列**实际存储的是续保单号**（保单号格式的字符串），而非布尔值。

transform.py 的处理逻辑：
1. 检测到"是否续保"列内容是保单号格式
2. 将其重命名为 `续保单号`
3. 重新生成布尔值 `是否续保` = `续保单号 IS NOT NULL`

## 6. Parquet 列名命名规则

| 规则 | 示例 |
|------|------|
| 禁止连字符 `-` | `交叉销售保费-驾意` → `交叉销售保费_驾意` |
| 使用中文列名 | Parquet 中统一用中文，mapping.ts 负责映射到英文域字段 |
| 布尔字段 | Python `bool` 类型，Parquet `BOOLEAN` |
| 日期字段 | Python `datetime64[us]`，Parquet `TIMESTAMP` |

## 7. ETL 命令速查

```bash
# 完整 ETL（自动检测 + 同步 VPS）
node 数据管理/daily.mjs

# 仅转换，不同步
node 数据管理/daily.mjs --no-sync

# 强制指定域
node 数据管理/daily.mjs claims    # 赔付域
node 数据管理/daily.mjs quotes    # 报价域
node 数据管理/daily.mjs all       # 全部

# 手动转换单个文件
python3 数据管理/pipelines/transform.py \
  -i "数据管理/每日数据_20200101_20231231.xlsx" \
  -o "数据管理/warehouse/fact/policy/current/output.parquet"

# 维度表生成
python3 数据管理/warehouse/dim/generate_dim_tables.py

# 同步到 VPS
node scripts/sync-vps.mjs
```

## 9. 新增字段清单（供未来 ETL 扩展参考）

当源 xlsx 新增字段时，需要修改以下文件：

1. `transform.py` — `process_new_fields()` 添加处理逻辑 + `finalize_schema()` 的 `optional_fields` 列表
2. `mapping.ts` — `DomainField` 类型 + `COLUMN_ALIASES` 别名 + `OPTIONAL_FIELDS` 集合 + `ColumnMapping` 接口 + `DEFAULT_MAPPING` 对象
3. `validator.ts` — `EXPECTED_TYPES` Record 添加类型约束
4. `PARQUET_SCHEMA_KNOWLEDGE.md` — 更新字段文档和自然语言映射
