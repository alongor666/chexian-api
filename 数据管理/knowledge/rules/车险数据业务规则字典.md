## 🎯 核心业务概念

### 1. 险类体系 (险类)

**定义**: 保险产品的分类维度,区分强制保险和商业保险

**值域**:

- `交强险` (76.2%): 机动车交通事故责任强制保险,国家强制购买
- `商业保险` (23.8%): 自愿购买的商业险种

**业务规则**:

- ✅ 一个保单可能同时包含交强险和商业险(通过 `险别组合`区分)
- ✅ 交强险保费固定(与车型无关),商业险保费可变
- ✅ `商车自主定价系数` 仅对商业险生效

### 2. 险别组合 (险别组合)

**定义**: 同一保单下包含的险别组合类型

**值域**:

- `单交` (54.3%): 仅交强险
- `交三` (23.0%): 交强险 + 第三者责任险
- `主全` (22.7%): 交强险 + 商业险全险种
- `其他` (0.0%): 其他组合

**业务规则**:

- ✅ `单交` = 仅交强险,无商业险
- ✅ `交三` = 基础组合,保额较低
- ✅ `主全` = 全险组合,包含车损险、三者险、盗抢险等
- ✅ 批改时 `险别组合`可能变更(如从 `主全`改为 `单交`)

### 3. 统保类型 (是否交商统保)

**定义**: 交强险和商业险是否在同一保单(统保优惠)

**值域**:

- `单交` (54.3%): 仅交强险
- `套单` (44.1%): 交强险 + 商业险一同在我司投保
- `单商` (1.6%): 仅商业险(交强险不在我司投保)
- `其他` (0.0%): 特殊情况

### 4. 客户类别体系 (客户类别)

**定义**: 按车辆使用性质和车主类型分类

**值域** (11个类别):

| 类别               | 占比  | 业务特征                  |
| ------------------ | ----- | ------------------------- |
| `非营业个人客车` | 58.8% | 个人自用轿车,SUV等        |
| `摩托车`         | 28.6% | 二轮/三轮摩托车           |
| `非营业货车`     | 4.8%  | 个人/企业自用货车         |
| `非营业企业客车` | 4.8%  | 公司公务车                |
| `营业货车`       | 2.9%  | **🎯 重点分析对象** |
| `营业出租租赁`   | 0.1%  | 出租车/网约车             |
| `特种车`         | 0.1%  | 工程车、消防车等          |
| `营业公路客运`   | 0.1%  | 长途大巴                  |
| `挂车`           | 0.0%  | 半挂车                    |

**业务规则**:

- ✅ **营业货车**是专项分析重点(对应 `吨位分段`字段)
- ✅ 营业性车辆保费高于非营业车辆(风险更高)
- ✅ 摩托车保费最低,单交占比高
- ✅ 特种车、挂车需要特殊核保规则

## 🔑 关键字段详解

### A. 主键字段

#### 保单号 (policy_no)

- **数据类型**: STRING
- **唯一性**: 100% (607,455个唯一值)
- **非空率**: 100%
- **格式**: `6102101030120240000857` (22位数字)
- **业务含义**: 保单唯一标识
- **业务规则**:
  - ✅ 同一保单号可能对应多条记录(交强险+商业险分开记录)
  - ✅ 通过 `保单号 + 险类`组合去重获取唯一保单

#### 续保单号 (renew_policy_no)

- **数据类型**: STRING
- **唯一性**: 221,002个唯一值
- **非空率**: 36.4% (221,002 / 607,455)
- **业务含义**: 上一年度保单号
- **业务规则**:
  - ✅ **空值 = 非续保**,非空值 = 续保
  - ✅ 通过 `是否续保`字段快速判断(True=有续保单号)
  - ✅ 续保率 = 36.4% (221,002 / 607,455)

#### 批单号 (endorsement_no)

- **数据类型**: STRING
- **唯一性**: 10,890个唯一值
- **非空率**: 1.8% (10,890 / 607,455)
- **格式**: `6102101030120240000857-001` (保单号-批单序号)
- **业务含义**: 批改业务凭证号
- **业务规则**:
  - ✅ **非空 = 该保单发生批改**(退保/过户/变更信息等)
  - ✅ 批改率 = 1.8%
  - ✅ 一个保单可能有多条批改记录(-001, -002, ...)

### B. 日期字段

#### 签单日期 (sign_date)

- **数据类型**: STRING (格式: YYYY-MM-DD)
- **唯一性**: 725个不同日期
- **非空率**: 100%
- **值域**: 2023-12-05 ~ 2026-01-08 (实际数据覆盖范围)
- **业务含义**: 保单签订日期
- **业务规则**:
  - ✅ 签单日期 <= 保险起期 (通常提前1-30天)
  - ✅ 签单日期用于统计**业绩**(按签单日归属)
  - ✅ 日期字符串需转换为 DATE 类型才能进行日期计算

#### 保险起期 (start_date)

- **数据类型**: STRING (格式: YYYY-MM-DD)
- **唯一性**: 817个不同日期
- **非空率**: 100%
- **值域**: 2023-12-29 ~ 2026-01-08
- **业务含义**: 保险责任开始时间
- **业务规则**:
  - ✅ 保险起期 - 签单日期 = 提前承保天数(通常1-30天)
  - ✅ 保险起期用于统计**业务量**(按生效日归属)
  - ✅ 续保判断: 当前保险起期 - 上一年度保险起期 ≈ 365天

### C. 金额字段

#### 保费 (premium)

- **数据类型**: FLOAT64
- **非空率**: 100%
- **值域**: -18,902.75 ~ 30,185.35
- **统计特征**:
  - 平均值: 822.98
  - 中位数: 672.59
  - 标准差: 1,119.86
- **业务含义**: 保险费(可能为负数)
- **⚠️ 关键业务规则**:
  - ✅ **负保费 = 批改退费**(退保/减费批改)
  - ✅ `保费 > 0` = 正常承保保费
  - ✅ `保费 < 0` = 批改退费金额
  - ✅ 统计总保费时需 **SUM(保费)** (正负抵消后为实收保费)
  - ✅ 统计毛保费时需 **SUM(ABS(保费))** (不考虑退费)

#### 新车购置价 (car_price)

- **数据类型**: FLOAT64
- **非空率**: 100%
- **值域**: 0 ~ 12,600,000
- **统计特征**:
  - 平均值: 137,238.59
  - 中位数: 75,900.00
  - 标准差: 249,339.55
- **业务含义**: 新车购买价格(车损险保额依据)
- **业务规则**:
  - ✅ 车损险保费与新车购置价正相关
  - ✅ 0值 = 摩托车/仅投保交强险(无车损险)
  - ✅ 高值车辆(>100万)通常需要特殊核保

#### 商车自主定价系数 (pricing_factor)

- **数据类型**: FLOAT64
- **非空率**: 23.8% (仅商业险有值)
- **值域**: 0.5 ~ 1.5
- **统计特征**:
  - 平均值: 0.91
  - 中位数: 0.85
  - 标准差: 0.18
- **业务含义**: 商业险自主定价系数(NCD系数)
- **业务规则**:
  - ✅ **仅商业险生效**,交强险该字段为空
  - ✅ 系数范围 [0.5, 1.5] (监管规定)
  - ✅ 系数 < 1.0 = 优惠(连续无赔款年份多)
  - ✅ 系数 > 1.0 = 上浮(出险次数多)
  - ✅ 系数 = 1.0 = 基准费率

- **数据类型**: BOOL

#### 赔案件数 (claim_cases)

- **数据类型**: INT64
- **非空率**: 96.1% (3.9%空值填充为0)
- **值域**: 0 ~ N（整数）
- **统计特征**:
  - 平均值: 0.09
  - 最大值: 需要进一步统计
- **业务含义**: 累计赔案件数量
- **业务规则**:
  - ✅ 用于计算**赔付率** = 已报告赔款 / 保费收入
  - ✅ 用于计算**变动成本率** = 已报告赔款 / 保费收入
  - ✅ 用于计算**综合成本率** = (已报告赔款 + 费用金额) / 保费收入
  - ✅ 用于计算**边际贡献率** = (保费收入 - 已报告赔款 - 费用金额) / 保费收入
  - ✅ 空值默认填充为0，表示无赔案

#### 已报告赔款 (reported_claims)

- **数据类型**: FLOAT64
- **非空率**: 96.1% (3.9%空值填充为0)

- **业务规则**:

  - ✅ 与 `保费`对比分析盈利能力
  - ✅ 结合 `赔案件数`计算平均案均赔款
  - ✅ 空值默认填充为0，表示无赔款支出

#### 费用金额 (fee_amount)

- **数据类型**: FLOAT64
- **非空率**: 100%
- **值域**: 0 ~ 高值（元）
- **统计特征**:

- **业务含义**: 累计费用金额（含各项手续费、服务费等）
- **业务规则**:

  - ✅ 用于计算**变动成本率** = 费用金额 / 保费收入
  - ✅ 用于计算**综合成本率** = (已报告赔款 + 费用金额) / 保费收入
  - ✅ 费用结构分析可优化成本控制

- **数据类型**: STRING

### 1. 续保识别逻辑

**关键点**:

- ✅ 续保率 = COUNT(IF(是否续保, 1, NULL)) / COUNT(*)

### 1.1 续保分析统计逻辑（重要业务知识）

在实际业务中，**续保分析存在两种不同的统计模式**，AI 和开发者必须理解。

#### 【核心公式】到期日计算规则

```
到期日 = 起保日 + 1年 - 1天
```

**⚠️ 这个 "-1天" 非常关键！** 它导致每个月的1日起保的保单实际在**上个月最后一天**到期：

| 起保日 | 到期日 | 实际到期月份 |
|--------|--------|-------------|
| 2025-02-01 | 2026-01-31 | **1月** |
| 2025-02-02 | 2026-02-01 | **2月** |
| 2025-03-01 | 2026-02-28 | **2月** |
| 2025-03-02 | 2026-03-01 | **3月** |

**闰年影响**：
- 2024-02-29 起保 → 2025-02-28 到期（闰年到非闰年）
- 2024-03-01 起保 → 2025-02-28 到期

**SQL 实现**：
```sql
-- 正确：按到期日月份筛选
MONTH(DATE_ADD(CAST(insurance_start_date AS DATE), INTERVAL '1 year') - INTERVAL '1 day') = 2

-- 错误：按起保月份筛选（会漏掉3月1日起保的保单）
MONTH(CAST(insurance_start_date AS DATE)) = 2
```

#### 模式一：按到期月份统计

| 项目 | 说明 |
|------|------|
| **触发条件** | 用户选择特定"到期月份"（如"2月到期"） |
| **应续件数** | 上一年该月起保的**所有**保单（不管当前是否已到期） |
| **已续件数** | 这些应续保单中，已经成功续保的数量（含提前续保） |
| **cutoffDate** | ❌ 不应用（要看完整月份的数据） |

**使用场景**：
- 📊 **当月到期追踪**：现在是1月，看1月到期保单的续保率
- ⚠️ **次月到期预警**：现在是1月，提前看2月到期保单有多少已提前续保
- 📈 **历史月份回顾**：回顾去年某月的最终续保率

```sql
-- 示例：查询2月到期的应续件数（不受cutoffDate限制）
SELECT COUNT(*) as due_count
FROM PolicyFactRenewal
WHERE YEAR(CAST(insurance_start_date AS DATE)) = 2025  -- 上一年
  AND MONTH(CAST(insurance_start_date AS DATE)) = 2    -- 2月起保 = 今年2月到期
```

#### 模式二：按到期日范围统计

| 项目 | 说明 |
|------|------|
| **触发条件** | 用户选择"全部月份" + 设置 cutoffDate |
| **应续件数** | 到期日在 [年初, cutoffDate] 范围内的保单 |
| **已续件数** | 这些应续保单中，已经成功续保的数量 |
| **cutoffDate** | ✅ 应用（限制到期日范围） |

**使用场景**：
- 📊 **年度累计续保率**：从年初到今天的整体续保情况
- 📅 **特定时间段统计**：某个日期范围内已到期保单的续保率

```sql
-- 示例：查询2026-01-01到2026-01-27到期的应续件数
SELECT COUNT(*) as due_count
FROM PolicyFactRenewal
WHERE YEAR(CAST(insurance_start_date AS DATE)) = 2025
  AND DATE_ADD(CAST(insurance_start_date AS DATE), INTERVAL '1 year') - INTERVAL '1 day'
      BETWEEN '2026-01-01' AND '2026-01-27'
```

#### 两种模式的关系

```
用户操作                    SQL 生成逻辑
─────────────────────────────────────────────────────────
选择"2月到期"          →    按月份筛选，忽略 cutoffDate
选择"全部月份"         →    应用 cutoffDate 的日期范围
─────────────────────────────────────────────────────────
```

**⚠️ 关键原则**：这两种模式**互斥**，不会同时生效。代码中通过 `if (cutoffDate && !dimension.dueMonth)` 实现。

**📍 代码位置**：`src/shared/sql/renewal-drilldown.ts`

### 4. 批改退费识别

```sql
-- 批改退费统计
SELECT
    批改类型,
    SUM(premium) AS refund_amount  -- 注意:premium为负数
FROM data
WHERE 批单号 IS NOT NULL
  AND premium < 0  -- 退费记录
GROUP BY 批改类型
```

**关键点**:

- ✅ `保费 < 0` = 退费金额
- ✅ `批单号 IS NOT NULL` = 存在批改
- ✅ 退保批改(16退保)会导致负保费

### 5. 实收保费统计

```sql
-- 实收保费(正负抵消)
SELECT
    SUM(premium) AS net_premium  -- 承保保费 - 退费金额
FROM data

-- 毛保费(不考虑退费)
SELECT
    SUM(ABS(premium)) AS gross_premium  -- 所有记录绝对值求和
FROM data
```

**关键点**:

- ✅ 实收保费 = SUM(保费) (正负相抵)
- ✅ 毛保费 = SUM(ABS(保费)) (不考虑退费)

## 🚨 开发注意事项

### 1. 日期处理

⚠️ **问题**: 日期字段存储为字符串,无法直接计算

✅ **解决方案**:

```sql
-- DuckDB转换
SELECT
    CAST(sign_date AS DATE) AS sign_date_parsed,
    CAST(start_date AS DATE) AS start_date_parsed,
    DATEDIFF('day', CAST(sign_date AS DATE), CAST(start_date AS DATE)) AS advance_days
FROM data
```

### 2. 布尔型字段

⚠️ **问题**: Python pandas读取Parquet后,布尔字段自动映射为bool类型

✅ **解决方案**:

```python
# 正确判断
df[df['是否续保'] == True]  # ✅
df[df['是否续保']]  # ✅ (布尔索引)

# 错误判断
df[df['是否续保'] == 'True']  # ❌ (类型错误)
```

### 3. 字符型字段

⚠️ **问题**: 分类字段存储为字符串,需精确匹配

✅ **解决方案**:

```sql
-- 正确查询
SELECT * FROM data
WHERE 客户类别 = '营业货车'  -- ✅ 字符串比较

-- 错误查询
SELECT * FROM data
WHERE 客户类别 = 营业货车  -- ❌ 未加引号
```

### 4. 负保费处理

⚠️ **问题**: 批改退费导致负保费,影响统计

✅ **解决方案**:

```sql
-- 统计实收保费
SELECT SUM(premium) FROM data  -- ✅ 正负抵消

-- 统计承保保费(不含退费)
SELECT SUM(premium)
FROM data
WHERE premium > 0  -- ✅ 仅统计正保费

-- 统计退费金额
SELECT SUM(ABS(premium))
FROM data
WHERE premium < 0  -- ✅ 仅统计负保费
```

### 5. 去重逻辑

⚠️ **问题**: 同一保单可能有多条记录(交强险+商业险)

✅ **解决方案**:

```sql
-- 按保单号+险类去重
SELECT DISTINCT
    policy_no,
    险类
FROM data

-- 获取唯一保单数
SELECT COUNT(DISTINCT policy_no) AS unique_policies
FROM data
```
