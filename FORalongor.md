## 一、这个项目是什么？

一句话：**一个跑在浏览器里的保险数据分析实验室。**

想象你有一堆保险业务数据（CSV、Parquet），你想知道今季度哪个分公司保费增长最快、哪类商车赔付率最高、续保率是不是在下跌。传统方案是把数据上传到服务器，服务器跑 SQL，返回结果。

这个项目做了一件疯狂的事：**把整个数据库引擎塞进了浏览器里**。你拖拽上传数据文件，浏览器本地就能执行任意 SQL 查询，直接渲染图表。不需要服务器，不需要云，不需要 DBA。

就像把一台冰箱从厨房搬到了客厅——数据在你手边，想用随时用，不用跑去便利店。

---

## 二、为什么要造这个？

这个项目的技术决策从一个痛点开始：**数据不能离开本地**。

保险公司的业务数据涉及客户信息，上传到公有云天然带来隐私风险。而传统的"上传→服务器处理→返回结果"方案还有另外两个问题：网络延迟让分析变慢，服务器维护费用又不便宜。

解决方案是 **DuckDB-WASM**。DuckDB 本身是一个嵌入式分析数据库（你可以把它想象成 SQLite 的分析版哥哥），而 WASM 技术让它能在浏览器里跑。这意味着数据从不离开用户的机器，SQL 在本地执行，结果即时可用。

然后就问了第二个问题：**Worker 线程里的数据怎么高效地传回主线程？**

答案是 **Apache Arrow IPC**（进程间通信格式）。Arrow 是列式存储格式，序列化速度比 JSON 快几个数量级。把数据从 Worker 传回主线程，用 Arrow 就像寄精包装的快递——每个字段紧紧打包，无冗余，极其高效。而 JSON 更像散装货物，每个字段都要贴标签、加引号，体积膨胀了几倍。

**核心技术决策总结：**

| 决策 | 选了什么 | 为什么不选另一个 |
|------|---------|----------------|
| 数据库 | DuckDB-WASM | 不用服务器，数据不出本机 |
| 线程通信 | Arrow IPC | 比 JSON 快 N 倍，适合列式数据 |
| 前端框架 | React 19 | 组件化 UI，状态管理成熟 |
| 构建工具 | Vite | 开发体验快，ESM 原生 |
| 图表库 | ECharts | 功能丰富，自定义能力强 |

## 三、代码结构漫游——四层楼

整个项目的代码有一个清晰的四层架构。你可以把它想象成一栋大楼：

### 地下室：Core（基础层）

最底层的基础设施。事实上这层目前很薄，因为项目把大量通用逻辑放在了 Shared 层。

### 大厅：Shared（共享层）— `src/shared/`

**这是整个项目的心脏。** 所有业务模块都依赖它，它提供了这个项目跑起来所需的一切基础能力：

- **`duckdb/`** — DuckDB 客户端和 Worker 通信的核心。`client.ts`（578 行）是主线程的"大脑"，负责初始化数据库、发起查询、解码 Arrow 数据；`worker.ts`（105 行）是跑在后台的"副厨"，专门执行 SQL、把结果序列化成 Arrow 格式。
- **`sql/`** — 12 个 SQL 模板生成器，共 7,262 行。这里是所有分析逻辑的核心。KPI 计算、趋势分析、增长率对比、成本分析——每个板块的 SQL 都在这里。可以把每个 `.ts` 文件想象成一份"菜谱"，需要这个板块的数据就按菜谱做一顿。
- **`normalize/`** — 数据标准化。`mapping.ts` 定义了列名别名规则（支持中英文多别名），`validator.ts` 做数据质量校验。
- **`types/`** — TypeScript 全局类型定义，12 个类型文件覆盖了 Worker 协议、过滤器状态、数据模型等。
- **`ui/`** — 基础 UI 组件（Card、Table、Button 等），所有业务组件都基于这些。
- **`utils/`** — 工具函数，包括数据格式化（件数/均值/百分比/保费/系数）、SQL 安全校验、预警检测等。
- **`hooks/` & `contexts/`** — React 自定义 Hook 和全局 Context，数据加载、筛选状态、权限控制都在这里管理。

### 办公室：Features（业务层）— `src/features/`

13 个独立的业务功能模块，每个模块自成一派，但都依赖 Shared 层的基础设施。你可以把它们想象成大楼里不同的办公室，每个办公室有自己的业务，但共享大楼的电梯、暖气和网络：

- **`dashboard/`** — 主仪表盘，KPI 卡片 + 趋势图 + 营业货车分析
- **`growth/`** — 增长率分析，同比/环比/年累计对比
- **`cost/`** — 成本分析四子板块（赔付率/费用率/综合率/变动分析）
- **`coefficient/`** — 商车自主定价系数监控
- **`sql-query/`** — 交互式 SQL 查询页面，带 17 个预置模板
- **`premium-report/`** — 保费报表
- **`marketing-report/`** — 营销战报
- **`filters/`** — 全局筛选面板
- **`home/`** — 数据导入入口（拖拽上传）
- 还有认证、设置、查询助理等

### 展示厅：Widgets（组件层）— `src/widgets/`

可复用的高级 UI 组件，比如 KPI 卡片（含目标进度环）、高级表格、ECharts 图表包装器等。这些组件在不同的 Features 模块里反复使用，是 UI 层面的"积木"。

### 快速定位手册

| 你想找什么 | 去哪里看 |
|-----------|---------|
| SQL 查询逻辑 | `src/shared/sql/*.ts` |
| 数据库初始化/查询执行 | `src/shared/duckdb/client.ts` |
| Worker 后台处理 | `src/shared/duckdb/worker.ts` |
| 列名映射/别名 | `src/shared/normalize/mapping.ts` |
| 数据格式化规范 | `src/shared/utils/formatters.ts` |
| 某个板块的展示逻辑 | `src/features/{板块名}/` |
| 通用按钮/卡片/表格 | `src/shared/ui/` 或 `src/widgets/` |
| 全局样式定义 | `src/shared/styles/` |

## 四、数据是怎么流动的？

从你拖拽上传一个文件到屏幕上出现图表，数据经历了一段有趣的旅程。

### 第一站：拖拽上传

你把 Parquet 或 CSV 文件拖到首页（`src/features/home/DataImportPage.tsx`），浏览器接住了它。

### 第二站：验"身份证"

数据进门之前要过安检——**列名验证与标准化**。你的数据可能叫 `保单号`，也可能叫 `policy_no`，也可能叫 `单位编号`。`normalize/mapping.ts` 里有一张大约翻译表，把各种各样的名字统一成系统内部的标准名。这一步非常重要：映射表里的别名**只能追加新的，绝不能删除已有的**，因为删了就会有旧数据无法识别。

### 第三站：洗身去重

数据被装入数据库后，系统会自动创建一个 **PolicyFact 视图**。这个视图做一件关键的事：如果同一个保单号有多条记录，它会用 `MAX(premium) GROUP BY policy_no` 保留最高保费的那条。这是业务口径决定的去重规则，不能随意修改。

### 第四站：SQL 查询

你要看 KPI 仪表盘？系统从 `sql/kpi.ts` 里取"菜谱"，生成对应的 SQL。你要看增长率？`sql/growth.ts` 来。每个分析板块都有自己的 SQL 模板生成器，它们根据你的筛选条件（时间范围、机构、险种等）动态拼接 SQL，然后发送给 Worker 线程执行。

### 第五站：快速传送带（Arrow IPC）

SQL 在 Worker 线程里执行完毕，结果需要传回主线程渲染。这里是 **Arrow IPC** 出场的地方。Worker 把查询结果序列化成 Arrow 的列式格式，主线程接收后反序列化。整个过程高效得像传送带运送精密零件，不浪费任何字节。（⚠️ 架构协议禁止把这里换成 JSON，那就像用散装袋装精密机械。）

### 第六站：UI 渲染

数据到达主线程后，各个 Features 模块拿到自己需要的数据，通过 Widgets 层的组件渲染成图表、表格和 KPI 卡片。ECharts 负责画折线图和柱状图，react-window 保证大数据表格不会卡顿（虚拟化渲染只渲染屏幕上能看到的行）。

**数据流程图（文字版）：**

```
文件上传 → 列名标准化 → PolicyFact去重 → SQL生成 → Worker执行 → Arrow IPC传输 → UI渲染
   ↑          ↑              ↑               ↑           ↑              ↑            ↑
  入口      mapping.ts    client.ts       sql/*.ts    worker.ts      client.ts    features/
```

## 五、为什么这样设计？深入技术决策

### Arrow IPC vs JSON：精包装 vs 散装

当你在 Worker 里得到一个查询结果，比如 10,000 行 × 5 列的数据，你得把它传回主线程。

如果用 JSON，每一个数字都要变成字符串，每一个字段都要加引号和逗号，你原本 80KB 的数据可能膨胀到 400KB，还要解析这些字符串才能变回数字。

Arrow 不是这样的。它是列式存储：同一列的所有数据连续排列在内存里，类型统一，没有冗余标签。序列化就是"直接把内存块打包寄过去"，反序列化就是"拆开放回原地"。速度提升不是一点点，是数量级的提升。

### Worker 线程：厨房里的副厨

SQL 查询可能很慢（扫描几十万行数据），如果放在主线程里执行，整个浏览器标签页会卡住，用户连光标都移不了。

Worker 线程就是厨房里请的一个副厨——主厨（主线程）负责和客人打招呼、端菜上桌（UI 渲染），副厨在后厨默默炒菜（跑 SQL）。主厨把菜单给副厨，副厨把做好的菜传出来，两个人互不打扰。

### 四层护栏：为什么有些东西不能改

你可能会发现，某些文件上面贴着"禁止删除"的标签。比如 `mapping.ts` 里的列名别名，比如 `PolicyFact` 视图的去重逻辑。

这些是**业务口径**，是保险业务团队经过讨论确定下来的数据处理规则。改了它们，所有依赖这些规则的分析结果都会变得不可信。就像餐厅菜单上写的"本店招牌菜配方不得修改"——你可以加新菜，但招牌菜的味道不能变。

## 六、踩过哪些坑？血泪教训

这一部分是真正最值钱的东西。每一个坑都是"踩了一脚、痛了三天"式的教训。

### 坑 #1：CORS 头缺失，DuckDB 白屏

**故事：** 项目第一次在浏览器里跑 DuckDB-WASM 的时候，页面一片空白。控制台没有明显报错，调试了半天。

**根本原因：** DuckDB-WASM 使用 `SharedArrayBuffer`，而 `SharedArrayBuffer` 在现代浏览器里必须配合 **COOP + COEP** 安全头才能启用。缺了这两个头，浏览器会无声地拒绝，WASM 模块根本装不进去。

**怎么修的：** 在 `vite.config.ts` 里加上：
```
Cross-Origin-Opener-Policy: isolate-origin
Cross-Origin-Embedder-Policy: require-corp
```

**怎么避免再踩：** 任何引入 WASM + SharedArrayBuffer 的库，第一件事就问："需要什么安全头？" 这不是代码 bug，是浏览器安全策略的门槛。

### 坑 #2：日期计算炸了，DuckDB 和 PostgreSQL 不一样

**故事：** 开发者之前习惯用 PostgreSQL 的 `DATE_TRUNC('month', date)` 来按月汇总数据。到了 DuckDB，一样的语法，结果却不对。

**根本原因：** DuckDB 和 PostgreSQL 的时间函数虽然名字相似，但边界行为有微妙差异。特别是在"月末"和"年末"的处理上，两者可能返回不同的截断结果。

**怎么避免再踩：** DuckDB 不是 PostgreSQL 的平移。每次写时间相关的 SQL，**必须拿样本数据验证边界值**（月初、月末、年初、年末、闰年二月）。不要靠记忆，靠测试。

### 坑 #3：SQL 注入漏洞

**故事：** 早期的交互式查询功能把用户在文本框里输入的内容直接拼进 SQL 字符串。表面上看没问题——用户嘛，怎么会恶意输入呢？

**根本原因：** "用户不会恶意输入"这个假设是错误的。即使是善意用户也可能意外输入特殊字符（引号、分号），导致 SQL 语法崩溃。恶意用户更可能借此删改数据。

**怎么修的：** 引入 `sql-validator.ts`，所有用户输入的 SQL 必须经过安全校验：**只允许只读查询（SELECT）**、**禁止 DROP/DELETE/UPDATE**、**必须是聚合查询**。用参数化方式处理用户输入。

**怎么避免再踩：** 所有面向用户输入的路径，默认假设"输入有害"。这是安全开发的第一准则。

### 坑 #4：79 个 `any` 类型的技术债

**故事：** 项目初期为了快速开发，很多地方图省事用了 `any`。某天一个类型变更传播了一圈，十几个文件同时报错，排查耗费了大量时间。

**根本原因：** `any` 是 TypeScript 的"请勿打扰"牌——类型检查器在这里直接放弃了。当你用 `any` 的地方发生变化时，编译器不会提醒你，你只能靠运行时报错来发现问题。

**怎么避免再踩：** 类型安全就像保险单——平时看起来一团废纸，真出事了是你的救命稻草。项目现在严格限制 `any` 的使用，新代码必须有明确类型定义。

### 坑 #5：没在浏览器测就提交，反复返工

**故事：** 某个功能在本地 Node 环境测试完美，提交后在实际浏览器里运行却崩溃。原因是 WASM 模块在 Node 和浏览器里的行为不同。

**根本原因：** DuckDB-WASM 是浏览器级别的技术。Node 模拟环境和真正的浏览器执行环境有本质区别，特别是在内存管理和线程模型上。

**怎么避免再踩：** 这是**三层验证**协议诞生的原因：① 单元测试 → ② 浏览器实测 → ③ 端到端验收。不准跳级，必须依次通过。

## 七、新技术怎么学？入门路径

### DuckDB-WASM 入门

1. **先理解 DuckDB 本身**：它是什么？为什么要在浏览器里跑数据库？读一遍 DuckDB 官网的 "Why DuckDB" 就好。
2. **看 WASM 版本的特殊要求**：COOP/COEP 安全头、SharedArrayBuffer、Worker 线程——这些是浏览器环境带来的"额外票务要求"。
3. **跑第一个查询**：上传一个 CSV，写一句 `SELECT * FROM data LIMIT 10`，看看结果。
4. **理解查询模板模式**：看看 `src/shared/sql/kpi.ts`，了解怎么动态生成 SQL。

### Apache Arrow 核心概念

Arrow 就一个核心理念：**列式存储**。传统表格是"行行行行"，Arrow 是"把同一列的所有值放在一起"。好处是：压缩好、序列化快、跨语言传递零拷贝。你不需要深入 Arrow 的所有规格，只需要理解"列式 = 高效"就够了。

### Web Worker 别害怕

Worker 就是一个独立的 JavaScript 线程。用 `new Worker('worker.ts')` 创建，通过 `postMessage` 和 `onmessage` 对话。就像两个人用对讲机说话——发消息、收消息、各自干活。

核心要点：Worker 里没有 `document`、没有 `window`，它只能跑纯计算。但正好，SQL 查询就是纯计算！

## 八、好工程师在这里怎么工作？

### DC-001 协议：开工前的三要素

在这个项目里，**不完整的信息不开工**。开始任何数据分析任务之前，必须明确三件事：

1. **分析年度**：你要分析的是哪个时间段的数据？
2. **数据口径**：保费是写照口径还是续保口径？哪种保险类型？
3. **时间段**：具体的起止日期是什么？

这三个要素是"开工令"。缺了哪一个，开工就停。这不是吹毛求疵，这是避免"做了半天发现方向错了"的pragmatic智慧。

### 缺口清单：信息不完整就登记

遇到信息缺口（比如文档没说清楚、字段含义模糊、需求矛盾）？**立即登记到 `开发文档/缺口清单.md`**，当前任务标记为 BLOCKED。

这不是拖延，这是专业。模糊信息下做出来的东西，返工概率比停下来确认高几个数量级。

### 三层验证：不准自我安慰

代码写完之后，不准自己说"看起来对就行"：

1. **单元测试**：机器帮你验证逻辑
2. **浏览器实测**：亲手在浏览器里跑一遍
3. **端到端验收**：模拟真实用户场景，跑完整流程

三层都过，才算交付。缺一层，交出去的东西不过数。

### 治理体系：规矩是为了让每个人都能快速上手

这个项目有三大索引和两本账：

- **三大索引**：`shared/INDEX.md`、`features/INDEX.md`、`widgets/INDEX.md` — 模块注册表，每个模块的功能、入口、依赖都记在里面。
- **两本账**：`BACKLOG.md`（需求账本）和 `PROGRESS.md`（进展账本）— 什么要做、什么在做、什么做完了，一清二楚。

有了这些，新人加入第一天就能知道"我要做的东西在哪里、谁依赖我、相关规范是什么"。
