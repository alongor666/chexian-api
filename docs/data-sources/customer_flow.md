# 客户来源去向 (customer_flow) — 数据源说明

> 最后更新: 2026-05-15 · 治理批次 #1

## 1. 业务定义

- **previous_insurer**（上年承保主体）：保单上一年由哪家保险公司承保
- **next_insurer**（次年保险公司）：保单到期后客户去了哪家保险公司
- 一行记录 = 一张华安保单 + 它的「来源」和「去向」

## 2. 数据特性（重要）

| 指标 | 数值 | 说明 |
|---|---:|---|
| 整体行数 | ~1,040,000 | 覆盖 2025-01-01 ~ T-1 全量保单 |
| `previous_insurer` 非空率 | **~9.5%** | 业务侧填充，覆盖部分 |
| `next_insurer` 非空率 | **~2.9%** | **业务侧填充滞后 1-3 月**，覆盖低 |
| 文内 `(policy_no, start_date)` 重复率 | 8~15% | **源系统特性**，非 ETL bug |
| `(policy_no, start_date)` 复合主键 | 唯一 | ETL 去重后保证 |

**关键认知**：`next_insurer` 在源系统里**单次写入式** — 同保单同起期在多个文件里要么都空、要么都同值，不会随时间从空变非空。也就是说**当前看到的空值不会被后续 ETL 自动补齐**，必须靠业务侧重新生成全量 Excel 才能更新。详见 [§5 业务推动建议](#5-业务推动建议)。

## 3. 文件分层（17 个源文件，缺一不可）

### A. 历史底库（永久保留）
`08_客户来源去向_20250101_20260419.xlsx` — 1.04M 行

- **不可删** — 独家覆盖 2025-01-01 ~ 2025-01-26 共 26 天（约 5.5 万 keys）
- 「每周日更新」是滚动窗口（保留约 14 个月），早期数据已滚出
- 此文件是 2025 年 1 月上旬的**唯一历史快照**

### B. 滚动底库（每周替换）
`08_客户来源去向_每周日更新.xlsx` — 1.04M 行

- 覆盖 2025-01-27 ~ T-7 左右（每周日全量更新）
- 是 next_insurer 数据填充的**主要来源** — 业务侧只在这张表里逐步补录

### C. 单日 / 跨窗口补齐
| 文件 | 角色 | 起因 |
|---|---|---|
| `08_客户来源去向_20260426.xlsx` | 2026-04-26 唯一来源 | 该日没有 `20260426_*` 日增量 |
| `08_客户来源去向_20260430_20260504.xlsx` | 五一假期 5 天唯一来源 | 节假日断档，业务方一次性补 |

### D. 日增量（T-1 每日产生）
`YYYYMMDD_08_客户来源去向.xlsx`，从 2026-04-27 起每日一份

- 命名约定：日期前缀 = 该文件覆盖的「保险起期」单天
- 每文件 1.5k~2.8k 行，包含 prev_insurer ~40% 填充率
- `next_insurer` 在日增量中**几乎全空**（0%）— 因为新签保单根本还没到期

### 日期覆盖关系图

```
2025-01-01  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  T-1
            └─ A (历史底库) ────┐
            01-26               │
                01-27 ─────────── B (滚动底库) ──────────── 2026-04-25
                                                                04-26 ─ C (单日)
                                                                  04-27~04-29 ─ 日增量
                                                                          04-30~05-04 ─ C (跨窗口)
                                                                                  05-05~T-1 ─ 日增量
```

## 4. ETL 处理逻辑

`pipelines/convert_customer_flow.py` (BaseConverter 模板方法)

1. **加载**：扫描 `08_客户来源去向*.xlsx` + `[0-9]*_08_客户来源去向*.xlsx`，并集合并
2. **去重**：按 `(policy_no, insurance_start_date)` 复合主键，`ORDER BY insurance_start_date DESC NULLS LAST, (next_insurer IS NULL) ASC, (previous_insurer IS NULL) ASC` 保留首条
   - 防御性排序：即使将来出现「同保单同起期 next 一边空一边非空」也优先保留非空
3. **校验**：
   - `expect_dates_continuous`: 2025-01-01 ~ T-1 不允许日期缺口
   - `min_rows`: ≥ 900,000
   - `require_non_null`: prev/next 至少各有 1 条非空（防止全量回归到 0）
4. **输出**: `warehouse/fact/customer_flow/latest.parquet`

## 5. 业务推动建议

### 问题
`next_insurer` 整体非空率 ~2.9%，**自贡 534 张续保追踪保单中仅 1 条有值**。这导致「流失公司」这一关键运营指标基本不可用。

### 根因
1. 业务系统不主动回传 `次年保险公司` —— 它是**从同业信息共享平台查询回来**，依赖人工 / 半自动流程
2. 数据回流滞后 1-3 个月 —— 客户去对家投保后，同业系统通常隔月才能查到
3. 一旦填上就**不会再变化**，源系统是单次写入而非动态更新

### 推动方向（按推动难度排序）

**短期（1 个月内）**：
- 数据治理团队牵头，每周生成一份 **`next_insurer 补录批次表`**，按 `(保单号 → 次年保险公司)` 两列回填
- 在 ETL 增加一个 `customer_flow_backfill.xlsx` 入口，专门处理补录
- 估计可将填充率从 3% 提到 8-10%

**中期（3 个月内）**：
- 推动接入「同业信息共享平台 API」直读，替代月度 Excel 导出
- 自贡、天府等高优先级机构每月手工核对 + 补录

**长期（6+ 个月）**：
- 与 IT 系统对接 → 客户续保数据落表时自动写回 next_insurer
- 流失客户分析模型上线，从呼叫记录 / 退保备注里提取竞品名称

## 6. 增量化方案评估（暂不实施）

### 现状
- 17 个源文件总计 **332 MB** xlsx
- 每次 ETL 全量重读所有文件，耗时 **~30 秒**（17 个转换 + 1 次 DuckDB ROW_NUMBER 合并）
- 输出 parquet 22 MB

### 增量化设想
- 仅处理 `start_date > MAX(latest.parquet.start_date)` 的日增量文件
- 跳过 1.04M 行的大底库重读
- 预期节省：~20 秒 + 70 MB IO

### 为什么暂不实施
**`next_insurer` 字段在源系统中是「单次写入式」，主要在「每周日更新」全量文件里逐步补齐**。如果增量化跳过周底库重读：
- 业务侧本周补录的 next_insurer **不会反映到 latest.parquet**
- 「流失公司」字段会滞后 1-N 周才更新
- 收益 20 秒，但代价是核心字段时效性下降 → 性价比不划算

### 替代方案（推荐）
保持当前 multi_file_merge 全量重建模式。如果未来 IO 成为瓶颈：
- 方案 A：仅当「每周日更新.xlsx」mtime 变化时全量重建，否则跳过
- 方案 B：用 SQLite/DuckDB 持久化中间结果，避免 xlsx 重复解析
- 方案 C：推动业务方接入 API，直接落表跳过 Excel 链路
