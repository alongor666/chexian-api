# 车险经营分析平台命名与路由注册表设计

> 日期：2026-07-12
> 基线：`origin/main@14e12a10`
> 状态：设计已在会话中逐节批准，等待书面规格复核

## 1. 决策结论

本轮采用渐进式统一注册表方案：建立 `productMetadata` 与 `routeRegistry` 两个手写事实源，驱动产品名称、侧栏导航、权限配置选项和相关一致性检查。

- 最终生产主品牌固定为 **车险经营分析平台**。
- 工程仓库名保持 `chexian-api`。
- 当前 canonical URL、页面、API、SQL 和数据口径保持不变。
- 旧 URL 仅作为兼容 alias，不进入导航和权限配置 UI。
- “五个决策工作台”的全面领域重构是未来理想态，不进入本轮 backlog、PR 或完成标准。

## 2. 必要性与极简性判断

当前代码存在多头命名与多份路由清单：顶部、登录页、浏览器标题、README、后端包描述使用不同产品名；App、侧栏和权限页分别手写路由，其中权限页仍展示已合并或不存在的旧路由。

只替换文案不能阻止再次漂移；全面重构页面与 API 又超出问题所需。本设计选择最小结构性修复：新增两个事实源并迁移直接消费者，不重写业务能力。

## 3. 目标架构

### 3.1 产品元数据

`productMetadata` 至少提供：

- `productName`：车险经营分析平台；
- `mobileName`：车险经营；
- 浏览器标题生成函数，格式为 `页面名｜车险经营分析平台`；
- AI 用户可见名：经营副驾。

代码内部 `copilot` 模块名和 API 路径可保留，避免无价值的大范围重命名。

### 3.2 路由注册表

每个 canonical 页面登记：

- 稳定 `routeId`；
- canonical `path`；
- 完整标签与窄栏短标签；
- 所属决策域；
- 图标键或图标引用；
- 是否出现在导航；
- 是否允许在权限管理中配置；
- 可选兼容 aliases；
- 可选特殊功能守卫标识。

注册表不承载 React 页面组件，不把 `App.tsx` 变成动态路由加载器。本轮保留显式 `<Route>`，以类型和代码分割清晰度优先；测试负责校验显式 App 路由与注册表一致。这避免注册表成为新的全能框架。

### 3.3 消费关系

```text
productMetadata ──→ 顶部 / 登录页 / HTML title / 用户可见 AI 名称

routeRegistry ────→ 侧栏导航
                ├─→ 权限管理选项
                ├─→ App 路由一致性测试
                └─→ README / INDEX 漂移检查
```

任何消费者引用未登记 route id、重复 canonical path 或 alias 与 canonical 冲突时必须 fail-loud；禁止静默隐藏或回退到第二份手写列表。

## 4. 信息架构与 UI 命名

侧栏按五个决策域和一个支撑域重组，但不改变 URL：

| 一级域 | 页面入口 |
|---|---|
| 经营总览 | 首页、经营看板、图表账本 |
| 增长达成 | 业绩分析、保费计划达成、增长对比 |
| 成本质量 | 成本分析、赔案分析、费用率发展、摩意成本 |
| 客户经营 | 续保追踪、报价转化、客户流向 |
| 专项资源 | 驾意险与货车、维修资源 |
| 平台管理 | 数据管理、权限管理 |

页面名称遵循“业务对象 + 分析任务”。不在本轮重做 Logo、颜色、布局或页面内容。

## 5. 兼容与权限不变量

1. 当前 canonical URL 全部保持可达。
2. 旧书签继续定向到同一业务能力。
3. 权限配置绑定 canonical route id/path，不展示 alias。
4. 无权页面的 fail-closed 行为保持不变。
5. 特殊功能守卫（摩意模型、费用率发展等）保持现有逻辑。
6. 导航重组不能扩大任何用户的页面或数据访问范围。

## 6. 文档与历史架构

- README 的项目名称、页面表和能力描述与注册表同步。
- ARCHITECTURE 主体只描述当前 `ETL/warehouse → Express/DuckDB → Web/CLI/MCP/AI` 架构。
- 旧 Python L2 `input/output` 子项目约定移入 legacy reference，避免与当前数据仓库模型并列为强制规则。
- C 方案只在架构演进说明中登记为非承诺方向，不创建当前 backlog 子任务。

## 7. 实施分项与 Agent 质量闭环

实施计划应拆为以下顺序任务，避免多个 Agent 并发修改共享注册表：

1. 事实源与契约测试；
2. 产品品牌消费者迁移；
3. 侧栏与权限消费者迁移；
4. 文档与治理闸；
5. 全量集成验证与最终审计。

每项严格执行：

```text
实施 Agent → 自审与定向测试
           → 独立规格符合性审查
           → 独立代码质量审查
           → 主代理复核 diff 与验证证据
           → backlog 追加 DONE 事件
```

规格或质量审查发现问题时，由原实施 Agent 修复并重新审查；不得以“基本符合”关闭。

## 8. Backlog 模型

- 使用仓库现有 append-only `backlog-events/` 登记父任务与分项任务。
- 开工前先核查事件 schema、uid 与派生视图生成器，不手改派生视图。
- 状态变化以事件追加表达；派生 `BACKLOG.md` / archive 由现有生成器重渲染。
- PR 创建不等于 DONE。
- DONE 至少要求：分项验收通过、两级独立审查关闭、主代理复验、相关提交已进入目标 PR。
- 总任务 DONE 还要求 required CI 通过且无未解决 review comment。

## 9. 测试与验收

### 9.1 定向测试

- 产品元数据值与标题格式；
- route id/path 唯一性；
- aliases 不进入导航和权限配置；
- 注册表覆盖全部 canonical App 页面；
- 侧栏按六个一级域生成且顺序稳定；
- 权限页不再出现 `/renewal`、`/truck`、`/cross-sell`、`/comparison` 等旧配置项；
- 现有 redirect 回归测试继续通过；
- 特殊功能可见性与权限行为保持不变。

### 9.2 集成门禁

- `bun run typecheck`
- `bun run build`
- `bun run test --run`
- `bun run governance`

若仓库现有失败与本改动无关，必须给出基线对照证据；不得把失败静默忽略。

## 10. 非目标

- 不改仓库名、生产域名或部署拓扑；
- 不改 canonical URL；
- 不重写页面、API、SQL、ETL 或指标口径；
- 不合并成五个新页面；
- 不建设完整品牌视觉系统；
- 不修改 frozen 或 user-only policy 文件；
- 不把未来 C 方案转成当前承诺。

## 11. 回滚策略

迁移按消费者分提交。若 UI 或权限回归，可回退对应消费者迁移提交；事实源与契约测试可独立保留。旧 URL redirects 在整个迁移期保持不动，因此无需数据迁移或外部链接批量切换。

## 12. 完成定义

只有同时满足以下条件，父 backlog 才可标记 DONE：

1. 产品主名在所有用户可见主入口统一；
2. 侧栏和权限配置由 route registry 派生；
3. 权限 UI 不含旧 alias 或死路由；
4. README、架构文档与当前代码一致；
5. 定向测试、typecheck、build、全量测试、governance 均有新鲜通过证据；
6. 每个分项经过规格与质量双审；
7. PR required CI 通过且无未解决审查意见。
