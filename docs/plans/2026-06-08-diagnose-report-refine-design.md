# diagnose-report-refine 设计文档

> 生成：2026-06-08 · 经 brainstorming 敲定并获用户逐节确认 · 落地走 chexian-crystallize-skill 流水线
> 定位：诊断报告草案的**后处理结论精炼器**——替代人眼审读，回查数据核验，产出「全局概览 + 各表隐性结论」的正式版。

## 1. 背景与定位

`diagnose_renewal.py` 等生成器产出的报告，结论是**规则化草稿**（如「问题一·续保率缺口」），措辞机械、不够「替代人眼」。需要一个后处理层：输入已生成的报告草案，AI 替代人眼找出问题/规律/异常点，用精准且符合数据事实、符合行业通用表达的语言重写结论，另存正式版。草案保留可追溯。

现有 `rewrite-conclusion` 不具备此能力：它是出险率/费用率/保费达成 5Why 专用、输入是车型/机构诊断卡、不回查 Parquet、不另存正式版、领域红线全是出险率领域。故新建。

## 2. 归属决策

**项目级**（非共享仓）：强依赖本项目数据——`renewal_tracker` Parquet 回查、续保口径红线、`diagnose_renewal.py` 报告格式、`renewal_common` 阈值/目标值。按 crystallize 规则「强依赖单项目数据 → 项目级」+「不为共享强行上提」。「先续保后推广」的推广是未来 YAGNI。

落地形态与 `diagnose-renewal` 一致：
- 命令文档 `.claude/commands/diagnose-report-refine.md`（用户 `/diagnose-report-refine` 调用）
- Python `数据管理/pipelines/refine_verify.py`（复用 `renewal_common` 口径/阈值）
- 测试 `数据管理/pipelines/test_refine_verify.py`

## 3. 架构与数据流

```
report.md → refine_verify.py（确定性）：
              ① 解析报告(元信息/各表/草案结论)
              ② 每表算事实结构(漏斗/最大异常点/对比落差)
              ③ 续保 adapter 回查 Parquet 核验成熟表合计(防 AI 编数/生成器漂移)
              ④ 输出事实包 JSON(含校验结果)
          → AI(命令文档方法论)：
              ⑤ 读事实包；校验失败→中止报错
              ⑥ 按两层方法论写 概览 + 各表隐性结论
              ⑦ Write 正式版 md(原表保留 + 草案结论替换 + 概览置顶)
```

**分工铁律**：数字由 ③ 回查保真，判断由 ⑥ 方法论保质。Python 不写判断（写不出精准现象语言），AI 不肉眼读表格数字（一律引用 ④ 事实包，杜绝看错）。

## 4. refine_verify.py 输出契约（JSON 事实包）

```json
{
  "meta": {"domain":"renewal","view":"org-report","org":"乐山","year":2026,"month":6,"cutoff":"2026-06-08"},
  "tables": [
    {"idx":1,"title":"当月已到期续保表","maturity":"matured",
     "total":{"yc":78,"quoted":63,"renewed":36,"quote_rate":80.8,"renew_rate":46.2},
     "funnel":[78,63,36],
     "top_anomaly":{"name":"刘洋新","metric":"续保影响度","value":14.1,"quote_rate":23.1,"renew_rate":15.4},
     "gap":{"unquoted":15,"unquoted_top":{"name":"刘洋新","value":10}},
     "contrast":null}
  ],
  "verify": {"checked":["table1","table5"],"ok":true,
             "details":[{"table":"table5","field":"yc","report":1623,"parquet":1623,"match":true}]}
}
```

- `maturity`: `matured`(已到期=最终留存) / `progress`(未到期=进度)，决定 AI 措辞与是否亮灯
- `verify.ok=false` → AI 必须中止、不产出正式版、列出差异

## 5. 两层方法论（写在命令文档，AI 执行）

**通用写作层**（所有诊断报告通用）：
1. 金字塔原理：概览总判断先行，各表隐性结论承接
2. 定量后定性：判断由数据导出
3. **数字优先·节约认知资源为最高准则**：精确数字优先（`续保率 46.2%`），禁数字文字化（「五成」「过半」）；模糊词与数字文字化都是认知噪声，一并禁
4. 禁管理动作：只呈现现象/规律/杠杆，不写「该派单给谁/怎么做」
5. 不堆砌：每判断保留最有支撑力的 2-3 个数字

**续保领域口径层（adapter 红线）**：
- 续保影响度是**负面指标**（越高越坏），措辞「拉大整体续保缺口」，禁用「贡献」
- **已到期=最终留存**（matured）；**未到期=进度**（progress）——绝不把进度当塌方
- 临期 7 天 ≠ 已流失（仍在续保动作窗口内）
- 已续回=已签单续保（is_renewed），含提前签单
- 可续期锚点=到期前 30 天（四川规则）

## 6. 现象提问式写法 + 概览规范

**现象提问式（每表）**：陈列漏斗递进 → 缺口归因到异常点 → 对比落差锚点（落差自现杠杆，不加动作句）。不出现「结论/问题」字样。1-3 句，克制。
> 示例：`78 件应续 → 63 件报价 → 36 件续回。15 件缺口来自未报价，刘洋新占 10 件。已报价续保率 57.1%，未报价 0%。`

**概览（置顶）**：一句全局总判断 + 3-5 条跨表收敛要点（现象式）+ 一句口径提醒（已到期 vs 未到期）。

## 7. 质量自检与边界

**自检清单**：核心判断由数据决定无凑数 / 无模糊词无数字文字化 / 无管理动作 / 区分现象与根因 / 不把恒等式当发现 / 续保口径红线全守 / 概览总判断一句话说清 / **所有数字来自事实包（经回查）**。

**边界**：
- `verify.ok=false`（报告数字与 Parquet 不符）→ 中止报错，列差异，不产出错误正式版
- 报告无法解析 → 报错提示检查来源
- 无 adapter 的域（出险率等，推广前）→ 跳过回查、仅用通用写作层、显式标注「未回查」

## 8. 预留输入接口（内容逐渐积累）

| 接口 | 作用 | 缺省行为 |
|---|---|---|
| `--context <md>` | 注入关键人/机构/事件背景（AI 编不出的数字外上下文） | 缺省=仅基于数据，不确定处**标注「需人工确认背景」、不武断** |
| `--style <md>` | 表达范式样例，学行业遣词造句 | 缺省=用命令文档内置写法规范 |
| `--audience <角色>` | 读者视角（分公司总/机构负责人/团队主管）决定详略口吻 | 缺省=机构负责人 |

## 9. 推广路径（先续保后推广）

通用写作层（§5 上半）已与领域解耦。推广到出险率/费用率报告时，只需：① 加 `adapters/<domain>.py`（数据源映射 + 领域口径红线）② 命令文档加该域的 maturity/口径说明。无需重写写作层。
